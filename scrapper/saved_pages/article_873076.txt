xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-blurred="true" data-src="https://habrastorage.org/getpro/habr/upload_files/6d9/80d/b28/6d980db28edff29c39fad238be0c00db.jpeg" rel="noopener noreferrer nofollow">В прошлой статье</a> я разобрал как работают корутины под капотом, рекомендую ознакомиться прежде чем закапываться дальше. В этой статье мы разберем такую сущность как Job и как на самом деле под капотом отменяются корутины, в общем будет интересно, погнали!</p><h2>Знакомимся с Job и JobSupport </h2><p>Черканем небольшой пример:</p><pre><code class="kotlin">// parent coroutine
launch { 
    // child coroutine 1
    launch { 
        ...
    }
    // child coroutine 2
    launch { 
        ...
    }
}.cancel()</code></pre><p>По логике <a href="https://kotlinlang.org/docs/coroutines-basics.html#structured-concurrency" rel="noopener noreferrer nofollow">Structured Concurrency</a> отмена родительской корутины должна привезти к отмене дочерних корутин (1 и 2), это одно из самых базовых правил на котором строится библиотека.</p><p>Провалимся в метод <code>cancel()</code> и узнаем кому он принадлежит: </p><pre><code class="kotlin">interface Job : CoroutineContext.Element {
    // добавляет дочернюю Job 
    fun attachChild(child: ChildJob): ChildHandle
  
    // отменяет Job
    fun cancel(cause: CancellationException? = null)
}</code></pre><p>Метод <code>cancel()</code> принадлежит такой сущности как <code>Job</code>, это один из базовых элементов <code>CoroutineContext'а</code>, отвечающий за одно из следующих состояний корутины: </p><ol><li><p>выполняется</p></li><li><p>отменена</p></li><li><p>завершена</p></li><li><p>завершена с ошибкой</p></li><li><p>еще не начала выполняться (если это <a href="https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines/-coroutine-start/-l-a-z-y/" rel="noopener noreferrer nofollow">lazy запуск корутины</a>)</p></li></ol><p>В принципе <code>Job</code> можно назвать <a href="https://en.wikipedia.org/wiki/Finite-state_machine" rel="noopener noreferrer nofollow">конечным автоматом</a> - есть какое-то начальное состояние, например "выполняется" и есть возможные переходы в другие, например при выбросе <code>CancellationException</code> корутина перейдет в состояние "отменена" и тд.</p><p>С основными определениями разобрались, теперь переходим к базовой реализации <code>Job</code>, она содержится в <code>JobSupport</code>, здесь происходит вся работа с состоянием, также этот класс наследует сама корутина:</p><pre><code class="kotlin">abstract class AbstractCoroutine&lt;in T&gt;(
    parentContext: CoroutineContext,
    initParentJob: Boolean,
    active: Boolean
) : JobSupport(active), Job, Continuation&lt;T&gt;, CoroutineScope { ... }</code></pre><p>Класс <code>JobSupport</code> достаточно большой, более 1000 строк кода, но нас пока интересует только реализация метода <code>cancel()</code>:</p><pre><code class="kotlin">// реализация содержится в JobSupport
override fun cancel(cause: CancellationException?) {
    cancelInternal(cause ?: defaultCancellationException())
}

// дополнительный метод cancelInternal() нужен для переопределения
// логики отмены в других корутинах, например в ChannelCoroutine
open fun cancelInternal(cause: Throwable) {
    cancelImpl(cause)
}

// для большинства корутин базовая логика отмены содержится тут
internal fun cancelImpl(cause: Any?): Boolean {
    var finalState: Any? = COMPLETING_ALREADY
    if (onCancelComplete) {
        // тут логика токо для CompletableDeferred корутин, 
        // в этой статье я не буду в них углубляться
    }
    
    if (finalState === COMPLETING_ALREADY) {
        // нас интересует этот метод
        finalState = makeCancelling(cause)
    }
    // на основе финального состояния возвращаем true, если отмена удалась 
    // или false если произошла какая-то фигня
    return when {
        finalState === COMPLETING_ALREADY -&gt; true
        finalState === COMPLETING_WAITING_CHILDREN -&gt; true
        finalState === TOO_LATE_TO_CANCEL -&gt; false
        else -&gt; {
            afterCompletion(finalState)
            true
        }
    }
}

// здесь происходит отмена корутины
private fun makeCancelling(cause: Any?): Any? {
    var causeExceptionCache: Throwable? = null // lazily init result of createCauseException(cause)
    loopOnState { state -&gt;
        when (state) {
            ...
            // Incomplete это состояние выполнения корутины
            is Incomplete -&gt; {
                val causeException = causeExceptionCache ?: createCauseException(cause).also { causeExceptionCache = it }
                if (state.isActive) {
                    // вызывается в случае отмены корутины, наш случай
                    if (tryMakeCancelling(state, causeException)) return COMPLETING_ALREADY
                } else {
                    // вызывается в случае стандартного завершения корутины (не отмена)
                    val finalState = tryMakeCompleting(state, CompletedExceptionally(causeException))
                    when {
                        finalState === COMPLETING_ALREADY -&gt; error("Cannot happen in $state")
                        finalState === COMPLETING_RETRY -&gt; return@loopOnState
                        else -&gt; return finalState
                    }
                }
            }
            else -&gt; return TOO_LATE_TO_CANCEL // already complete
        }
    }
}</code></pre><p>Прежде чем идти дальше хотел бы уточнить значения следующих констант:</p><ol><li><p><code>COMPLETING_ALREADY</code> - корутина уже завершена.</p></li><li><p><code>COMPLETING_WAITING_CHILDREN</code> - корутина ожидает завершения детей.</p></li><li><p><code>COMPLETING_RETRY</code> - корутина не смогла успешно завершится и надо попробовать еще раз.</p></li><li><p><code>TOO_LATE_TO_CANCEL</code> - слишком поздно для отмены корутины.</p></li></ol><p>Это вспомогательные состояния на основе которых метод <code>cancelImpl</code> определяет итоговый результат и выполняет завершающее действие (если корутина завершилась):</p><pre><code class="kotlin">internal fun cancelImpl(cause: Any?): Boolean {
    val finalState = ...
  
    return when {
        finalState === COMPLETING_ALREADY -&gt; true
        finalState === COMPLETING_WAITING_CHILDREN -&gt; true
        finalState === TOO_LATE_TO_CANCEL -&gt; false
        else -&gt; {
            afterCompletion(finalState)
            true
        }
    }
}</code></pre><p><strong>Важный момент</strong>: вспомогательные состояния не хранятся в <code>Job'е</code>, а вычисляются на лету при выполнении методов таких как <code>cancel()</code> например.</p><p>В следующем разделе рассмотрим какие состояния хранятся в <code>Job'е</code> и как они могут быть изменены.</p><h2>Углубляемся в состояния JobSupport</h2><p>Для начала приведу простой пример, показывающий зачем вообще нужно хранить состояние для корутины:</p><pre><code class="kotlin">class Thread1 : Thread() {

    private var isRunning = false

    val isActive: Boolean
        get() = isRunning
    
    override fun run() {
        isRunning = true
        while (true) {
            // выполняем цикл пока isRunning равен true
            if (!isRunning) break
            ...
        }
    }
    
    fun cancel() {  
        // завершаем цикл
        isRunning = false
    }
  
}</code></pre><p>Когда поток <code>Thread1</code> будет отменен, логика внутри него завершится, состояние станет неактивным, на основе этого состояния можно:</p><ol><li><p>Узнать завершился поток или нет.</p></li><li><p>Выполнить определенную логику для нового состояния (завершить цикл).</p></li></ol><p>Аналогично для корутин, состояние позволяет узнать что происходит с корутиной в данный момент, а также выполнить какую-то логику в зависимости от этого состояния.</p><p>Посмотрим на самые часто используемые состояния в <code>JobSupport</code>:</p><pre><code class="kotlin">// дополнительные состояния, которые вычисляются на лету (не хранятся в JobSupport)
val COMPLETING_ALREADY = Symbol("COMPLETING_ALREADY")
val COMPLETING_WAITING_CHILDREN = Symbol("COMPLETING_WAITING_CHILDREN")
val COMPLETING_RETRY = Symbol("COMPLETING_RETRY")
val TOO_LATE_TO_CANCEL = Symbol("TOO_LATE_TO_CANCEL")

// при запуске корутины по умолчанию ей присваивается состояние EMPTY_ACTIVE
val EMPTY_NEW = Empty(false) // для lazy корутин
val EMPTY_ACTIVE = Empty(true) // для всех остальных корутин

// корутина в статусе выполнения у которой нет дочерних корутин
class Empty(override val isActive: Boolean) : Incomplete { ... }

// практически тоже самое что и Empty, токо дополнительно добавляется
// список CompletionHandler'ов, которые будут вызваны при завершении корутины
class InactiveNodeList(
    override val list: NodeList
) : Incomplete { ... }

// корутина в статусе выполнения у которой есть одна дочерняя корутина
class ChildHandleNode(
    val childJob: ChildJob
) : JobNode(), ChildHandle { ... }

// корутина в статусе выполнения у которой есть более 
// чем одна дочерняя корутина
class NodeList : LockFreeLinkedListHead(), Incomplete { ... }

// корутина в статусе отмены или завершения
// это не финальное состояние, об этом еще будет полноценный раздел
class Finishing(
    override val list: NodeList,
    isCompleting: Boolean,
    rootCause: Throwable?
) : SynchronizedObject(), Incomplete { ... }

// корутина была полностью завершена по причине выброса исключения: 
// 1) CancellationException, если корутина была отменена
// 2) любые другие ошибки
// это финальное состояние, об этом еще будет полноценный раздел
class CompletedExceptionally(val cause: Throwable) { ... }

// JobSupport хранит состояние в atomic типе чтобы обезопасить 
// корутину от ошибок многопоточности (race condition)
val _state = atomic&lt;Any?&gt;(if (active) EMPTY_ACTIVE else EMPTY_NEW)</code></pre><p>В <code>JobSupport'е</code> хранится только актуальное состояние, им может быть одно из вышеперечисленных. </p><p>Если вы хотите в прикладном коде получить сведения о состоянии корутины, можете воспользоваться следующими Boolean полями:</p><pre><code class="kotlin">// доступны в корутине, так как корутина наследует JobSupport
class JobSupport : Job {

    // выполняется корутина или нет
    override val isActive: Boolean get() {
        val state = this.state
        return state is Incomplete &amp;&amp; state.isActive
    }

    // завершена корутина полностью или нет (финальное состояние)
    override val isCompleted: Boolean get() {
        return state !is Incomplete
    }

    // отменена корутина или нет
    override val isCancelled: Boolean get() {
        val state = this.state
        return state is CompletedExceptionally || (state is Finishing &amp;&amp; state.isCancelling)
    }
  
}</code></pre><p>Думаю вы уже использовали одно из них, например вызывали <code>Job.ensureActive()</code> метод:</p><pre><code class="kotlin">fun Job.ensureActive(): Unit {
    if (!isActive) throw getCancellationException()
}</code></pre><p>С <code>JobSupport</code> состояниями более менее разобрались, можем снова возвращаться к логике:</p><pre><code class="kotlin">internal fun cancelImpl(cause: Any?): Boolean {
    ...
    val finalState = makeCancelling(cause)
    ...
}

private fun makeCancelling(cause: Any?): Any? {
    when (state) {
        ...
        is Incomplete -&gt; tryMakeCancelling(state, causeException)
    }
}                </code></pre><p>Я уже приводил код этих методов, можете вернуться в предыдущий раздел если его пропустили, главное понять цепочку вызовов, которая происходит при отмене:</p><blockquote><p>cancel() → cancelInternal() → cancelImpl() → makeCancelling() → tryMakeCancelling()</p></blockquote><p>По итогу вызывается метод <code>JobSupport.tryMakeCancelling()</code>, давайте провалимся туда:</p><pre><code class="kotlin">private fun tryMakeCancelling(state: Incomplete, rootCause: Throwable): Boolean {
    assert { state !is Finishing } // only for non-finishing states
    assert { state.isActive } // only for active states
    // берем дочерние Job'ы из текущего состояния корутины
    val list = getOrPromoteCancellingList(state) ?: return false
    // создаем новое состояние - состояние отмены корутины
    val cancelling = Finishing(list, false, rootCause)
    // состояние обновляется через потокобезопасный алгоритм "Сравнение с обменом"
    // у меня был пост на этот счет: https://t.me/android_under_the_hood/86

private inline fun notifyHandlers(list: NodeList, cause: Throwable?, predicate: (JobNode) -&gt; Boolean) {
    var exception: Throwable? = null
    list.forEach { node -&gt;
        // пробегаемся по дочерним Job'ам и отменяем их
        // как мы уже знаем конкретным JobNode узлом для 
        // дочерних корутин является ChildHandleNode

            try {
                node.invoke(cause)

                    exception = CompletionHandlerException("Exception in completion handler $node for $this", ex)

    // какое-то исключение, то оно обработается в CoroutineExceptionHandler'е, 
    // реализация которого по умолчанию завершает программу со stacktrace'ом
    exception?.let { handleOnCompletionException(it) }

    // через coroutineScope { ... } билдер, такие корутины работают 
    // как suspend блок: пробрасывают исключение через конструкцию

        with(continuation) { resumeUndispatched(Unit) }

        // который будет вызван при отмене корутины,

        val outcome: Result&lt;Any?&gt; =
            try {

                // каждый вызов должен приводить к COROUTINE_SUSPENDED состоянию
                // после того как функция завершит свое выполнение она вызовет 
                // Continuation.resumeWith() этого Continuation'а и выполнится 
                // следующая suspend функция в switch case блоке
                val outcome = invokeSuspend(param)
                if (outcome === COROUTINE_SUSPENDED) return
                Result.success(outcome)
            } catch (exception: Throwable) {
                Result.failure(exception)

        coroutine.resumeWith(outcome)
    }

    fun sleep(cont: Continuation&lt;Any?&gt;): Any {
        Thread.sleep(500)
        // нет состояния приостановки COROUTINE_SUSPENDED,
        // код выполняется синхронно
        return Unit
    }

        // тут также нет состояния приостановки

        return Unit

    override fun resumeWith(result: Result&lt;Any?&gt;) {
        val outcome: Result&lt;Any?&gt; =
            try {
                // при хорошо спроектированных suspend функциях,
                // каждый вызов должен приводить к COROUTINE_SUSPENDED состоянию
                // после того как функция завершит свое выполнение она вызовет 
                // Continuation.resumeWith() этого Continuation'а и выполнится 
                // следующая suspend функция в switch case блоке
                val outcome = invokeSuspend(param)
                if (outcome === COROUTINE_SUSPENDED) return
                Result.success(outcome)
            } catch (exception: Throwable) {
                // вот тут обрабатывается CancellationException,
                // поэтому если вы напишите свой try / catch в suspend блоке
                // и перехватите это исключение, у вас может сломаться 
                // отмена корутины
                Result.failure(exception)
        }
        // после завершения suspend блока должна завершится сама корутина
        coroutine.resumeWith(outcome)
    }

    fun sleep(cont: Continuation&lt;Any?&gt;): Any {
        // создаем DispatchedCoroutine, под капотом она добавится
        // как дочерняя Job'а в родительскую корутину
        val coroutine = DispatchedCoroutine(...)
        // для простоты не стал расписывать логику диспатчеризации, 
        // она есть в прошлой статье 
        coroutine.runSuspendBlockInIOThread {
            // выполняем какую-то работу на IO потоке
            Thread.sleep(500)
            // также упростил логику возвращения на предыдущий поток
            coroutine.runResumeInPreviousThread {
                // выходим из состояния приостановки
                // result вычисляется на основе текущего состояния 
                // DispatchedCoroutine  
                cont.resumeWith(result)
            }
        }
        // до вызова cont.resumeWith() метод будет находиться 
        // в состоянии приостановки
        return COROUTINE_SUSPENDED
    }

    fun welcome(cont: Continuation&lt;Any?&gt;): Any {
        // эта функция остается без изменений
        println("Welcome!")
        return Unit
    }

    var label = 0
  
    override fun invokeSuspend(result: Result&lt;Any?&gt;): Any {
        when(label) {
            0 -&gt; {
                result.throwOnFailure()
                sleep(this)
                return COROUTINE_SUSPENDED
            }
            1 -&gt; {
                // result может содержать исключение CancellationException,
                // которое должно будет прокинуться выше и отменить выполнение
                // suspend блока
                result.throwOnFailure()
                welcome()
            }
        }
        return result
    }

}</code></pre><p>В чем собственно заключается отмена? Да все просто, любая suspend функция, которая хочет отменить дальнейшее выполнение других функций в suspend блоке должна вернуть результат с исключением:</p><pre><code class="kotlin">fun welcome(continuation: Continuation&lt;Any&gt;): Any {
    continuation.resumeWith(Result.failure(CancellationException("Отмени suspend блок!")))
    return COROUTINE_SUSPENDED
}</code></pre><p>Как мы уже выяснили такое возможно только если такая функция находится в состоянии приостановки, в противном случае suspend блок продолжит выполняться.</p><p>После того как <code>Continuation</code>, сгенерированный для suspend блока, получил результат с исключением, он выбросит это исключение через вызов <code>Result.throwOnFailure()</code> и завершит выполнение всего switch case блока.</p><p>Остается только найти место где возвращается результат с исключением, в нашем случае это находится вот тут:</p><pre><code class="kotlin">fun sleep(cont: Continuation&lt;Any?&gt;): Any {
    val coroutine = DispatchedCoroutine(...)
    coroutine.runSuspendBlockInIOThread {
        Thread.sleep(500)
        coroutine.runResumeInPreviousThread {
            // result берется из текущего состояния DispatchedCoroutine
            // и он как раз содержит CancellationException, когда корутина
            // была отменена
            cont.resumeWith(result)
        }
    }  
    return COROUTINE_SUSPENDED
}

// Finishing, содержащее CancellationException, это исключение

        val outcome: Result&lt;Any?&gt; = ...

        coroutine.resumeWith(outcome)
    }

    ...

    val state = makeCompletingOnce(result.toState())
    if (state === COMPLETING_WAITING_CHILDREN) return
    afterResume(state)
}

fun makeCompletingOnce(proposedUpdate: Any?): Any? {
    loopOnState { state -&gt;
        val finalState = tryMakeCompleting(state, proposedUpdate)
        ...
    }
}

// я упростил логику до одного tryMakeCompleting() метода,
// если хотите досконально разобраться во всех нюансах 
// читайте исходники JobSupport.tryMakeCompleting()
fun tryMakeCompleting(state: Any?, result: Any?) {
    if (state !is Incomplete) return COMPLETING_ALREADY

    // получаем дочерние Job'ы
    val list = getOrPromoteCancellingList(state) ?: return COMPLETING_RETRY

    // получаем состояние завершения или создаем новое для отмены дочерних узлов
    val finishing = state as? Finishing ?: Finishing(list, false, null)

    // тут извлекается искючение либо из текущего состояния корутины state 
    // либо из пришедшего результата result, его может не быть если
    // корутина завершается сама без отмен и ошибок
    val notifyRootCause: Throwable? = ...

    // если пришло исключение отмены или внештатная ошибка 
    // нужно отменить дочерние Job'ы 
    notifyRootCause?.let { notifyCancelling(list, it) }

    // текущая корутина будет ждать пока выполняться / отменятся дочерние корутины
    val child = list.nextChild()
    if (child != null &amp;&amp; tryWaitForChild(finishing, child, proposedUpdate))
        return COMPLETING_WAITING_CHILDREN

    // вычисляются финальное исключение и финальное состояние, 
    // обычно если корутина завершается штатно конечным состоянием будет:
    // Unit для launch корутин, 
    // Any для withContext корутин, 
    // если с ошибкой / отменой то CompletedExceptionally
    val finalException = ...
    val finalState = ...
       
    if (finalException != null) {
        // для отмены cancelParent() всегда возращает true,
        // в случае 
        // внештатного исключения оно будет прокидываться до главного 
        // родителя, где вызовется метод handleJobException() и
        // исключение обработается через CoroutineExceptionHandler:
        // по умолчанию установлен CoroutineExceptionHandler,
        // который крашит программу со stacktrace'ом
        val handled = cancelParent(finalException) || handleJobException(finalException)
        if (handled) (finalState as CompletedExceptionally).makeHandled()
    }

    // устанавливаем финальное состояние
    _state.compareAndSet(state, finalState.boxIncomplete())
      
    return finalState
}</code></pre><p>В итоге после завершения корутины у нее будет одно из следующих состояний:</p><ol><li><p>Unit - для штатного завершения корутин, запущенных через launch билдер (<code>StandaloneCoroutine</code>).</p></li><li><p><code>CompletedExceptionally</code> - для отмененной корутины или завершенной внештатно (исключение отличное от <code>CancellationException</code>).</p></li><li><p>Любой другой тип - для корутин, запущенных через withContext билдер (<code>DispatchedCorotuine</code>).</p></li></ol><p>С финальным состоянием разобрались, теперь посмотрим как родительская корутина дожидается завершения своих детей, предположим у нас есть следующий код:</p><pre><code class="kotlin">// корутина 1
launch {
    // корутина 2
    launch(Dispatchers.IO) {
        ...
    }
    // корутина 3
    launch(Dispatchers.IO) {
        ...
    }
}</code></pre><p>Не секрет что запуск корутин 2 и 3 произойдет без приостановки так как тут не надо дожидаться результата, а это значит что suspend блок корутины 1 сразу вернет управление через вызов <code>Continuation.resumeWith()</code>, где как мы уже выяснили корутина должна будет завершится. </p><p>Чтобы корутина 1 дождалась детей, есть специальная логика:</p><pre><code class="kotlin">fun tryMakeCompleting(state: Any?, result: Any?) {
    ...
  
    // текущая корутина будет ждать пока выполняться / отменятся дочерние корутины
    val child = list.nextChild()
    if (child != null &amp;&amp; tryWaitForChild(finishing, child, proposedUpdate))
        return COMPLETING_WAITING_CHILDREN

    ...
  
    // только после завершения детей корутина поменяет свое состояние 
    // на полностью завершенное
    _state.compareAndSet(state, finalState.boxIncomplete())

    ...
}

private tailrec fun tryWaitForChild(state: Finishing, child: ChildHandleNode, proposedUpdate: Any?): Boolean {
    // в дочернюю Job'у будет добавлен специальный CompletionHandler,
    // который вызовется когда она будет завершаться
    val handle = child.childJob.invokeOnCompletion(
        invokeImmediately = false,
        handler = ChildCompletion(this, state, child, proposedUpdate)
    )
    // если дочерняя Job'а еще не завершена ждем
    if (handle !== NonDisposableHandle) return true // child is not complete and we've started waiting for it
    // иначе смотрим следующую дочернюю Job'у
    val nextChild = child.nextChild() ?: return false
    return tryWaitForChild(state, nextChild, proposedUpdate)
}

private class ChildCompletion(
    private val parent: JobSupport,
    private val state: Finishing,
    private val child: ChildHandleNode,
    private val proposedUpdate: Any?
) : JobNode() {
    override val onCancelling get() = false
    // когда дочерняя Job'а будет завершаться она вызовет этот обработчик
    override fun invoke(cause: Throwable?) {
        // родительская корутина продолжит завершаться 
        parent.continueCompleting(state, child, proposedUpdate)
    }
}</code></pre><p>Родительская корутина будет ждать завершения детей и пока последняя дочерняя корутина не вызовет  <code>ChildCompletion.invoke()</code> обработчик, она не перейдет в финальное состояние.</p><p>P.S. Спасибо чуваку <a class="mention" href="/users/manwtein">@manwtein</a> что упомянул в комментах про ожидание дочерних корутин в родителе! </p><h2>Итоги</h2><p>Ух, думал не выдержу до конца этого длинного текста, но чем заниматься в Новогодние праздники если не писать суперсложную техническую статью?).</p><p>Давайте подведем итоги:</p><ol><li><p><code>Job</code> - элемент <code>CoroutineContext'а</code>, отвечающий за хранение одного из состояний корутины: выполнение, отмена, завершение, завершение с ошибкой, для lazy корутин есть еще состояние "не начала выполняться".  </p></li><li><p>В <code>Job</code> есть специальные Boolean поля, которые могут дать информацию о текущем состоянии корутины: isActive - выполняется корутина или нет, isCompleted - завершена корутина полностью или нет, isCancelled - отменена корутина или нет. </p></li><li><p><code>JobSupport</code> - это одна из базовых реализаций <code>Job'ы</code>, от которой наследуются другие реализации такие как <code>SupervisorJob</code> например.</p></li><li><p><code>JobSupport</code> может содержать ссылку на родительскую <code>Job'у</code>, если у корутины есть родитель, это нужно для пробрасывания отмены родителю в случае внештатной отмены.</p></li><li><p><code>JobSupport</code> состояние хранит в себе дочерние <code>Job’ы</code> по следующей логике: если нет дочерних <code>Job</code>, хранится <code>Empty</code> состояние; если есть одна дочерняя <code>Job'а</code> хранится <code>JobNode</code> узел, конкретный класс - <code>ChildHandleNode</code>; если несколько дочерних <code>Job</code> хранится состояние <code>NodeList</code>, где дочерние <code>JobNode</code> элементы лежат в связанном списке.</p></li><li><p>Класс корутины <code>StandaloneCoroutine</code> является наследником <code>JobSupport</code>, поэтому вся логика из <code>JobSupport</code> доступна корутине.</p></li><li><p>Отмена корутины может быть штатной, например при вызове <code>Job.cancel()</code>, в таком случае отменятся только дочерние корутины, отмена не будет пробрасываться родителю.</p></li><li><p>Отмена корутины может быть внештатной, например при выбросе любого исключения отличного от <code>CancellationException</code>, в таком случае отменятся дочерние корутины и отмена будет проброшена родителю, который может либо не пробрасывать дальше отмену (если это <code>SupervisorJob</code>), либо пробросить ее до самого главного родителя, где исключение будет обработано <code>CoroutineExceptionHandler'ом</code>.</p></li><li><p>По умолчанию установлен <code>CoroutineExceptionHandler</code>, который крашит программу со stacktrace'ом.</p></li><li><p>Фактическая отмена логики осуществляется не в самой корутине, а в правильно организованных suspend функциях таких как <code>delay()</code> или в suspend блоках, содержащих такие функции.</p></li><li><p>Функция <code>delay()</code> под капотом использует <code>CancellableContinuation</code>, который добавляется в родительскую корутину как дочерняя <code>Job'а</code> и при отмене корутины аналогично дочерним корутинам отменяется. Самая главная фишка <code>CancellableContinuation</code> в том что при отмене он вызывает специальные обработчики, их можно добавить через метод <code>CancellableContinuation.invokeOnCancellation()</code>, в функции <code>delay()</code> такой обработчик отменяет задержку.</p></li><li><p>Отменить можно только тот suspend блок где suspend функции могут быть в приостановленном состоянии, в противном случае код выполнится как один большой метод.</p></li><li><p>После отмены или завершения корутины она переходит в одно из финальных состояний: Unit - для штатного завершения корутин, запущенных через launch билдер (<code>StandaloneCoroutine</code>), <code>CompletedExceptionally</code> - для отмененной корутины или завершенной внештатно, любой другой тип - для корутин, запущенных через withContext билдер (<code>DispatchedCorotuine</code>).</p></li></ol><p>Полезные ссылки:</p><ol><li><p><a
