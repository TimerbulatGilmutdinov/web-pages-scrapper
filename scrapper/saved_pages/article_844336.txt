xmlns="http://www.w3.org/1999/xhtml"><p>Entity Graph — это один из мощных инструментов JPA, который помогает разработчикам гибко управлять загрузкой связанных сущностей. Entity Graph позволяет динамически настраивать загрузку данных во время выполнения программы, что делает его особенно полезным в проектах со сложными структурами данных.</p><p>Команда <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> подготовила статью, в которой рассмотрела, как использовать Entity Graph.</p><hr/><p>В версии JPA 2.1 была добавлена функция <strong>Entity Graph</strong>, которая улучшает производительность загрузки связанных сущностей. Ранее, в JPA 2.0, для управления стратегиями загрузки данных использовались два подхода: <strong>FetchType.LAZY</strong> и <strong>FetchType.EAGER</strong>. Однако эти стратегии статичны, что не позволяет гибко переключаться между ними во время выполнения программы. </p><p>Конечно, существует <strong>join fetch</strong>, который позволяет превратить ленивую загрузку (lazy) в жадную (eager) со стратегией JOIN. Но обратного способа — чтобы из жадной сделать ленивую — средствами JPQL не предусмотрено. И тут на помощь приходит <code>@EntityGraph</code>! Давайте разберёмся, как с ним работать более подробно...</p><p>Прежде чем мы приступим к рассмотрению Entity Graph, определим предметную область, с которой собираемся работать. Допустим, мы хотим создать сайт-блог, где пользователи могут комментировать посты и делиться ими.</p><p>Итак, для начала объявим сущность <code>User</code>:</p><pre><code class="java">@Entity
public class User {
  
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    private String email;
}</code></pre><p>Пользователь может делиться различными публикациями, поэтому нам также нужна сущность <code>Post</code>:</p><pre><code class="java">@Entity
public class Post {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String subject;

    @OneToMany(mappedBy = "post", fetch = FetchType.LAZY)
    private List&lt;Comment&gt; comments = new ArrayList&lt;&gt;();

    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "user_id")
    private User user;
}</code></pre><p>Пользователь также может комментировать опубликованные сообщения, поэтому добавим сущность <code>Comment</code>:</p><pre><code class="java">@Entity
public class Comment {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String reply;

    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn
    private Post post;

    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn
    private User user;
}</code></pre><p>Сущность <code>Post</code> имеет связь с сущностями <code>Comment</code> и <code>User</code>. Сущность <code>Comment</code> имеет связь с сущностями <code>Post</code> и <code>User</code>. </p><p>Наша задача состоит в том, чтобы загрузить граф:</p><pre><code class="markdown">Post  -&gt;  user:User
      -&gt;  comments:List&lt;Comment&gt;
            comments[0]:Comment -&gt; user:User
            comments[1]:Comment -&gt; user:User</code></pre><p>До появления Entity Graph разработчики использовали стратегии FetchType для управления загрузкой связанных данных:</p><p>- <strong>FetchType.EAGER</strong>: Сущность загружается сразу вместе с ее связями (но не всегда именно одним запросом). Подход используется по умолчанию для аннотаций <code>@ManyToOne</code> и <code>@OneToOne</code>.</p><p>- <strong>FetchType.LAZY</strong>: Связанные данные загружаются только тогда, когда они запрашиваются. Это поведение по умолчанию для связей <code>@OneToMany</code>, <code>@ManyToMany</code> и <code>@ElementCollection</code>.</p><p>Например, при использовании LAZY для сущности <code>Post</code>, комментарии к посту (сущность <code>Comment</code>) не будут загружаться по умолчанию:</p><pre><code class="java">@OneToMany(mappedBy = "post", fetch = FetchType.LAZY)
private List&lt;Comment&gt; comments = new ArrayList&lt;&gt;();</code></pre><p>В то же время для связи <code>ManyToOne</code> поведение по умолчанию — <code>EAGER</code>, что приводит к автоматической загрузке связанных данных:</p><pre><code class="java">@ManyToOne
@JoinColumn(name = "user_id")
private User user;</code></pre><p>Однако, чтобы сделать загрузку более гибкой и управляемой на уровне выполнения, и используется Entity Graph.</p><p>Рассмотрим пример с использование <code>@EntityGraph</code> и Spring Data:</p><pre><code class="java">@EntityGraph(attributePaths = {"comments"})
List&lt;Post&gt; findEntityGraphTypeFetchBySubject(String subject);</code></pre><p>Граф будет включать в себя загрузку всех базовых полей пользователя, который создал пост (user), а также комментарии и их авторов.</p><pre><code class="sql">Hibernate: 
    select
        p1_0.id,
        c1_0.post_id,
        c1_0.id,
        c1_0.reply,
        c1_0.user_id,
        p1_0.subject,
        p1_0.user_id 
    from
        post p1_0 
    left join
        comment c1_0 
            on p1_0.id=c1_0.post_id 
    where
        p1_0.subject=?</code></pre><p>Стоит отметить, что у самого EntityGraph также есть 2 вида загрузки: <code>EntityGraph.EntityGraphType.FETCH</code>  и <code>EntityGraph.EntityGraphType.LOAD</code>. При выборе режима Fetch (используется по умолчанию) ассоциативные атрибуты, явно объявленные для загрузки, например comments в нашем случае, будут выгружены жадно (FetchType.EAGER), остальные же атрибуты будут загружены лениво (FetchType.LAZY).</p><p>В случае же если мы будем использовать режим <code>EntityGraph.EntityGraphType.LOAD</code><em>, </em>выбранные<em> </em>атрибуты будут загружены жадно, а остальные атрибуты будут загружены в соответствии с тем, какой FetchType указан в модели.</p><pre><code class="java">@EntityGraph(attributePaths = {"comments"}, type = EntityGraph.EntityGraphType.LOAD)
List&lt;Post&gt; findEntityGraphTypeLoadBySubject(String subject);</code></pre><pre><code class="sql">Hibernate: 
    select
        p1_0.id,
        c1_0.post_id,
        c1_0.id,
        c1_0.reply,
        c1_0.user_id,
        p1_0.subject,
        p1_0.user_id 
    from
        post p1_0 
    left join
        comment c1_0 
            on p1_0.id=c1_0.post_id 
    where
        p1_0.subject=?
Hibernate: 
    select
        u1_0.id,
        u1_0.email,
        u1_0.name 
    from
        user_ u1_0 
    where
        u1_0.id=?
Hibernate: 
    select
        u1_0.id,
        u1_0.email,
        u1_0.name 
    from
        user_ u1_0 
    where
        u1_0.id=?

//И так далее, N+1 запрос</code></pre><p>Как можно заметить, вся информация о пользователях грузится в соответствии с тем, как указано в модели – то есть жадно, хоть мы и не указывали поле user для @EntityGraph.</p><p>Отдельно отметим, базовые атрибуты будут загружены всегда, независимо от выбранного режима загрузки, будь то <strong>FETCH</strong> или <strong>LOAD</strong>. Поэтому указывать их в качестве значений для attributePaths нет смысла.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/2fa/c6a/074/2fac6a07463c37d88e1330dbe26a2b68.png" height="334"
