xmlns="http://www.w3.org/1999/xhtml"><p>Всем привет. В предыдущей <a href="https://habr.com/ru/companies/explyt/articles/874236/">статье</a> мы рассказывали о нашей версии HTTP клиента, которая основана на генерации OpenAPI файлов по Spring-контроллерам и Swagger UI в качестве готового интерфейса пользователя для работы с запросами. В <a href="https://habr.com/ru/companies/explyt/articles/874236/comments/#comment_27797056">комментариях</a> к ней писали, что поддержка http файлов весьма актуальна, так как они используются во многих существующих проектах. Мы решили изучить этот вопрос подробнее и вот что из этого вышло…</p><p>Признаюсь честно, к своему стыду, я всегда думал, что http формат файлов и environment для их выполнения – это идея JetBrains и сугубо их внутренний закрытый продукт. Но спасибо пользователю <a href="https://habr.com/ru/users/Rabestro/">@Rabestro</a> и его <a href="https://habr.com/ru/companies/haulmont/articles/868106/comments/#comment_27693576">комментарию</a> к одной из статей по схожей тематике, который помог мне лучше разобраться в этом вопросе.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/178/979/6ac/1789796ac28dd2f13fcb060c3cbdb850.png" height="385" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/178/979/6ac/1789796ac28dd2f13fcb060c3cbdb850.png" width="1311"/></figure><h3>Немного истории</h3><p><em><u>VSCode Rest Client</u></em></p><p>Оказывается, JetBrains были отнюдь не первыми в реализации HTTP клиента посредством написания запросов в *.http файле. Так, плагин для VSCode <a href="https://marketplace.visualstudio.com/items?itemName=humao.rest-client">Rest Client</a> увидел свет в марте 2016 года, что на 1,5 года раньше плагина от JetBrains. И в своей <a href="https://github.com/Huachao/vscode-restclient/blob/master/README.md#usage">документации</a> Rest Client явно ссылается на стандарт <a href="https://datatracker.ietf.org/doc/html/rfc2616">RFC 2616</a>, который можно использовать для написания запросов. IDEA HTTP Client в своем баг <a href="https://youtrack.jetbrains.com/issue/IDEA-239452/Rest-Client-support-new-file-type-for-HTTP-Requests">трекере</a> ссылается на стандарт <a href="https://datatracker.ietf.org/doc/html/rfc7230">RFC 7230</a>. Это более свежая версия RFC 2616, хотя на настоящий момент они обе уже устарели и есть уже более новые версии. Таким образом, мы можем сделать вывод, что это не какой то кастомный язык для написания запросов, а полноценный стандарт. </p><p><em><u>httpYac</u></em></p><p>Затем я познакомился с инструментом <a href="https://httpyac.github.io/">httpYac</a>, вышедшем в марте 2021 года. Он не только предоставляет свой VSCode <a href="https://marketplace.visualstudio.com/items?itemName=anweber.vscode-httpyac">плагин</a> для работы с .http/.rest файлами, но также имеет свой environment для их выполнения, оформленный в виде <a href="https://httpyac.github.io/guide/installation_cli.html">CLI “тулзы”</a>. Получается, что уже есть готовая среда выполнения, которая умеет парсить и исполнять файлы с запросами и не зависит ни от какой IDE. </p><p><em><u>IDEA HTTP Client</u></em></p><p>Работа с *.http файлами появилась в IntelliJ IDEA, начиная с <a href="https://habr.com/ru/companies/JetBrains/articles/343382/">версии 2017.3</a>, которая вышла в ноябре 2017 года. И у меня возник вопрос, а есть ли у JetBrains готовая среда для исполнения запросов, оформленная в виде отдельной “тулзы”, как это сделано у <a href="https://httpyac.github.io/">httpYac</a>? Оказывается, есть - <a href="https://www.jetbrains.com/help/idea/http-client-cli.html">IDEA HTTP Client CLI</a>. Он был <a href="https://blog.jetbrains.com/idea/2022/12/http-client-cli-run-requests-and-tests-on-ci/">анонсирован</a> в декабре 2022 года. Но самое интересное, что этот проект изначально был реализован <a href="https://github.com/restcli/restcli">энтузиастами</a> в 2020 году, а уже в конце 2022 года перешел под крыло JetBrains.</p><p>А раз у нас уже есть готовая среда выполнения, да не одна, а несколько реализаций от разных разработчиков, то добавить в IDEA поддержку работы с *.http файлами выглядит совсем несложной задачей. Кстати, на сайте httpYac есть <a href="https://httpyac.github.io/guide/#feature-comparisons">табличка</a> сравнения инструментов для работы с HTTP запросами. Правда, сейчас она уже не совсем актуальна, так как в ней отсутствует поддержка CLI у JetBrains, хотя она есть.</p><h3>Что в итоге получилось</h3><p>Напрашивается вывод, что писать свой движок для выполнения HTTP запросов непростая задача, да и надо ли это делать, когда уже есть готовые инструменты? Так что мы решили не изобретать велосипед и делегировать работу уже готовым инструментам. Пользователь может выбрать нужный ему в настройках.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/43d/012/a95/43d012a9557e80179b3741241afcc8db.png" height="421" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/43d/012/a95/43d012a9557e80179b3741241afcc8db.png" width="929"/></figure><p>В настройках Explyt Spring плагина можно задать путь для нужного environment’а. В качестве примера я буду использовать <a href="https://httpyac.github.io/">httpYac</a>. Он более легковесный и имеет более понятный API, чем другие клиенты, так он позволяет выполнять конкретный запрос из файла по номеру строки. На мой взгляд, он также имеет более приятный вывод в консоль, благодаря цветовой гамме и что он по умолчанию форматирует ответ. В “HttpClient CLI Path” нужно указать полный путь до исполняемого файла. Если в операционной системе у нас установлена environment variable, можем использовать ее, как в примере выше. </p><p>Если у нас не установлено никакой среды исполнения запросов, есть кнопка “Download HttpClient Cli”, которая скачает <a href="https://www.jetbrains.com/help/idea/http-client-cli.html">IDEA HTTP Client CLI</a> и пропишет нужный путь в настройках.</p><p>Почему мы предлагаем скачать клиента от JetBrains, а не httpYac, на котором показываем примеры? </p><ol><li><p>httpYac имеет более сложную установку и требует <a href="https://nodejs.org/en">NodeJS</a> для запуска, но даже с учетом этого, он минимум в два раза легковеснее других клиентов;</p></li><li><p>“IDEA HTTP Client CLI” требует JAVA для запуска.Так как IntelliJ IDEA – это JVM IDE, то у пользователя JAVA уже есть и будет проще начать ей пользоваться сразу.</p></li></ol><p>Для работы в IntelliJ IDEA должны быть включены bundled плагины <a href="https://plugins.jetbrains.com/plugin/13122-shell-script">Shell Script</a> и <a href="https://plugins.jetbrains.com/plugin/13123-terminal">Terminal</a>. С их помощью осуществляется запуск CLI инструментов и вывод результата.</p><p>На данный момент это выглядит так:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/132/7d1/6ad/1327d16adc3d35c9ac5cfe90065f72d0.png" height="1276" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/132/7d1/6ad/1327d16adc3d35c9ac5cfe90065f72d0.png" width="1030"/></figure><p>Мы можем:</p><ul><li><p>выполнить весь файл целиком;</p></li><li><p>выполнить отдельный запрос;</p></li><li><p>выбрать Environmental файл (кнопка “+”) и указать конкретный env;</p></li><li><p>в "Additional command line args" указать дополнительные CLI аргументы, так как мы не заточены под один конкретный “тул” для работы с запросами.</p></li></ul><p>Результат отображается в консоли Здесь можно увидеть, с какими параметрами была вызвана команда, что упрощает отладку и разбор "багов".</p><h3>Заключение</h3><p>На данный момент, у нас реализован первый прототип решения, который позволяет пользователю выбрать удобный ему CLI инструмент и использовать его для работы с http/rest файлами. Реализованы возможности выполнить файл целиком и есть “ран” лайн-маркеры для запуска конкретного запроса. </p><p>Для начала мы бы хотели собрать обратную связь: будет ли это кому-то интересно/полезно? И если да, то мы будем дальше улучшать поддержку и удобство работы с http/rest файлов. </p><p>На этапе прототипа у нас пока нет полноценной подсветки синтаксиса и автодополнений. А также если это будет востребовано, тогда нужно будет вынести данную логику в отдельный плагин, так как она напрямую на связана со Spring, в отличии от <a href="https://habr.com/ru/companies/explyt/articles/874236/">нашей реализации</a> HTTP клиента на основе Spring аннотаций. Пишите, что думаете об этом в комментариях!</p><p>Скачать последнюю версию плагина можно <a href="https://explyt.com/en-download/?utm_source=habr&amp;t.coutm_campaign=post_1">тут</a> или напрямую с <a href="https://github.com/explyt/spring-plugin/releases/">GitHub Releases</a>.  Поддержка http/rest файлов доступена, начиная с версии 2*.1.3333.</p><p>Для обратной связи и сообщений об ошибках: <a
