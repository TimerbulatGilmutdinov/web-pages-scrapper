xmlns="http://www.w3.org/1999/xhtml"><p>На прошлой неделе команда <a href="https://t.me/+8RZ1V1mxrHw0MmQy">Amplicode</a> провела митап, который длился целых 3,5 часа! За это время состоялось шесть докладов, затрагивающих все аспекты разработки приложений на Spring — от настройки окружения до создания бизнес-логики и внедрения админки на React Admin.</p><h4>История и философия Amplicode (Илья Кучмин)</h4><p>Илья Кучмин, разработчик Amplicode, открыл митап с рассказа о том, как зарождался продукт, к чему он движется сейчас и какие проблемы помогает решать.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b56087bb36d4b3e77317" data-style="" id="66e4b56087bb36d4b3e77317" width=""></div><ul><li><p><a href="https://youtu.be/gqKPGsa7hrk?feature=shared">Смотреть на YouTube</a></p></li><li><p><a href="https://rutube.ru/video/53f8dcbdb486633778a95421f9d2b664/?r=wd">Смотреть на Rutube</a></p></li></ul><h4>Локальное окружение с Docker Compose для Spring Boot (Кирилл Толкачёв)</h4><p>Затем к митапу присоединились Георгий Власов (Developer Advocate Amplicode) и Кирилл Толкачёв (участник программного комитета JUG RU Group). Они обсудили приложение, с которым предстояло работать, и задачи, которые нужно было решить. Первый шаг — настройка локального окружения с Docker Compose и добавление сервисов для PostgreSQL и pgAdmin.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b5646748d7d46ffbfac7" data-style="" id="66e4b5646748d7d46ffbfac7" width=""></div><ul><li><p><a href="https://youtu.be/1NcFdQX08EE?feature=shared">Смотреть на YouTube</a></p></li><li><p><a href="https://rutube.ru/video/b00503943da08dc33125030b4bc64081/?r=wd">Смотреть на Rutube</a></p></li></ul><p>В течение видео было продемонстрировано, как с помощью Amplicode можно эффективно настроить окружение для приложения, используя Docker Compose.</p><p>Кирилл отметил, что существует три основных варианта решения задачи:</p><ol><li><p>Создать Docker Compose файл</p></li><li><p>Воспользоваться TestContainers</p></li><li><p>Использовать docker-compose-starter для Spring Boot</p></li></ol><p>В ходе обсуждений ребята приняли решение сосредоточиться на создании Docker Compose файла. Было показано, каким образом используя Amplicode можно не только генерировать сервисы для Docker Compose с нуля, но и как редактировать уже существующие сервисы.</p><h3>Проблемы JPA в Spring PetClinic и способы их решения (Илья и Фёдор Сазоновы)</h3><p>Когда приложение было готово к работе, Илья и Георгий пригласили экспертов сообщества Spring АйО — Илью и Фёдора Сазоновых. За 40 минут они не только решили несколько бизнес-задач, но и устранили накопившиеся за долги годы существования приложения проблемы с JPA.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b56cee68229db683c24d" data-style="" id="66e4b56cee68229db683c24d" width=""></div><ul><li><p><a href="https://youtu.be/efhh31qKTCk?feature=shared">Смотреть на YouTube</a></p></li><li><p><a href="https://rutube.ru/video/e76111647f649cf3ef2d91539a67c260/?r=wd">Смотреть на Rutube</a></p></li></ul><p>Эксперты начали с оптимизация модели данных в JPA. Первым шагом стало исправление односторонних связей, снижающих производительность приложения. Также обратили внимание на множественные <code>Eager</code> ассоциации, которые также могут негативно сказаться на производительности, но было принято решение не вносить изменения в рамках текущей задачи.</p><p>Далее был рассмотрен тип ID, используемый в приложении. Обсудили, почему использование стратегии <code>Identity</code> может быть не лучшим решением и решили заменить ее на <code>Sequence</code>. После этого добавили новое поле <code>salary</code> и модифицировали соответствующий экран. Илья и Фёдор очень удивились, когда увидили, что приложение смогло запуститься после изменений в типе идентификатора и добавления нового поля. Оказывается, в приложении не был настроен параметр <code>hibernate.ddl-auto</code>.</p><p>Наконец, эксперты и ведущие погрузились в написание Flyway миграций, и решению проблемы N+1. Были рассмотрены различные варианты решения этой проблемы: от использования <code>join fetch</code> и <code>@EntityGraph</code> и до <code>@BatchSize</code> и <code>spring.jpa.properties.hibernate.default_batch_fetch_size</code>.</p><h3>Разрабатываем правильный REST API для Spring Boot (Рустам Курамшин)</h3><p>Далее к митапу присоединился Рустам Курамшин, ещё один эксперт сообщества Spring АйО. Вместе они разработали REST API для интеграции с внешними сервисами и исправили проблемы с существующими эндпоинтами. Также заложили основу для CRUD API, который позже будет использоваться фронтом написанном на React Admin.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b573d1b5b66d8d215145" data-style="" id="66e4b573d1b5b66d8d215145" width=""></div><ul><li><p><a href="https://youtu.be/ya6KF2539QE?feature=shared">Смотреть на YouTube</a></p></li><li><p><a href="https://rutube.ru/video/c09b2fb1bdb71cd24e2b2cf82043155f/?r=wd">Смотреть на Rutube</a></p></li></ul><p>В течение 40 минут эксперт и ведущие выполнили три основные задачи: исправили существующий endpoint, реализовали CRUD REST API для будущего Admin UI и разработали API для интеграций. </p><p>В процессе решения первой задачи ребята обсудили преимущества и недостатки использования MapStruct.</p><p>Далее был создан CRUD REST-контроллер с использованием возможностей Amplicode. При генерации тестов с использованием MockMvc Рустам выразил предпочтение к WebTestClient, отмечая, что такие тесты хотя и выполняются дольше и требуют решения вопросов безопасности, но обеспечивают большую надёжность. Ребята из команды Amplicode признали это и упомянули планы по поддержке WebTestClient и WireMock.</p><p>В заключительной части эксперт и ведущие разработали кастомное API для внешних интеграций, предоставляющее информацию о владельцах только при наличии ключевых данных.</p><h3>Настраиваем Spring Security и Keycloak (Павел Кислов)</h3><p>Следующим на очереди был Павел Кислов, эксперт сообщества Spring АйО, который помог настроить Spring Security и интегрировать Keycloak для управления доступом.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b57b87bb36d4b3e77323" data-style="" id="66e4b57b87bb36d4b3e77323" width=""></div><ul><li><p><a href="https://youtu.be/CYfzwTt755Q?feature=shared">Смотреть на YouTube</a></p></li><li><p><a href="https://rutube.ru/video/0da9e56741b314bffba8305a16c47fe2/?r=wd">Смотреть на Rutube</a></p></li></ul><p>Ребята из Amplicode приступили к решению вопросов безопасности, пригласив для помощи Пашу Кислова. Вместе они проанализировали текущую и планируемую инфраструктуры, включающую legacy админку на Thymeleaf для менеджеров и администраторов, API для внешних интеграций и стандартизированное CRUD API для админки на React Admin. Было решено использовать FormLogin для legacy UI из-за небольшого количества пользователей, а для остальной системы внедрить OpenID/OAuth2 на базе KeyCloak. </p><h3>Admin UI с использованием React Admin для Spring Boot (Александр Шустанов)</h3><p>В завершении митапа Александр Шустанов (Product Manager Amplicode) реализовал современный AdminUI с помощью React Admin прямо в VS Code, добавив последний штрих в приложении.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b580315cc3d4a168fc4c" data-style="" id="66e4b580315cc3d4a168fc4c" на Rutube</a></p></li></ul><p>В процессе создания админки на React Admin ребята затронули довольно много интересных вопросов, касающихся как технологий, которые уже поддерживает Amplicode Fullstack, так и тех, которые только планирует поддержать. </p><h3>Полная запись мероприятия и ответы на вопросы</h3><p>Для тех, кто пропустил митап или хочет пересмотреть его целиком, мы также опубликовали полную запись вместе с сессией вопросов и ответов.</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/66e4b585d1b5b66d8d21514b" data-style="" id="66e4b585d1b5b66d8d21514b"
