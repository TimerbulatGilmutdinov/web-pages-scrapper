xmlns="http://www.w3.org/1999/xhtml"><p>Недавно состоялся <a href="https://t.me/spring_aio/278">релиз Spring Framework 6.2</a>, вслед за ним последовал релиз Spring Boot 3.4.</p><p>Команда <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> подготовила перевод статьи Josh Long об основных изменениях и улучшениях, которые вошли в последний релиз.</p><hr/><p>Этот релиз объединяет все нововведения вместе. Spring Boot упрощает интеграцию всех Spring проектов и старается решить возникающие проблемы. Кроме того, он предоставляет удобные инструменты для пользователей Spring фреймворков.</p><p>Например: когда мы добавили поддержку GraalVM native image в Spring Framework 6 и Spring Boot 3, это происходило в несколько этапов. Сначала в Spring Framework была реализована модель компонентов, жизненный цикл и базовые SPI. Затем эта модель позволила другим проектам на базе Spring Framework (например, Spring Data, Spring Security, Spring Batch, Spring Integration и др.) обеспечить поддержку GraalVM для своих решений. Наконец, Spring Boot объединил всё это в одно целое, добавив интеграцию с GraalVM для себя и сторонних проектов, предложив плагины для инструментов сборки и организовав процессы, необходимые ещё на этапе компиляции, до запуска вашего кода. Spring Boot также поддерживает репозиторий метаданных для GraalVM и Buildpacks для контейнеризации GraalVM native image. Когда всё сделано правильно, результат впечатляет — именно поэтому Spring Boot так популярен.</p><p>В этом релизе снова полно удивительных возможностей! Давайте, как обычно, начнем с обзора нового и интересного из <a href="https://github.com/spring-projects/spring-boot/wiki/Spring-Boot-3.4-Release-Notes">release notes</a>, а затем углубимся в полюбившиеся именно мне функции.</p><ul><li><p>Улучшен и упрощён процесс автоматической настройки для реализаций <code>HttpRequestFactory</code>, которые подключаются к <code>RestClient</code> или <code>RestTemplate</code>. Если у вас Apache HTTP Components, будет использоваться <code>HttpComponentsClientHttpRequestFactory</code>. Если есть Jetty, то будет выбран <code>JettyClientHttpRequestFactory</code>. Если и Jetty нет, но есть Reactor Netty, то будет использоваться <code>ReactorClientHttpRequestFactory</code>. Если ничего из этого нет, но вы используете более современные версии JDK, то будет работать <code>JdkClientHttpRequestFactory</code>. Если же нет ничего из перечисленного, то будет применён <code>SimpleClientHttpRequestFactory</code>, который, по сути, представляет собой <code>HttpURLConnection</code> из JDK. Не лучший вариант! Кстати, в этом релизе также изменено поведение по умолчанию: раньше использовался <code>HttpURLConnection</code>, даже если JVM поддерживала <code>HttpClient</code></p></li><li><p>Появился новый <code>ClientHttpRequestFactoryBuilder</code>, который позволяет создавать эти реализации программно и согласованно</p></li><li><p>Все клиенты теперь по умолчанию следуют редиректам, но вы можете отключить эту функцию</p></li><li><p>Приложения теперь завершают работу "аккуратно" по умолчанию (graceful shutdown). В Spring Boot такая возможность существует уже несколько лет. Основная идея в том, что Spring Boot завершает работу сразу после того, как оркестратор контейнеров, платформа или оператор отправляет команду завершения (<code>SIGTERM</code>) JVM. При этом текущим операциям даётся время завершиться в течение настраиваемого периода. После этого приложение останавливается. В течение этого времени оно отклоняет новые HTTP-запросы и т.д.</p></li><li><p>Теперь вы можете использовать аннотацию <code>@AutoConfigureTestDatabase</code> с контейнерами, явно не указывая, что вы не хотите заменять тестовую базу данных, как это было ранее с <code>@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)</code>.</p></li><li><p>В этом релизе добавлена более гибкая настройка видимости эндпоинтов Actuator</p></li><li><p>Базовая версия для этого релиза — HTMLUnit 4.3, которая теперь располагается по другой Maven ссылке. Обновите свой код соответствующим образом.</p></li><li><p>Добавлена поддержка структурного логирования с встроенной поддержкой Elastic Common Schema (ecs), Graylog Extended Log Format (gelf) и Logstash (logstash). Чтобы включить структурное логирование в файлы, установите параметр <code>logging.structured.format.file</code> в значение <code>ecs</code>, <code>gelf</code> или <code>logstash</code>. Аналогично, чтобы включить структурное логирование в консоль, установите <code>logging.structured.format.console</code>.</p></li><li><p>Теперь возможно отправлять OTLP-spans через gRPC</p></li><li><p>Для подключения к кластеру Couchbase теперь можно использовать клиентские сертификаты вместо традиционной аутентификации с помощью имени пользователя и пароля</p></li><li><p>Теперь переменные FreeMarker, используемые автоматически настроенным объектом FreeMarker’s Configuration, можно настраивать</p></li><li><p>Поскольку ActiveMQ Classic снова поддерживает встроенный брокер сообщений, автонастройка была обновлена, чтобы это учитывать. (Встроенный ActiveMQ снова доступен, если вы добавите зависимость <code>org.apache.activemq:activemq-broker</code>!)</p></li><li><p>Появился новый механизм для документирования и уведомления о том, что автонастройка устарела: <code>META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.replacements</code></p></li><li><p><code>OtlpMeterRegistry</code> и Undertow теперь корректно работают с виртуальными потоками</p></li><li><p>Spring Boot теперь по умолчанию использует <code>paketobuildpacks/builder-jammy-java-tiny</code>. Этот сборщик поддерживает платформы ARM и x64 из коробки</p></li><li><p>Docker Compose теперь поддерживает несколько конфигурационных файлов Docker Compose, а также позволяет передавать пользовательские параметры в командную строку при запуске Docker Compose файлов</p></li><li><p>Эндпоинты Spring Boot Actuator теперь отображают больше информации о SSL-бандлах</p></li><li><p>Дополнительная информация о запланированных задачах доступна по эндпоинту <code>/actuator/scheduledtasks</code></p></li></ul><p>Так много новых функций! С чего же начать? Я не был уверен, поэтому решил сосредоточиться не на коде, а на работе приложения в рантайме, особенно на двух функциях: поддержке GraalVM native images в build-packs на Apple Silicon и "аккуратном" завершении работы (graceful shutdown). Давайте остановимся на "аккуратном" завершении. Здесь не так уж много сложного.</p><p>Посмотрим на один небольшой, но полезный класс, который выполняет многие задачи правильно.</p><pre><code class="java">package com.example.bootiful_34.boot;

import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.ResponseBody;
import org.springframework.web.client.RestClient;

@Controller
@ResponseBody
class GracefulController {

	private final RestClient http;

	GracefulController(RestClient.Builder http) {
		this.http = http.build();
	}

	@GetMapping("/delay")
	String delay() {
		return this.http.get().uri("https://httpbin.org/delay/5").retrieve().body(String.class);
	}

}
</code></pre><p>Во-первых, этот класс использует обновлённый автонастраиваемый <code>RestClient</code> builder, который появился в Spring Boot 3.4. Я использую Java 21+ (точнее, Java 23), так что у меня доступен новый <code>JdkClientHttpRequestFactory</code>, основанный на <code>java.net.http.HttpClient</code>. Неплохо начали, дальше будет только интереснее!</p><p>Каждый запрос к этому контроллеру вызывает HTTP-запрос на сайт HTTPBin. HTTPBin специально задерживает ответ на столько секунд, сколько мы укажем в запросе. В нашем случае — 5 секунд. Это долго! И не забывайте, что мы используем контейнер сервлетов, а значит, каждый запрос требует отдельного потока. Вы вызываете <code>localhost:8080/delay</code> (помните, что, возможно, нужно будет войти в систему, так как мы настроили защиту через Spring Security ранее...), и запрос будет блокироваться на целых пять секунд. Поток просто будет простаивать, тратя время впустую, а таких потоков по умолчанию очень мало. Но, к счастью, это Spring Boot, и благодаря простой настройке (<code>spring.threads.virtual.enabled=true</code>), мы можем включить виртуальные потоки Java 21.</p><p>Теперь, когда пользователь делает запрос, RestClient отправляет его, и пока запрос ждёт эти пять секунд, JVM автоматически снимает его с потока операционной системы и помещает в режим ожидания. Почему это важно? Потому что теперь кто-то другой может использовать этот поток в это время! Это значительно улучшает масштабируемость, без лишних затрат. И это не новая функция — её можно было использовать ещё с Spring Boot 3.2.</p><p>Ещё одно преимущество в этой ситуации в том, что каждый запрос требует времени для выполнения. Пять секунд плюс задержка сети. Что произойдёт, если оркестратор, например Kubernetes или Cloud Foundry, решит завершить приложение? Мы всё ещё ждём завершения запроса! Поэтому JVM тоже подождёт. По умолчанию это 30 секунд, но можно настроить с помощью <code>spring.lifecycle.timeout-per-shutdown-phase=30s</code>. То есть, если платформа завершает работу приложения (это можно симулировать, остановив приложение в IntelliJ IDEA), она будет ждать до 30 секунд, чтобы завершить все текущие запросы, а потом остановится. Иначе приложение завершится немедленно. Удобно.</p><p>Также есть улучшение для пользователей Apple Silicon — поддержка <a href="https://buildpacks.io/">Buildpacks</a> теперь корректно работает на этой архитектуре.</p><p>Теперь давайте превратим это приложение в нативный образ GraalVM. Выполните следующую команду в корневой папке <code>bootiful-34</code>.</p><pre><code class="bash">./mvnw -DskipTests clean -Pnative spring-boot:build-image</code></pre><p>Вот скрипт, который я использую, чтобы скомпилировать, а затем запустить на своем ноутбуке Apple M4 с процессором Apple Silicon.</p><pre><code class="bash">#!/usr/bin/env bash

rm -rf target

./mvnw -DskipTests spring-javaformat:apply 

./mvnw -DskipTests -Pnative spring-boot:build-image </code></pre><p>Вот скрипт, который я использую, чтобы запустить его. Я предполагаю, что у вас уже установлен Docker или что-то похожее, и что у вас настроена переменная окружения (как у меня) в операционной системе хоста, которая содержит ключ API OpenAI с именем <code>SPRING_AI_OPENAI_API_KEY</code>.</p><pre><code class="bash">#!/usr/bin/env bash

export SPRING_DATASOURCE_URL=jdbc:postgresql://host.docker.internal/mydatabase

docker run \

  -e SPRING_DATASOURCE_URL=$SPRING_DATASOURCE_URL \

  -e SPRING_AI_OPENAI_API_KEY=$SPRING_AI_OPENAI_API_KEY \

  docker.io/library/bootiful-34:0.0.1-SNAPSHOT</code></pre><p>Знаете, что меня поражает? Иногда, по крайней мере на моем компьютере, Linux-контейнер с бинарным файлом GraalVM для Apple Silicon работает быстрее, чем если бы я его просто скомпилировал в macOS-совместимый бинарный файл и запустил  на macOS! Понятия не имею, как это происходит! Это просто универсальное решение! У нас есть веб-сервер, Spring Batch <code>Job</code>, несколько потоков Spring Integration, множество обращений к SQL <code>DataSource</code>, Spring Modulith, Spring AI, который вызывает функции локально, три типа безопасности и полноценный OAuth IDP (Spring Authorization Server), HTTP эндпоинты, сущности Spring Data JDBC и репозитории, и многое другое. Так что поверьте мне, когда я говорю, что это нестандартная нагрузка, и я не ожидал, что бинарник под Linux на моем компьютере будет работать быстрее. На macOS приложение иногда запускается за 0,450 — 0,5 секунды. В Docker я видел, как оно запускалось за 0,264 секунды. Ума не приложу, как так может быть!</p><p>Цифры ещё заметнее в  обычных сервисах. У меня есть более простое приложение в том же репозитории GitHub, которое называется <code>demo</code>. <code>demo</code> использует Spring Data JDBC для общения с PostgreSQL и Spring MVC. Оно ничего особенного не делает. Версия на Buildpacks запускается примерно за 0,050 секунды, а версия для macOS — за 0,080 секунды! Это не  в два раза быстрее, но всё равно  разница заметная! Обожаю Buildpacks! И так приятно тестировать все на локальной машине.</p><p>Spring Boot — это отличная основа для других наших проектов. Уже выпущены два проекта с поддержкой Spring Boot 3.4: Spring Modulith и Spring AI. О них я расскажу здесь. Ещё один проект, который базируется на Spring Boot 3.4 — Spring Cloud — скоро тоже будет выпущен.</p><p>Spring Boot 3.4 — это просто невероятно. Помните, что этот релиз — один из последних перед выпуском Spring Boot 4.0 в следующем году, так что при обновлении обращайте внимание на устаревшие функции. Лучше разобраться с ними прямо сейчас!</p><figure class="full-width"><img data-src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/725/c0c/ddb/725c0cddbe78ce9d7a9dd868c3632ad6.png"
