id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-blurred="true" data-src="https://habrastorage.org/getpro/habr/upload_files/3e9/f8f/b1f/3e9f8fb1fe52a6f2e8529872e29abb59.jpg" height="720" src="https://habrastorage.org/r/w780q1/getpro/habr/upload_files/3e9/f8f/b1f/3e9f8fb1fe52a6f2e8529872e29abb59.jpg" width="1280"/></figure><p>Room (абстракция над SQLite) — одна из основных технологий, используемая почти во всех Android-приложениях для кэширования данных, оффлайновости, как cross-process хранилище данных и тому подобное.</p><p>При этом скорость работы приложения обычно напрямую зависит от скорости работы с БД, особенно если речь идет о холодном старте, когда все данные хранятся в кэше (в БД).</p><p>Встает вопрос: “<strong>А не является ли сама работа с БД узким местом скорости старта приложения?</strong>”</p><p>В Wildberries это особенно актуально, так как приложение построено на парадигме offline-first, когда почти вся информация кэшируется в БД, чтобы приложение работало даже с медленным интернетом или без него.</p><p>Для ответа на этот вопрос в статье разберем рантайм реализацию автоматического трекинга скорости выполнения запросов и транзакций в Room Database на основе Java Dynamic Proxy.</p><a class="anchor" id="1" name="1"></a><h2>Мотивация</h2><p>Профилировщик Android Studio не даёт ответ на вопрос о производительности БД, так как:</p><ul><li><p>содержит слишком много ненужной информации (скорость работы всех методов, не только БД)</p></li><li><p>работает только локально, а не у конечных пользователей (а как мы знаем:  пользователи попадают в такие ситуации, которые тестировщикам только  снятся)</p></li></ul><p>Основные проблемы производительности БД связаны с тем, что по сути вся работа с ней является синхронной очередью — <a href="https://www.sqlite.org/atomiccommit.html#_acquiring_a_read_lock"><u>чтение блокирует запись и наоборот</u></a>. <a href="https://www.sqlite.org/draft/wal.html"><u>Write-Ahead Logging (WAL)</u></a> позволяет обойти это, создавая буфер для записей и периодически скидывая их в БД, но при этом технология подвержена другим проблемам: каждая N-ная запись приводит к тормозам (из-за сброса буфера), сам буфер при определенных условиях может бесконтрольно расти и т.п..</p><p>Среди очевидных проблем производительности БД можно выделить:</p><ol><li><p>в <code>database.withTransaction{}</code> выполняется долгий/блокирующий код (например, лок мьютексов, сетевые запросы, тяжелые мэппинги или cross-database взаимодействия),</p></li><li><p>множество чтений/записей, которые выгоднее было бы выполнить в одной транзакции,</p></li><li><p>множество чтений одних и тех же данных (которые можно закэшировать в рантайме),</p></li><li><p>тяжелые единичные транзакции, которые можно разбить на более мелкие, чтобы пропустить вперед более приоритетные запросы.</p></li></ol><a class="anchor" id="2" name="2"></a><h2>Цель</h2><p>Научиться узнавать о проблемах производительности БД, а именно:</p><ul><li><p>Получать информацию как о транзакциях, так и об обычных DAO-запросах, сгенерированных Room.</p></li><li><p>В информации должны быть данные о том, что именно замерялось, например, <code>userDao_all</code> для запросов к DAO (так как мы имеем имя DAO и название метода запроса) или <code>UserWorker_runUpdate</code> в случае если был вызван <code>db.withTransaction</code> из метода <code>runUpdate</code> в классе <code>UserWorker</code> (так как транзакции не имеют своего имени).</p></li><li><p>Информацию хочется получать и локально, и с реальных пользователей.</p></li><li><p>По возможности механизм должен работать без больших переделок существующего кода работы с БД.</p></li></ul><a class="anchor" id="3" name="3"></a><h2>Целевой код</h2><p>Для замеров скорости будем использовать абстрактный <code>Profiler.Trace</code>, который может отправлять данные в firebase, логкат или куда-либо ещё:</p><pre><code class="kotlin">interface Profiler {
   fun startTrace(name: String): Trace
   interface Trace {
       fun stop()
   }
}</code></pre><p>В качестве Room БД возьмем базу пользователей:</p><pre><code class="kotlin">@Entity
data class User(
   @PrimaryKey val uid: Int,
)

@Database(entities = [User::class], version = 1)
abstract class AppDatabase : RoomDatabase() {
   abstract fun userDao(): UserDao
}

@Dao
interface UserDao {
   @Query("SELECT * FROM user")
   fun observe(): Flow&lt;List&lt;User&gt;&gt;

   @Query("SELECT * FROM user")
   suspend fun all(): List&lt;User&gt;

   @Insert(onConflict = REPLACE)
   fun insert(user: User): Long
}</code></pre><p>И как она используется в проекте:</p><pre><code class="kotlin">// Настройка
val db = Room.databaseBuilder(
   appContext, AppDatabase::class.java, "app-db"
).build()

class Worker {
   suspend fun payload(db: AppDatabase) {
       val userDao = db.userDao()

       // Какие трейсы мы хотим получить на какие вызовы:

       // Должно логировать "UserDao_insert"
       userDao.insert(User(1))

       // Должно логировать "UserDao_all"
       userDao.all()

       // Должно логировать "UserDao_observe", но только для самого первого элемента в отдельном collect{}
       userDao.observe().first()

       // Должно логировать "Worker_payload", так как транзакции не имеют имени - берем имя вызывающего
       db.withTransaction { userDao.all() }
   }
}
</code></pre><a class="anchor" id="4" name="4"></a><h2>Реализация</h2><p>Трекинг можно реализовать двумя способами — в рантайме и на уровне компиляции.<br/>Остановимся на рантайм реализации, так как:</p><ul><li><p>такая реализация не имеет значительного оверхэда по сравнению с ручными замерами,</p></li><li><p>байткод манипуляции сложны в плане написания и поддержки (для этого нужен старший+ разработчик).</p></li></ul><a class="anchor" id="4.1" name="4.1"></a><h3>Автотрекинг транзакций: db.withTransaction{}</h3><p>Идея основывается на том, что работа с транзакциями почти всегда идёт через единственный библиотечный метод-экстеншн:</p><pre><code class="kotlin">package androidx.room

suspend fun &lt;R&gt; RoomDatabase.withTransaction(block: suspend () -&gt; R): R</code></pre><p>При этом в приложениях обычно используются наследники БД (у нас это <code>AppDatabase</code>), а не <code>RoomDatabase</code> напрямую. Значит, мы можем сделать автотрекинг за 3 шага:</p><p>1. Создать аналог библиотечного метода в том же пакете, но как экстеншн к собственному интерфейсу (<code>ProfiledDatabase</code>).</p><pre><code class="kotlin">suspend fun &lt;R&gt; ProfiledDatabase.withTransaction(block: suspend () -&gt; R): R {
   // Что это и как работает - позднее в статье
   val profiler = (this as ProfilerProvider).profiler()

   val trace = profiler.startTrace(transactionName())
   val result = (this as RoomDatabase).withTransaction(block)
   trace.stop()
   
   return result
}</code></pre><p>2. Наследовать этот новый интерфейс во всех БД приложения, изменив иерархию.</p><pre><code class="kotlin">interface ProfiledDatabase

// Выделили отдельный интерфейс со всеми Dao
interface AppDatabase : ProfiledDatabase {
   abstract fun userDao(): UserDao
}

@Database(entities = [User::class], version = 1)
abstract class RoomAppDatabase : RoomDatabase(), AppDatabase</code></pre><p>Теперь вызовы <code>AppDatabase.withTransaction{}</code> будут приводить к вызову нашего <code>ProfiledDatabase.withTransaction{}</code> вместо <code>RoomDatabase.withTransaction{}</code>.</p><p>3. Получать класс и имя метода, в которых был вызван <code>withTransaction</code>. </p><p>Идея в том, чтобы у потока получить стектрейс, через который сможем найти информацию о том, какой код вызвал <code>withTransaction</code>.</p><pre><code class="kotlin">@Suppress("NOTHING_TO_INLINE") // inline для корректности stackTrace
internal inline fun transactionName(): String? {
   // Получаем стектрейс текущего потока
   val stackTrace = Thread.currentThread().stackTrace

   // Отбрасываем все вызовы библиотеки замера производительности,
   // а так же dalvik что появляется при дебаге
   val stackIndex = (if (stackTrace[0].className.startsWith("dalvik")) 3 else 2)
  
   // Получаем информацию о коде, вызвавшем withTransaction
   val caller = stackTrace.getOrNull(stackIndex)
       ?: return null

   // Нас интересует имя класса без пакета
   val className = caller.className.substringAfterLast('.')

   // Имена сгенерированных классов/методов/лямбды нужно подчистить перед использованием
   return if (className.contains('$')) {
       // Class name:
       // UserRepository$localUser$2$$inlined$$invokeSuspend
       val tokens = className.splitToSequence('$').iterator()
       "${tokens.nextOrNull()}_${tokens.nextOrNull()}"
   } else {
       //
       // Class name: DatabasePerformanceTrackerTest
       // Method name: withTransaction_logged__when_called_from_local_function$lambda$17$doLocalTransaction
       "${className}_${caller.methodName.substringBefore('$')}"
   }
}</code></pre><a class="anchor" id="4.2" name="4.2"></a><h3>Автотрекинг запросов (query)</h3><p><code>AppDatabase</code> и <code>UserDao</code> — интерфейсы, а значит можно использовать механизм <a href="https://docs.oracle.com/javase/8/docs/technotes/guides/reflection/proxy.html"><u>java dynamic Proxy</u></a>, который позволяет через рефлексию <em>в рантайме</em> имплементировать интерфейсы (например, его использует <a href="https://www.youtube.com/watch?v=FGYbjBsCZ7c"><u>Retrofit</u></a>).<br/>Для этого нужно реализовать <code>InvocationHandler</code>, который будет перехватывать все вызовы методов к Proxy:</p><pre><code class="kotlin">public interface InvocationHandler {
   public Object invoke(Object proxy, Method method, Object[] args)
       throws Throwable;
}</code></pre><p>То есть, мы сможем создать Proxy <code>AppDatabase</code>, который будет создавать Proxy <code>UserDao</code>, который в свою очередь будет для вызовов всех методов DAO автоматически стартовать и завершать трейсы.</p><p>Note: Несмотря на то, что Proxy использует рефлексию, в актуальных андроидах оверхеда почти нет (порядка 0.01мс на один вызов, тогда как вызовы БД обычно занимают 1мс+).</p><a class="anchor" id="4.2.1" name="4.2.1"></a><h4>Proxy AppDatabase &amp; UserDao</h4><p>Сначала создадим Proxy над <code>AppDatabase</code> интерфейсом:</p><pre><code class="kotlin">inline fun &lt;reified TDB : ProfiledDatabase&gt; RoomDatabase.withProfiler(profiler: Profiler) : TDB {
   val dbInterface = TDB::class.java
   require(dbInterface.isInstance(this))

   return Proxy.newProxyInstance(
       this::class.java.classLoader,
       arrayOf(dbInterface, ProfilerProvider::class.java),
       DatabaseInvocationHandler(
           db = this,
           profiler = profiler,
       ),
   ) as TDB
}</code></pre><p>Это позволит одним вызовом <code>db.withProfiler(profiler)</code> включать автотрекинг производительности любой <code>RoomDatabase</code> в проекте (после того, как <code>AppDatabase</code> переделан на интерфейс).</p><p>Тут мы объявили, что наша Proxy будет наследовать <code>AppDatabase</code>, чтобы создавать Proxy для отдельных DAO, а также <code>ProfilerProvider</code>, через который мы в транзакциях будем получать доступ до <code>Profiler</code> для создания трейсов.</p><p>Далее реализация самой прокси для БД:</p><pre><code class="kotlin">class DatabaseInvocationHandler(
   private val db: RoomDatabase,
   private val profiler: Profiler,
) : InvocationHandler {
   // Кэш всех настоящих прокси-Dao (UserDao и тому подобных)
   private val daos = ConcurrentHashMap&lt;Method, Any&gt;()

   // Здесь и далее @Throws нужен для устранения проблем несовместимости
   // работы с исключениями у Kotlin и Java, в частности UndeclaredThrowableException.
   @Throws(Throwable::class)
   override fun invoke(proxy: Any, method: Method, args: Array&lt;*&gt;?): Any? {
       // Если запросили profiler, а не dao - возвращаем его.
       // За счет этого работает withTransaction.
       if (method.name == ProfilerProvider::profiler.name) {
           return profiler
       }

       return daos.computeIfAbsent(method) {
           // Если в кэше нет нужного Dao - получаем его, оборачиваем в прокси и кэшируем
           val realDao = method.invokeSafe(db, args)
           // Мы не можем использовать realDao::class, так как он будет сгенерирован Room - UserDao_Impl
           val daoType = method.returnType
           daoType.createDaoProxy(realDao)
       }
   }

   private fun Class&lt;*&gt;.createDaoProxy(realDao: Any) = Proxy.newProxyInstance(
       DatabaseInvocationHandler::class.java.classLoader,
       arrayOf(this),
       DaoInvocationHandler(
           daoName = simpleName,
           realDao,
           profiler,
       ),
   )
}</code></pre><p>Здесь мы создаем (и кэшируем) Proxy DAO для каждого метода в <code>AppDatabase</code> (в нашем случае <code>userDao()</code>, а также возвращаем <code>profiler</code> для поддержки <code>withTransaction</code>.</p><p>Для вызовов настоящих методов из Proxy используем специальную обертку над <code>invoke</code>, которая умеет разворачивать оберточные Kotlin-исключения. Это связано с тем, что Kotlin и Java имеют разные системы исключений и, так как Proxy работает на уровне Java рефлексии, необходимо быть особенно осторожным:</p><pre><code class="kotlin">@Throws(Throwable::class)
fun Method.invokeSafe(obj: Any, args: Array&lt;*&gt;?) =
   try {
       invoke(obj, *(args ?: emptyArray()))
   } catch (e: InvocationTargetException) {
       throw e.targetException ?: e.cause ?: e
   }</code></pre><p>Сама DAO Proxy:</p><pre><code class="kotlin">class DaoInvocationHandler(
   private val daoName: String,
   private val dao: Any,
   private val profiler: Profiler,
) : InvocationHandler {

   @Throws(Throwable::class)
   override fun invoke(proxy: Any, method: Method, args: Array&lt;*&gt;?): Any? {
       // Все методы реализуем позднее
       return when {
           isSuspend(args) -&gt; handleSuspendMethod(method, args)
           isObserver(method) -&gt; handleObserverMethod(method, args)
           else -&gt; handleSyncMethod(method, args)
       }
   }

   fun isSuspend(args: Array&lt;*&gt;?): Boolean = TODO()
   @Throws(Throwable::class)
   fun handleSuspendMethod(method: Method, args: Array&lt;*&gt;?): Any = TODO()

   fun isObserver(method: Method): Boolean = TODO()
   @Throws(Throwable::class)
   fun handleObserverMethod(method: Method, args: Array&lt;*&gt;?): Any = TODO()

   @Throws(Throwable::class)
   fun handleSyncMethod(method: Method, args: Array&lt;*&gt;?): Any = TODO()</code></pre><p>Всего есть 3 типа методов, которые нужно обработать отдельно:</p><a class="anchor" id="4.2.2" name="4.2.2"></a><h4>UserDao.insert()</h4><p>Сигнатура <code>insert()</code> проста — это обычный синхронный метод:</p><pre><code class="kotlin">@Insert(onConflict = REPLACE)
fun insert(user: User): Long</code></pre><p>Поэтому <code>invokeSafe()</code> справится с ним сам:</p><pre><code class="kotlin">@Throws(Throwable::class)
fun handleSyncMethod(method: Method, args: Array&lt;*&gt;?): Any {
   val trace = profiler.startTrace("${daoName}_${method.name}")

   return method.invokeSafe(dao, args)
       .also { trace.stop() }
}</code></pre><a class="anchor" id="4.2.3" name="4.2.3"></a><h4>UserDao.observe()</h4><p><code>observe()</code> хоть и обычный синхронный метод, но возвращает <code>Flow</code>.</p><pre><code class="kotlin">@Query("SELECT * FROM user")
fun observe(): Flow&lt;List&lt;User&gt;&gt;</code></pre><p>Так как нам важно слушать каждый отдельный <code>collect</code> (пользователь может получить вызвать <code>observe</code> один раз, но переиспользовать полученный <code>Flow</code> в разных местах), то единственный подходящий вариант — через наследование:</p><pre><code class="kotlin">fun isObserver(method: Method): Boolean =
   method.returnType == Flow::class.java

@Throws(Throwable::class)
fun handleObserverMethod(method: Method, args: Array&lt;*&gt;?): Any {
   val dbFlow = method.invokeSafe(dao, args) as Flow&lt;*&gt;

   return object : Flow&lt;Any?&gt; {
       override suspend fun collect(collector: FlowCollector&lt;Any?&gt;) {
           val trace = profiler.startTrace("${daoName}_${method.name}")

           dbFlow
               .onEach { trace.stop() }
               .collect(collector)
       }
   }
}</code></pre><p>Мы замеряем только производительность первого элемента потому что:</p><ul><li><p>Время между <code>collect</code> и первым элементов включает в себя время создания <code>SQLStatement</code>, что также важная часть производительности БД.</p></li><li><p>Если замерять выдачу всех элементов, то количество данных будет слишком большим.</p></li></ul><a class="anchor" id="4.2.4" name="4.2.4"></a><h4>UserDao.all()</h4><p><code>all()</code> — это suspend метод, которых как таковых не существует в Java.</p><pre><code class="kotlin">@Query("SELECT * FROM user")
suspend fun all(): List&lt;User&gt;</code></pre><p>Так как Java Proxy ничего не знает о suspend, необходимо вручную научить его. Для этого нужно разобраться с тем, как работает suspend с точки зрения кодогенерации Kotlin в Java.</p><a class="anchor" id="4.2.4.1" name="4.2.4.1"></a><h3>Корутины в Java</h3><p>Код, который будем декомпилировать:</p><pre><code class="kotlin">class Worker {
   suspend fun payload1() { println("1") }
   suspend fun payload2() { println("2") }

   suspend fun doWork() {
       payload1()
       payload2()
   }
}</code></pre><p>После декомпиляции сигнатура suspend метода <code>doWork</code> в Java выглядит так:</p><pre><code class="kotlin">public final Object doWork(Continuation parentContinuation)</code></pre><p>В сигнатуру последним параметром был добавлен <code>Continuation</code> — “колбэк”, который будет вызван по окончанию выполнения метода <code>doWork</code>. Это позволяет методу прерывать свою работу на полпути, переключаться между потоками или блокироваться на другие suspend методы.<br/><em>На самом деле, </em><code>Continuation</code><em> вызывается не всегда, об этом далее.</em></p><p>Основная часть java реализации <code>doWork</code> выглядит так (это вручную переписанный декомпилированный код для упрощения понимания):</p><pre><code class="kotlin">public final Object doWork(Continuation parentContinuation) {
   // Continuation создается и хранится для каждого отдельного вызова doWork.
   // Для простоты, опускаем реализацию хранения doWork_continuation.
   if (doWork_continuation == null) {
       doWork_continuation = new ContinuationImpl(parentContinuation) {
           int label = 0;

           @Nullable
           public Object invokeSuspend(Object omitted) {
               return doWork(this);
           }
       };
   }

   // О том, что это такое - позднее
   Object SUSPENDED = IntrinsicsKt.getCOROUTINE_SUSPENDED();

   switch (doWork_continuation.label) {
       case 0:
           doWork_continuation.label = 1;
           if (payload1(doWork_continuation) == SUSPENDED) {
               return SUSPENDED;
           }
           // Нет break. Если не было suspend - проваливаемся в следующий case
       case 1:
           doWork_continuation.label = 2;
           if (payload2(doWork_continuation) == SUSPENDED) {
               return SUSPENDED;
           }
           // Нет break. Если не было suspend - проваливаемся в следующий case
       case 2:
           return Unit.INSTANCE;
   }
}</code></pre><p>Методы payload вызываются с Continuation-оберткой, которая ведет на текущий <code>doWork</code> метод. Он в свою очередь восстанавливает свою работу за счет <code>label</code>, пропуская с его помощью уже выполненную часть кода. </p><p>То есть, по сути метод <code>doWork</code> может быть неявно вызван 3 раза:</p><ul><li><p>самый первый вызов,</p></li><li><p>вызов после окончания <code>payload1</code>,</p></li><li><p>вызов после окончания <code>payload2</code>.</p></li></ul><p>Но есть нюанс. Suspend методы могут быть не suspend. Причины могут быть разные, но это приводит к тому что <code>Continuation</code> может быть не вызван. Например, так выглядит метод <code>payload1</code>:</p><pre><code class="kotlin">public final Object payload1(Continuation completion) {
   System.out.println("1");
   return Unit.INSTANCE;
}</code></pre><p><code>Continuation</code> тут не вызывается, а сразу возвращается результат, ведь внутри <code>payload1</code> нет ничего, что могло бы вызвать прерывание метода.</p><p>Такие ситуации обрабатываются за счет специального объекта — <code>Intrinsics.COROUTINE_SUSPENDED</code>. Если suspend функция его вернет, код снаружи знает, что произошло прерывание выполнения, например, смена потока, и нужно выходить из текущего стека вызовов, так как продолжение выполнения со следующего лейбла будет через <code>Continuation.invokeSuspend</code>.</p><a class="anchor" id="4.2.4.2" name="4.2.4.2"></a><h3>Корутины в Proxy</h3><p>В итоге мы получаем следующую реализацию трекинга производительности:</p><pre><code class="kotlin">fun isSuspend(args: Array&lt;*&gt;?): Boolean =
   args?.lastOrNull() as? Continuation&lt;Any?&gt;

@Throws(Throwable::class)
fun handleSuspendMethod(method: Method, args: Array&lt;*&gt;?): Any {
   val trace = profiler.startTrace("${daoName}_${method.name}")

   val continuation = args?.lastOrNull() as? Continuation&lt;Any?&gt;
   // В случае если внутри метода будет прерывание, оборачиваем
   // старый Continuation в наш новый, добавляя остановку трейса.
   val newContinuation = continuation.withBefore {
       trace.stop()
   }

   val newArgs = args.dropLast(1) + newContinuation
   method.invokeSafe(dao, newArgs.toTypedArray()).also {
       if (it != COROUTINE_SUSPENDED) {
           // Если же прерывания не случилось и метод отработал синхронно -
           // останавливаем трейс сразу.
           trace.stop()
       }
   }
}</code></pre><p>Самая большая проблема в этой логике в <code>withBefore</code> — это самописный метод, использующий hidden coroutine api:</p><pre><code class="kotlin">private fun Continuation&lt;Any?&gt;.withBefore(
   action: (Result&lt;Any?&gt;) -&gt; Unit,
) = object : Continuation&lt;Any?&gt; {
   override val context = this@withBefore.context
   override fun resumeWith(result: Result&lt;Any?&gt;) {
       action(result)

       /**
        * Room внутри использует [Executor.asCoroutineDispatcher], который является
        * [ContinuationInterceptor] и отвечает за создание [DispatchedContinuation].
        * [DispatchedContinuation] использует скрытое api [ContinuationImpl.intercepted]
        * для обеспечения редиспатчинга [Continuation] на нужный поток.
        *
        * Здесь мы делаем то, что делает [DispatchedCoroutine.afterResume], используя скрытое api.
        */
       this@withBefore.intercepted().resumeCancellableWith(result)
   }
}</code></pre><p>Связано это с тем, что механизм “интерцептов” в корутинах отвечает за переключение потоков использования такого же механизма,  <code>userDao.all()</code> может продолжить свое выполнение на потоке Room, что сломает приложение.<br/><em>Изначально было опасение что данная конструкция будет часто ломаться, но на данный момент ей уже год и проблем не наблюдалось. Известно, что ещё такая конструкция использовалась в ktor.</em></p><a class="anchor" id="5" name="5"></a><h2>Итог</h2><p>Автотрекинг работает в рантайме, имеет малый оверхед и код достаточно простой для расширения (за исключением “интерцепт”-магии корутин). При этом позволяет отследить все основные проблемы медленной работы БД.</p><p>Как примеры расширений на основе этого механизма, можно упомянуть:</p><ul><li><p>Возможность отслеживать дедлоки (когда вызван <code>trace.start</code>, но не вызван <code>trace.stop</code>).</p></li><li><p>Возможность логировать нонфаталы для слишком долгих транзакций (если не нужны логи обо всех БД операциях).</p></li><li><p>Возможность поддержать DAO запросы с одинаковыми именами, но разными параметрами.</p></li></ul><p>Из недостатков:</p><ol><li><p>Черная магия корутин (есть вероятность что это просто недосмотр, так как в <code>ContinuationKt</code> утилитах есть возможность создавать свои обертки над <code>Continuation</code> с учетом <code>intercepted</code>, нет лишь нужного нам метода).</p></li><li><p>Для интеграции с обфусцированными приложениями нужно делать дополнительные манипуляции на бэкенде (например, выгружать mapping.txt от R8 и де-обфусцировать названия).</p></li><li><p>Код использования БД в приложении почти не нужно менять (нужно только изменить иерархию БД классов и добавить враппер над <code>RoomDatabase</code>), за исключением <code>db.inTransaction</code>. В этом случае придется делать свой метод типа <code>ProfiledDatabase.isInTransaction() = (this as RoomDatabase).inTransaction()</code> и менять код в проекте.</p></li><li><p>Не поддерживаются ручные <code>db.beginTransaction/endTransaction</code> (однако эти методы использовать не рекомендуется, так как высока цена ошибки).</p></li><li><p><code>@Transaction</code> запросы в DAO поддерживаются, но если внутри такого метода используются другие запросы из DAO, они залогированы не будут. Это связано с тем, что, когда Room генерирует имплементацию таких методов, он передает внутрь <code>@Transaction</code> метода настоящий DAO, а не наш Proxy. И обойти это в рантайме нельзя. Но это не является большим недостатком, так как время самого <code>@Transaction</code> метода всё ещё будет логироваться, поэтому проблемы производительности не пройдут незамеченными.</p></li></ol><p></p></div></div></div><!-- --><!-- --></div><!-- --><!-- --></div><!--]--><!-- --><div class="tm-article-presenter__meta" data-test-id="article-meta-links"><div
