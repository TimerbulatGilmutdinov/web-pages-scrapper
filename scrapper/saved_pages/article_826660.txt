article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>В этой статье эксперт сообщества <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> – <a href="https://www.baeldung.com/author/polivakhamikhail">Михаил Поливаха</a> рассмотрит процесс миграции с компилятора Kotlin старой версии на новый компилятор K2. <a href="https://habr.com/ru/companies/spring_aio/articles/823164/">В предыдущей статье</a> Михаил рассмотрел компилятор K2, а здесь сфокусировался только на процедуре миграции.</p><h2>1. Введение</h2><p>В этой статье мы рассмотрим процесс миграции с компилятора Kotlin старой версии на новый компилятор K2. <a href="https://habr.com/ru/companies/spring_aio/articles/823164/"><u>В другой статье</u></a> мы сделали обзор компилятора K2 в общем, а здесь мы сфокусируемся только на процедуре миграции. Далее, под K2, очевидно, мы будем подразумевать новый компилятор, а под K1 - старую версию компилятора, так как его мы будем тоже довольно часто упоминать.</p><h2>2. Процедура миграции</h2><p>K2 не полностью обратно совместим с K1. Нам нужно выполнить некоторые дополнительные шаги, чтобы наш код компилировался на K2. Подробное объяснение миграции описано в <a href="https://kotlinlang.org/docs/k2-compiler-migration-guide.html#changes"><u>официальном руководстве по миграции</u></a>. Здесь же мы просто объясним наиболее важные изменения, которые могут затронуть обычных пользователей.</p><h2>3. Инициализация open свойств</h2><p>K2 требует, чтобы все open свойства с<a href="https://www.baeldung.com/kotlin/backing-fields"> <u>backing полями</u></a> были инициализированы в момент декларации. Раньше компилятор требовал инициализации в месте объявления только для свойств <em>open var</em>. Например:</p><pre><code class="kotlin">open class BaseEntity { 
	open val points: Int 
	open var pages: Long? 
 
	init { 
    	points = 1 
    	pages = 12 
	} 
}</code></pre><p>Такой код теперь компилироваться не будет. Это, конечно, несколько странно, поскольку приведенный выше код и следующий:</p><pre><code class="kotlin">open class BaseEntity { 
	open val points: Int = 1 
	open var pages: Long? = 12  
}</code></pre><p>компилируются в один и тот же Java bytecode - инициализация переменных происходит внутри конструктора в обоих случаях. Тем не менее, один из этих примеров кода компилируется, а другой - нет. Исключением из этого правила являются свойства типа  <code>lateinit open var</code>. Они все еще могут иметь отложенную инициализацию:  </p><pre><code class="kotlin">open class WithLateinit { 
  open lateinit var point: Instant 
}</code></pre><p>K2 успешно скомпилирует приведенный выше фрагмент кода.</p><h2>4. Синтетические сеттеры на Projected типах</h2><p>Чтобы понять это изменение, нам нужно немного уйти в сторону и поговорить о том, какие ограничения могут иметь generic типы.</p><p><strong>4.1 Ограничения generic типов<br/></strong>Предположим, у нас есть следующий код на Java:</p><pre><code class="java">public void add(List&lt;?&gt; list, Object element) { 
	list.add(element); 
}</code></pre><p>Этот код не компилируется. <strong>И не без причины: ссылка типа  <em>List&lt;?&gt;</em> может указывать на объект типа  <em>List&lt;Number&gt;</em>, <em>List&lt;List&lt;Object&gt;&gt;</em>, <em>List&lt;File&gt;</em> и т.д</strong>. Было бы безопасно позволить добавить объект любого конкретного типа к этому списку? Нет, мы не можем добавлять ничего кроме <em>null</em> к этому списку<em>, </em>поскольку на самом деле мы не знаем: какие именно объекты представлены в <em>List&lt;?&gt;</em>. В Kotlin у нас есть star projections, похожие на<a href="https://www.baeldung.com/java-generics-type-parameter-vs-wildcard"> <u>wildcards в Java</u></a>. Поэтому в Kotlin такой код тоже не будет компилироваться: </p><pre><code class="kotlin">fun execute(list: MutableList&lt;*&gt;, element: Any) { 
	list.add(element) 
}</code></pre><p>По той же самой причине. Пока что все понятно.</p><p><strong>4.2 Дефект в компиляторе Kotlin</strong> <br/>Теперь представьте себе, что у нас есть следующий Java класс:</p><pre><code class="java">public class Box&lt;E&gt; { 
 
	private E value; 
 
	public E getValue() { 
    	return value; 
	} 
 
	public void setValue(E value) { 
    	this.value = value; 
	} 
}</code></pre><p>И если мы попытаемся использовать этот Java класс в Kotlin следующим образом:</p><pre><code class="kotlin"> fun explicitSetter() { 
	val box = Box&lt;String&gt;() 
	val tmpBox : Box&lt;*&gt; = box 
	tmpBox.setValue(12) // Compile Error! That's unsafe! 
	val myValue : String? = box.value 
}</code></pre><p>Компилятор также выбросит сообщение об ошибке. <strong>Причина все та же: небезопасно выполнять такую операцию, поскольку</strong> <strong>Box&lt;*&gt;  может содержать все что угодно. </strong>Компилятор спасает нас, поскольку следующая строка вызвала бы ошибку. Но вот такой код:</p><pre><code class="kotlin">fun syntheticSetter() { 
	val box = Box&lt;String&gt;() 
	val tmpBox : Box&lt;*&gt; = box 
	tmpBox.value = 12 // That compiles! 
	val foo : String? = box.value // And here we fail with ClassCastException 
} </code></pre><p>успешно компилировался старым компилятором Kotlin! Хотя такой код компилируется, он всегда вызывает ошибку <em>ClassCastException</em> во время выполнения. Причина состоит в том, что мы использовали синтетический сеттер для поля <em>value</em> через ссылку типа <em>Box&lt;*&gt;</em> чтобы установить значение 12, которое является значением типа <em>Int,</em> объекту box, который в действительности должен содержать тип<em> String</em>. Следовательно, код, содержащий ссылку на <em>Box&lt;String&gt;</em> по праву ожидает, что значение внутри <em>Box</em> будет экземпляром класса <em>String</em>, и компилятор Kotlin, как и <em>javac</em>, добавит bytecode инструкцию cast, когда мы выполним функцию <em>getValue().</em>  И это приведение типов (cast), естественно, терпит неудачу, поскольку значение внутри Box не является <em>String</em>, а является <em>Int</em>. <strong> Kotlin K2 исправляет данный дефект. </strong> Это применимо не только к<a href="https://www.baeldung.com/kotlin/generics#3-star-projections"> <u>start projection type</u></a>, но и к другим projected типам, например контравариантным типам с <a href="https://www.baeldung.com/kotlin/generics#2-the-in-keyword"><u>in-аннотацией</u><em><u>:</u></em></a></p><pre><code class="kotlin">fun syntheticSetter_inVariance() { 
	val box = Box&lt;String&gt;() 
	val tmpBox : Box&lt;in String&gt; = box 
	tmpBox.value = 12 // Wow, thats a trap again! 
	val foo : String? = box.value // Blast! ClassCastException 
} </code></pre><p>Здесь точно такая же проблема, но с контравариантным типом, помеченным <code>in</code> аннотацией. Компилятор K2 не скомпилирует такой код, в то время как K1 не выдаст никаких ошибок.</p><h2>5. Постоянный порядок разрешения (resolution) свойств</h2><p>При использовании Kotlin, мы можем расширять Java классы и наоборот. И может случиться, что как суперкласс, так и базовый класс будут иметь одинаковые поля.  Поскольку Kotlin не позволяет нам объявлять простые поля, а требует использовать вместо них свойства (properties), под “полем” следует понимать стандартное поле языка Java как члена класса, а также backing поле свойства в случае с Kotlin. Таким образом, предположим, что у нас есть 2 класса, один из них базовый Java класс:</p><pre><code class="java">public class AbstractEntity { 
	public String type = "ABSTRACT_TYPE"; 
	public String status = "ABSTRACT_STATUS"; 
}</code></pre><p>А второй дочерний Kotlin класс:</p><pre><code class="kotlin">class AbstractEntitySubclass(val type: String) : AbstractEntity() { 
	val status: String 
    	get() = "CONCRETE_STATUS" 
} 
 
fun main() { 
	val sublcass = AbstractEntitySubclass("CONCRETE_TYPE") 
	println(sublcass.type) 
	println(sublcass.status) 
}</code></pre><p>Если бы мы попытались скомпилировать этот код компилятором K1, у нас бы все получилось, но во время выполнения мы получили бы <em>java.lang.IllegalAccessError</em>. В то же время, компилятор K2 тоже скомпилировал бы такой код успешно, но и во время выполнения не было бы никаких ошибок. Давайте попробуем понять почему.</p><p><strong>5.1 Resolution свойств объекта<br/></strong>Давайте сначала поймем во что именно компилируется приведенный выше код. Родительский класс очень простой, в bytecode он бы действительно содержал два публичных поля, ничего необычного. Но дочерний класс содержал бы одно поле - <em>type -</em> и два геттера - <em>getType</em> и <em>getStatus</em>. Сам <em>status</em> не имел бы backing поля, потому что нет причины его создавать - свойство <em>status</em> в Kotlin немутабельно, и его значение является строковым литералом. Таким образом, у нас есть два поля <em>type</em> - одно у родителя, другое у дочернего класса. Разница, однако, состоит в том, что в дочернем классе это поле было бы <em>private</em>, а в родительском <em>type</em> было бы <em>public</em> полем. <strong>Таким образом</strong>, <strong>в K1 была проблема при разрешении полей в тех случаях, когда у нас существовала иерархия расширяющихся классов в Java и Kotlin</strong>. И это действительно несколько запутывающий случай.  Например,  <em>AbstractEntitySubclass().type, </em>в мире Kotlin, может ссылаться как <strong>на поле родительского класса <em>type</em>, так и на свойство дочернего класса type</strong>. И если к свойству в родительском классе в bytecode можно обращаться через простую байткод инструкцию  <em>getfield</em>, то обращение к свойству <em>type</em> в дочернем классе требует вызова синтетически сгенерированного геттера. Следовательно, компилятор должен решить, к каким именно полям мы хотим получить доступ и как - напрямую или через геттер. <strong>Таким образом, у K1 был недостаток - он компилировал описанный выше код в простую bytecode-инструкцию <em>getfield</em>, но для дочернего класса. </strong> Это, конечно, неправильно, поскольку, как мы уже сказали, в дочернем классе поле <em>type</em> является <em>private</em>. К нему надо обращаться через синтетический геттер. Соответственно, код компилировался, но во время выполнения JVM замечала, что мы пытаемся обратиться к чему-то нелегальным способом, и выбрасывала <em>IllegalAccessError</em>.</p><p><strong>5.2. Resolution свойств объекта в K2</strong>  <br/>K2 решает эту проблему и устанавливает конкретный порядок разрешения свойств для таких случаев.  <strong>Общее правило таково – свойства дочерних классов имеют приоритет. Это значит, что свойства более конкретных классов имеют высший приоритет перед свойствами предков  при равном уровне видимости. </strong>Поэтому, при компиляции с помощью K2 упомянутого выше кода, мы получим следующее:  </p><p>CONCRETE_TYPE <br/>CONCRETE_STATUS  </p><p>Как видите, значения из дочернего класса имеют приоритет перед полями родителей. </p><h2>6. Сохранение Nullability для примитивных массивов</h2><p>Компиляторы Kotlin (как K1, так и K2) поддерживают nullability аннотации в Java коде, такие как, например, <code>@Nullable</code> и <code>@NotNull</code> от JetBrains. Следовательно, если мы имеем следующий Java метод:</p><pre><code class="java">public static @Nullable String fromCharArray(char[] s) { 
	if (s == null || s.length == 0) return null; 
 
	return new String(s); 
}</code></pre><p>Компилятор Kotlin (как K1 так и K2) успешно распознает, что возвращаемое значение метода <em>fromCharArray</em> - это nullable строка, а не просто <em>String</em>. Следовательно, в Kotlin следующий код работать не будет:</p><pre><code class="kotlin">val resultString : String = fromCharArray(null) // Correct type should be String?</code></pre><p>Проблема, однако, состояла в том, что K1 не мог вывести информацию о nullability для примитивных массивов с аннотациями для уточнения типа (аннотации со значением <code>ElementType.TYPE_USE</code> в <code>@Target</code>). Например, для этой функции на Java:</p><pre><code class="java">public static char @Nullable [] toCharArray(String s) { 
	if (s == null) return null; 
 
	return s.toCharArray(); 
}</code></pre><p>K1 не смог бы сделать вывод по информации о nullability. Поэтому это привело бы к NullPointerException во время выполнения кода на Kotlin:</p><pre><code class="kotlin">val array : CharArray = toCharArray(null) // That compiles fine in K1 
println(array[0]) // NPE </code></pre><p><strong>K2, напротив, успешно делает вывод о nullability для примитивных массивов. </strong>Таким образом, K2 не стал бы компилировать приведенный выше код, поскольку возвращаемое значение <code>toCharArary()</code> на самом деле <code>CharArray?</code>, а не <code>CharArray</code>.</p><h2>7. Вывод</h2><p>K2 привнес много улучшений в процессы компиляции кода на Kotlin. В общем, если подвести итог, он может выявить множество проблем на этапе компиляции. Улучшились взаимодействие с синтетическими сеттерами и дженериками, разрешение свойств, взаимодействие с Java в плане обработки null и многое другое.</p><figure class="full-width"><img
