На связи Дима Котиков, и это вторая статья из цикла про File Templates. <a href="https://habr.com/ru/companies/tbank/articles/872684/">В предыдущей части </a>мы описали проблему с необходимостью написания шаблонного кода и вариантами решения, познакомились с инструментом шаблонов File Templates и разобрали синтаксис написания шаблонов.</p><p>В этой статье перейдем к практике — напишем шаблоны для кода типового экрана SomeFeature мобильного приложения.</p><h2>Подготовка условия для создания шаблона</h2><p>Структура файлов и папок, которые мы хотим видеть:</p><figure class="full-width"><img alt="Структура файлов и папок. Желтыми блоками обозначены папки (пакеты), зеленым — файлы " data-src="https://habrastorage.org/getpro/habr/upload_files/e5e/3dc/434/e5e3dc434f3b5bfa1bd49763863335ed.png" height="679" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/e5e/3dc/434/e5e3dc434f3b5bfa1bd49763863335ed.png" title="Структура файлов и папок. Желтыми блоками обозначены папки (пакеты), зеленым — файлы </figcaption></div></figure><p>Определяемся с тем, что нам нужно:</p><p>1. Возможность задать имя root folder.</p><p>2. Возможность задать название фичи, которое будет являться префиксом в именах файлов — будет задаваться переменной по умолчанию <code>$NAME</code>.</p><p>3. После ввода имени рутовой папки и названия фичи должна быть сгенерирована структура файлов и папок, в файлах — сгенерирован шаблонный код со всеми корректными импортами между файлами.</p><p><br/>А еще нам нужно определить файл, который будет являться точкой входа для нашего шаблона и относительно которого мы будем формировать дочерние файлы и папки в нашей структуре. Для этого выбираем самый верхнеуровневый по вложенности файл, например <code>SomeFeatureViewModel.kt</code>.</p><h2>Возможность задать root folder</h2><p>Для указания имени root folder идем в меню создания шаблонов, жмем на вкладке <code>Files</code> на значок <code>+</code> и указываем в форме информацию:</p><figure class="full-width"><img alt="Создание входного файла для генерации шаблонов и помещение в ROOT_PACKAGE" data-src="https://habrastorage.org/getpro/habr/upload_files/d37/268/566/d372685663c70a42c61be67c3150331e.png" height="1170" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d37/268/566/d372685663c70a42c61be67c3150331e.png" title="Создание входного файла для генерации шаблонов и помещение в ROOT_PACKAGE" width="1600"/><div><figcaption>Создание входного файла для генерации шаблонов и помещение в <code>ROOT_PACKAGE</code></figcaption></div></figure><p>В меню создания шаблонов мы указали:</p><ul><li><p>название нашего набора шаблонов как <code>Compose feature</code>;</p></li><li><p>расширение файла <code>kt</code>, так как нам нужен kotlin-файл;</p></li><li><p>в поле <code>File name</code> новую переменную <code>${ROOT_PACKAGE}</code> и через слеш название файла, состоящее из <code>${NAME}</code> и суффикса <code>ViewModel</code>. Содержимое файла заполним позже.</p></li></ul><p>Нажимаем кнопку <code>Apply</code> для сохранения изменений или кнопку <code>OK</code> для сохранения изменений и выхода из меню.</p><p>Пробуем наш шаблон в деле. Для этого в папке в дереве проекта среды разработки жмем правую кнопку мыши и в контекстном меню выбираем <code>New → Compose feature</code>:</p><figure class="full-width"><img alt="Выбор шаблона из из контекстного меню</figcaption></div></figure><p>Видим диалоговое окно, в котором отображается поле <code>File name</code>, за которое отвечает переменная <code>$NAME</code>, а еще поле <code>ROOT PACKAGE</code> для нашей кастомной переменной <code>$ROOT_PACKAGE</code>:</p><figure class="full-width"><img alt="Диалоговое окно создания шаблона" создания шаблона</figcaption></div></figure><p>Заполняем поля, нажимаем на <code>OK</code> и видим, что структура из файла и папки создалась:</p><figure class="full-width"><img alt="Сгенерированный файл и папка" data-src="https://habrastorage.org/getpro/habr/upload_files/a8a/597/69c/a8a59769ccaf1ec18e3fc9390296d655.png" height="1600"     #set($OUT_PACKAGE = "")
#end</code></pre><p>Проверяем переменную <code>$OUT_PACKAGE</code>, которую создавали выше, и, если она не пустая, указываем ее вместе с ключевым kotlin-словом <code>package</code>.</p><pre><code class="kotlin">#if ($OUT_PACKAGE != "")package ${OUT_PACKAGE}#end</code></pre><p>Конфигурируем строку для последующего применения в импортах сущностей <code>State</code>, <code>SideEffect</code> и <code>UiEvent</code> (они будут созданы чуть ниже).</p><p>Далее по коду мы применяем переменную <code>$IMPORT_PREFIX</code>, а также задаем префикс <code>${NAME}</code> для названий сущностей.</p><pre><code class="kotlin">#if ($OUT_PACKAGE != "")
    #set($IMPORT_PREFIX = "import ${OUT_PACKAGE}.")
#else
    #set($IMPORT_PREFIX = "import ")
#end</code></pre><p>Шаблон готов — выглядит легче, чем нарисовать сову 😄</p><figure class=""><img alt="План по рисованию совы" data-src="https://habrastorage.org/getpro/habr/upload_files/577/200/824/577200824c79ecde4a090eee44ed151d.png" height="382" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/577/200/824/577200824c79ecde4a090eee44ed151d.png" title="План по рисованию совы" width="471"/><div><figcaption>План по рисованию совы</figcaption></div></figure><p>Проверяем, чтобы написанный шаблон генерировал тот код, что ожидается.</p><h2>Переиспользование VTL-кода в шаблонах</h2><p>Вышеупомянутые блоки VTL-кода нам нужно будет переиспользовать в других дочерних файлах. Варианты переиспользования VTL-кода:</p><ul><li><p>копипастить из файла в файл, но, как мы понимаем, это не путь самурая;</p></li><li><p>обобщить код в отдельных файлах шаблонов и переиспользовать — наш путь.</p></li></ul><p>Для того чтобы переиспользовать код, нам потребуется выделить его в отдельные файлы. Переходим в <code>Settings → Editor → File and Code Templates → вкладка Includes</code>, жмем на иконку <code>+</code> для создания нового файла и копируем в новый файл VTL-код для конфигурирования переменной <code>$OUT_PACKAGE</code>. А еще копируем строку с VTL-кодом, в которой задается <code>package</code>.</p><figure class="full-width"><img alt="Обобщенный код для расчета $OUT_PACKAGE и применения package" data-src="https://habrastorage.org/getpro/habr/upload_files/dca/549/c05/dca549c05fe3aedfe3346de252dfa221.png" height="1017" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/dca/549/c05/dca549c05fe3aedfe3346de252dfa221.png" title="Обобщенный код для расчета $OUT_PACKAGE и применения package" width="1600"/><div><figcaption>Обобщенный код для расчета <code>$OUT_PACKAGE</code> и применения package</figcaption></div></figure><p>Вынесем в отдельный файл и блок с VTL-кодом расчета <code>$IMPORT_PREFIX</code>:</p><figure class="full-width"><img alt="Обобщенный код для расчета $IMPORT_PREFIX" data-src="https://habrastorage.org/getpro/habr/upload_files/da0/3c5/888/da03c5888ab0c7b64ba9a5744fb6551a.png" height="1017" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/da0/3c5/888/da03c5888ab0c7b64ba9a5744fb6551a.png" title="Обобщенный код для расчета $IMPORT_PREFIX" width="1600"/><div><figcaption>Обобщенный код для расчета <code>$IMPORT_PREFIX</code></figcaption></div></figure><p>Теперь обобщенный код можно использовать в шаблонах через вызов конструкции <code>#parse</code>. Расчет строки <code>package</code> и импорты в итоговом варианте шаблона в файле ViewModel-и будут выглядеть так:</p><pre><code class="kotlin">#parse("CalculatePackage.kt")
 
#parse("CalculateImportPrefix.kt")
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
...
import kotlinx.coroutines.flow.flowOn
${IMPORT_PREFIX}model.${NAME}State
${IMPORT_PREFIX}model.${NAME}SideEffect
${IMPORT_PREFIX}model.${NAME}UiEvent

class ${NAME}ViewModel : ViewModel() {
...</code></pre><h2>Создание дочерних файлов с шаблонами</h2><p>Теперь в соответствии с нашей структурой файлов и папок нужно добавить шаблоны классов, находящихся в папке (пакете) model. Чтобы файлы создавались автоматически при создании шаблона ViewModel-и, их нужно создавать не как отдельные шаблоны, а как Child Template File-ы:</p><figure class="full-width"><img alt="Создание child template — сначала выделяем parent template, затем жмем на иконку создания" data-src="https://habrastorage.org/getpro/habr/upload_files/537/045/28b/53704528b715f7d67f8897899b7747bd.png" height="703" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/537/045/28b/53704528b715f7d67f8897899b7747bd.png" title="Создание child template — сначала выделяем parent template, затем жмем на иконку создания" width="1294"/><div><figcaption>Создание child template — сначала выделяем parent template, затем жмем на иконку создания</figcaption></div></figure><p>Задаем имя файлу как <code>${ROOT_PACKAGE}/model/${NAME}State</code>, меняем extension на kt и в окне создания шаблона прописываем:</p><pre><code class="kotlin">#parse("CalculatePackage.kt")
import androidx.compose.runtime.Immutable

@Immutable
data class ${NAME}State(
    val isLoading: Boolean,
)
</code></pre><p>Жмем OK, пробуем создать шаблон и видим, что импорт в файле со State-классом некорректный:</p><figure class="full-width"><img alt="Некорректный импорт в итоговом файле дочернего шаблона" data-src="https://habrastorage.org/getpro/habr/upload_files/140/211/ced/140211ceda464c83c58daa552ec7cf01.png" height="433" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/140/211/ced/140211ceda464c83c58daa552ec7cf01.png" title="Некорректный импорт в итоговом файле дочернего шаблона" width="1600"/><div><figcaption>Некорректный импорт в итоговом файле дочернего шаблона</figcaption></div></figure><p>Мы рассчитали <code>package</code> одинаково для всех сущностей — и для корневого файла, и для дочернего, который лежит во вложенном пакете <code>model</code>. К сожалению, конструкция <code>#parse</code> не может принимать в себя какие-либо параметры. Есть несколько решений ошибки неправильного package name:</p><p><strong>В вызывающем конструкцию #parse</strong> шаблоне можно задать новую переменную <code>$PACKAGE_SUFFIX</code>, а в обобщенном шаблоне <code>CalculatePackage.kt</code> использовать ее. В этом случае обобщенный шаблон <code>CalculatePackage.kt</code> будет выглядеть так:</p><pre><code class="kotlin">#if ($PACKAGE_NAME &amp;&amp; $PACKAGE_NAME != "" &amp;&amp; $ROOT_PACKAGE &amp;&amp; $ROOT_PACKAGE != "")
    #set($OUT_PACKAGE = "$PACKAGE_NAME.$ROOT_PACKAGE")
#elseif ($PACKAGE_NAME &amp;&amp; $PACKAGE_NAME != "")
    #set($OUT_PACKAGE = $PACKAGE_NAME)
#elseif ($ROOT_PACKAGE &amp;&amp; $ROOT_PACKAGE != "")
    #set($OUT_PACKAGE = $ROOT_PACKAGE)
#else
    #set($OUT_PACKAGE = "")
#end
#if ($OUT_PACKAGE != "" &amp;&amp; $PACKAGE_SUFFIX &amp;&amp; $PACKAGE_SUFFIX != "")
package ${OUT_PACKAGE}.${PACKAGE_SUFFIX}
#elseif ($OUT_PACKAGE != "")
package ${OUT_PACKAGE}
#elseif ($PACKAGE_SUFFIX &amp;&amp; $PACKAGE_SUFFIX != "")
package $PACKAGE_SUFFIX
#end</code></pre><p>Расчет package name в рутовом файле шаблона <code>Compose feature</code> будет выглядеть так:</p><pre><code class="kotlin">#set($PACKAGE_SUFFIX = "")
#parse("CalculatePackage.kt")
 
#parse("CalculateImportPrefix.kt")
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.channels.Channel
...
</code></pre><p>В файле с шаблоном для State код будет выглядеть вот так:</p><pre><code class="kotlin">#set($PACKAGE_SUFFIX = "model")
#parse("CalculatePackage.kt")
 
import androidx.compose.runtime.Immutable
 
@Immutable
data class ${NAME}State(
    val isLoading: Boolean,
)</code></pre><p><strong>Из обобщенного шаблона можно убрать конструкцию </strong><code>#if ($OUT_PACKAGE != "")package ${OUT_PACKAGE}#end</code>, вставить ее в вызывающем классе, но перед этим переприсвоить переменную <code>$OUT_PACKAGE</code>, примерно так:</p><pre><code class="kotlin">#parse("CalculatePackage.kt")
#if ($OUT_PACKAGE != "")
package ${OUT_PACKAGE}.model
#else
package model
#end
 
import androidx.compose.runtime.Immutable
 
@Immutable
data class ${NAME}State(
    val isLoading: Boolean,
)</code></pre><p>У каждого способа есть свои плюсы и минусы. В первом случае мы заводим новую и не самую очевидную для использования переменную <code>$PACKAGE_SUFFIX</code>, но избавляем себя от копипасты <code>if-else</code>-блока из второго варианта в местах использования.</p><p>Во втором варианте мы обходимся без необходимости задавать переменную <code>$PACKAGE_SUFFIX</code>, но вынуждены добавить <code>if-else</code>-блок для корректной обработки полученного результата в переменной <code>$OUT_PACKAGE</code>.</p><p>Я выберу первый способ, так как с ним придется писать чуть меньше кода в других шаблонах. Если у вас есть другие способы решения данной проблемы — делитесь в комментариях.</p><p>После применения шаблона с нашими доработками получаем корректный результат:</p><figure class="full-width"><img alt="Корректная генерация шаблонов" data-src="https://habrastorage.org/getpro/habr/upload_files/7b7/667/650/7b7667650bf61af99468b557aed7bc48.png" height="988" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/7b7/667/650/7b7667650bf61af99468b557aed7bc48.png" title="Корректная генерация шаблонов" width="1600"/><div><figcaption>Корректная генерация шаблонов</figcaption></div></figure><p>Далее нужно создать шаблоны для классов <code>SideEffect</code>, <code>UiEvent</code>, <code>Screen</code>, <code>Router</code> и <code>RouterImpl</code>. Принцип создания у них ровно такой же, как и у шаблонов для <code>ViewModel</code> и <code>State</code>. Подробный код в статье приводить не буду, но итоговые шаблоны можно <a href="https://github.com/Dmitriy1892/FileTemplatesSample/tree/main"><u>посмотреть на GitHub.</u></a></p><p>В итоге получаем  алгоритм генерации шаблона:</p><ol><li><p>Жмем на создание шаблона, вводим данные в два поля:</p><figure class="full-width"><img alt="Заполнение формы для генерации шаблонов" data-src="https://habrastorage.org/getpro/habr/upload_files/b6a/1a2/ecb/b6a1a2ecb0d2ca107cb711a9847595c9.png" height="354" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/b6a/1a2/ecb/b6a1a2ecb0d2ca107cb711a9847595c9.png" title="Заполнение формы для генерации шаблонов" width="970"/><div><figcaption>Заполнение формы для генерации шаблонов</figcaption></div></figure></li><li><p>Любуемся результатом генерации семи файлов:</p><figure class="full-width"><img alt="Сгенерированные файлы на основе шаблонов" data-src="https://habrastorage.org/getpro/habr/upload_files/a46/161/cbb/a46161cbb829b38a0b644e728d665b55.png" height="926" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/a46/161/cbb/a46161cbb829b38a0b644e728d665b55.png" title="Сгенерированные файлы на основе шаблонов" width="1600"/><div><figcaption>Сгенерированные файлы на основе шаблонов</figcaption></div></figure></li></ol><p>Так всего в пару кликов и заполненных полей мы можем генерировать пачку файлов с шаблонным кодом.</p><h2>Выводы по второй части</h2><p>В этой части мы задали входящие переменные на примере root folder, разобрали, как генерировать структуру из файлов и папок и как выносить обобщенный VTL-код для переиспользования в шаблонах. А еще разобрались в том, как генерировать дочерние файлы в подпапках и как решать проблему с некорректным package name.</p><p>Выводы после использования File Templates:</p><ul><li><p>Инструмент позволяет гибко настраивать генерацию как одного файла, так и нескольких.</p></li><li><p>Есть возможность настроить генерацию файлов в папках и подпапках.</p></li><li><p>Есть возможность задать
