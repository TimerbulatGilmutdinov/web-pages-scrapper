id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><h2>О чем эта статья?</h2><p>В данной статье я хочу показать, почему развитая система типов в языке программирования это здорово. Я подробно расскажу как устроен класс <a href="https://apidocs.arrow-kt.io/arrow-core/arrow.core/-either/index.html" rel="noopener noreferrer nofollow">Either</a> из библиотеки <a href="https://arrow-kt.io/" rel="noopener noreferrer nofollow">Arrow</a>, разберу особенности системы типов Kotlin - sealed-иерархии, ковариантность и Nothing, без которых решение задачи становится практически невозможным.</p><p>Я попробую провести небольшой ликбез о таких на первый взгляд сложных вещах на понятном практическом примере.</p><h2>Что такое Either?</h2><p>Either - это класс, который содержит в себе значение одного из двух типов. По сути, это тип <strong>либо</strong> A, <strong>либо</strong> B (either A or B). Далее по тексту я буду делать акцент на фразе <strong>либо</strong>-<strong>либо</strong>, чтобы подчеркнуть, что речь идет о типах класса Either.</p><p><strong>Важно: </strong>не путайте Either с типом пары (Pair), потому что пара содержит в себе два значения одновременно, а Either содержит одно значение одного из двух типов.</p><p>Где он может пригодиться? Давайте разбираться на примере.<br/>Допустим, мы выполняем поход в сервис по API и хотим вернуть <strong>либо</strong> полезные данные, <strong>либо</strong> сообщить об ошибке, если во время вызова что-то пошло не так.</p><pre><code class="kotlin">data class Data(val payload: String)

enum class Error {
    UNAUTHORIZED,
    FORBIDDEN,
    BAD_REQUEST,
    INTERNAL_ERROR;
}

interface MyService {
    fun getData(): Either&lt;Error, Data&gt; 
}</code></pre><p>Мы определили класс Data, содержащий полезные данные, которые возвращаются в случае, если все прошло без ошибок.<br/>Класс Error, который содержит перечисление возможных ошибок.<br/>Интерфейс MyService с функцией getData, которая возвращает <strong>либо</strong> данные, <strong>либо</strong> ошибку.</p><p>По соглашению, тип ошибки обычно располагается слева и называется Left, а тип полезных данных называется Right и располагается справа (спасибо, кэп).</p><p>Казалось бы, в чем проблема? Можно реализовать тип просто в виде пары с проверкой ограничений в конструкторе.</p><pre><code class="kotlin">class Either&lt;A : Any, B : Any&gt;(val a: A?, val b: B?) {
  init {
    require(a == null &amp;&amp; b == null || a != null &amp;&amp; b != null) {
      "Either should have only one value!"
    }
  }
}</code></pre><p>Проблема такого решения в том, что оно не обеспечивает <strong>типобезопасность</strong>. Мы хотим гарантировать, что предоставлено <strong>ровно одно</strong> значение, а другого не существует. В примере выше, это свойство проверяется во время исполнения и в случае его нарушения выкидывается исключение.<br/>Еще одна проблема реализации выше - использование null для того, чтобы представить отсутствующее значение. Это ограничивает область применения нашего класса и запрещает использовать опциональные типы.<br/>Если же написать типобезопасную реализацию этого класса, мы сможем гарантировать нужные нам свойства программы еще на этапе компиляции, а не во время исполнения, что снижает количество ошибок во время разработки.</p><p>Давайте рассмотрим приемы, необходимые для построения правильного класса Either. </p><h2>Sealed классы и интерфейсы</h2><p>Начнем с важной фичи системы типов Kotlin - это sealed-иерархии классов (они же алгебраические типы данных). Объявление класса или интерфейса с модификатором sealed запрещает создание наследников класса (или реализаций интерфейса) в других модулях и делает иерархию полностью известной на этапе компиляции.</p><p>Это, например, позволяет компилятору проверять ветки выражения when.</p><pre><code class="kotlin">sealed interface Animal {
  fun say()
}

class Cat : Animal {
  override fun say() = println("meow")
    
  fun purr() = println("cat can purr")
}

class Dog : Animal {
  
  override fun say() = println("woof")

  fun howl() = println("dog can howl")
}

fun test(animal: Animal) {
  when (animal) {
    is Cat -&gt; animal.purr()
    is Dog -&gt; animal.howl()
  }
}

fun main() {
  test(Cat())
  test(Dog())
}</code></pre><p>В данном примере за счет запечатанной (sealed) иерархии классов компилятор понимает, что выражение when перебрало все возможные варианты и нет необходимости в ветке else. При этом если появится какой-то новый наследник, то возникнет ошибка на этапе компиляции, а не во время выполнения.</p><p>Вернемся к нашему классу Either. У него всего два состояния - либо значение класса A, либо значение класса B. Попробуем представить его в виде sealed класса.</p><pre><code class="kotlin">sealed class Either&lt;A, B&gt; {
  class Left&lt;A&gt;(val a: A)//: Either&lt;A, ???&gt;
  class Right&lt;B&gt;(val b: B)//: Either&lt;???, B&gt;
}</code></pre><p>Теперь у нашего класса существует два наследника (состояния) и мы на полпути к решению задачи. Однако сейчас они не являются наследниками Either, потому что не реализуют тип для значения B. Возникает проблема - как предоставить ровно одно значение, да еще и любого типа, который указал пользователь? Чтобы решить эту проблему, на помощь приходит ковариантность типов. </p><h2>Ковариантность типов (in / out)</h2><p>Ковариантность типов (как и инвариантость, и контрвариантность) используется в классах с generic-параметрами и показывает как именно они наследуются между собой.</p><p>В примере ниже, мы объявляем класс Desk с ковариантным параметром User. Это разрешает присваивания класса Desk с любым наследником класса User там, где ожидается Desk&lt;User&gt;.</p><pre><code class="kotlin">open class User
class Manager : User()
class Engineer : User()

class Desk&lt;out U : User&gt;(val user: U)

fun main() {
  val desk: Desk&lt;User&gt; = Desk(Manager())
}</code></pre><h2>Any? Nothing.</h2><p>Еще одна особенность системы типов в Kotlin - наличие типов Any и Nothing.</p><p>С Any более менее все знакомы - это тип, который является родителем любого другого типа. А вот тип Nothing, его полная противоположность, встречается в современных языках программирования реже. Nothing - это тип, который является наследником любого типа.</p><p>Само по себе это не имеет смысла - понятно, что невозможно занаследовать все классы сразу, поэтому экземпляр Nothing не может быть создан. Сам класс имеет приватный конструктор, а тип означает значение, которое не существует. Например, функция, которая всегда кидает исключение, будет иметь тип возврата Nothing.</p><pre><code class="kotlin">public class Nothing private constructor()</code></pre><p>Тогда отношение классов из предыдущего примера выглядит следующим образом.</p><figure class="bordered full-width"><img data-blurred="true" data-src="https://habrastorage.org/getpro/habr/upload_files/8d2/657/88e/8d265788e50486eeaf5c0c5ab587f837.jpg" height="397" src="https://habrastorage.org/r/w780q1/getpro/habr/upload_files/8d2/657/88e/8d265788e50486eeaf5c0c5ab587f837.jpg" width="573"/></figure><p>И точно так же выглядит иерархия для классов с generic-параметром Desk за счет ковариантности.</p><h2>Собираем воедино</h2><p>Итак, теперь мы знаем, как представить значение, которое не существует и как разрешить подставлять тип-наследник вместо исходного типа за счет ковариантности.<br/>Теперь, вооружившись всем необходимым, напишем свою итоговую реализацию Either.</p><pre><code class="kotlin">sealed class Either&lt;out A, out B&gt; {
  class Left&lt;A&gt;(val value: A): Either&lt;A, Nothing&gt;()
  class Right&lt;B&gt;(val value: B): Either&lt;Nothing, B&gt;()
}

fun main() {
    val a: Either&lt;Int?, String&gt; = Either.Left(null)
    val b: Either&lt;Int, String&gt; = Either.Right("string")
}</code></pre><p>Сначала про использование. В примере видно, что в качестве Either&lt;Int?, String&gt; можно использовать <strong>либо</strong> Either.Left&lt;Int?&gt;, <strong>либо</strong> Either.Right&lt;String&gt;. Мы можем предоставлять одно из значений, не предоставляя другое.</p><p>А теперь, рассмотрим три ключевых момента, почему все устроено именно так:</p><ol><li><p><code>sealed class Either</code> - у нашего класса Either всего два возможных состояния и других быть не может. Для реализации этого свойства и его гарантии на этапе компиляции отлично подходит sealed class.</p></li><li><p><code>&lt;out A, out B&gt;</code> - эта конструкция позволяет подставлять вместо самих A и B любой из их наследников (в том числе Nothing) </p></li><li><p><code>Either&lt;A, Nothing&gt;()</code> - именно за счет Nothing гарантируется, что если у нас есть значение A, то значения B быть не может и наоборот. Значение одного из типов - Nothing.</p></li></ol><p>Именно сочетание этих трех приемов вместе и дает возможность реализовать типобезопасный класс Either.</p><h2>Заключение</h2><p>Мы рассмотрели несколько важных аспектов системы типов в Kotlin и смогли написать свою типобезопасную реализацию класса Either.
