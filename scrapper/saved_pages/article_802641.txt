lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/3ac/171/fbe/3ac171fbeda2e01b9f2c4eadadc429f5.png" height="1080" количество мобильных приложений: для покупателей, для продавцов, банк и т. д. В каждом из них требуется авторизация. Для этого существует наша команда Ozon ID с SDK собственного производства. Частью команды Ozon ID являюсь я — Android-разработчик с непомерной любовью к синтаксическому сахару Kotlin.</p><h2>Введение</h2><p>Поговорим сегодня про context receivers — фиче Kotlin, про которую я узнал давно, но смог найти применение лишь пару месяцев назад. Расскажу о том, что такое context receivers, где их можно использовать, и, конечно же, про «успешный успех» — минус 60% самописного DI в Ozon ID SDK. Но обо всём по порядку. </p><h2>Функции расширения</h2><p>В коде Ozon ID SDK есть следующее расширение для <code>ComponentActvity</code>.</p><pre><code class="kotlin">inline fun &lt;T&gt; ComponentActvity.collectWhenStarted(
    data: Flow&lt;T&gt;,
    crossinline collector: (T) -&gt; Unit
) {
    lifecycleScope.launchWhenStarted { 
        data.collect {
            collector.invoke(it)
        } 
    }
}</code></pre><details class="spoiler"><summary>Ремарка</summary><div class="spoiler__content"><p>Я знаю, что существует <a href="https://medium.rip/androiddevelopers/a-safer-way-to-collect-flows-from-android-uis-23080b1f8bda">более надёжный способ collect’ить Flow на ui-потоке</a>. Например, с помощью <code>flowWithLifecycle</code>. Но в нём больше параметров, что будет только отвлекать от основной темы.</p></div></details><p>Реализовано расширение <code>collectWhenStarted</code> исключительно для удобства сбора данных с множества <code>Flow</code> из <code>ViewModel</code> внутри <code>Activity</code>. Ниже пример использования этого расширения.</p><pre><code class="kotlin">class AuthFlowActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        ...
        collectWhenStarted(viewModel.backStack) { onBackStackChanged(it) }
        collectWhenStarted(viewModel.navigationEvents) { onNavigationEvent(it) }
        ...
    }
}</code></pre><p>Удобно? В общем-то да. Вызов бесспорно короче, чем без использования расширения. Идеально? Отнюдь. Лично мне хотелось бы видеть вызов <code>collectWhenStarted</code> примерно следующим образом:</p><pre><code class="kotlin">class AuthFlowActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        ...
        viewModel.backStack.collectWhenStarted { onBackStackChanged(it) }
        viewModel.navigationEvents.collectWhenStarted { onNavigationEvent(it) }
        ...
    }
}</code></pre><p>Вызвать <code>collect*</code> у <code>Flow</code> более интуитивно, чем <code>collect*</code> у <code>Activity</code>, не правда ли?</p><p>Для того чтобы реализовать улучшенный вариант расширения <code>collectWhenStarted</code>, технически нам понадобится, чтобы механизм методов-расширений мог принимать 2 receiver-параметра. К сожалению, JetBrains пока не реализовали такую возможность в языке Kotlin... Или реализовали? </p><h3>Реализация через scope</h3><p>На самом деле есть такой подход, как scope. При таком подходе метод-расширение реализуется в новом классе. В случае с <code>Activity</code> решение на scope можно применить следующим образом.</p><pre><code class="kotlin">interface LifecycleOwnerScope : LifecycleOwner {
    fun &lt;T&gt; Flow&lt;T&gt;.collectWhenStarted(collector: (T) -&gt; Unit) {
        lifecycleScope.launchWhenStarted { collect { collector.invoke(it) } }
    }
}</code></pre><p>Имплементируем в <code>Activity</code> интерфейс <code>LifecycleOwnerScope</code>, и готово — у <code>Flow</code> появляется расширение <code>collectWhenStarted</code>. При этом реализация интерфейса <code>LifecycleOwnerScope</code> внутри <code>Activity</code> не требуется за счёт «реализации по умолчанию».</p><pre><code class="kotlin">class AuthFlowActivity : AppCompatActivity(), LifecycleOwnerScope {
                                            // ^ Добавили scope ^
    override fun onCreate(savedInstanceState: Bundle?) {
        ...             // collectWhenStarted берётся из LifecycleOwnerScope
        viewModel.backStack.collectWhenStarted { onBackStackChanged(it) }
        viewModel.navigationEvents.collectWhenStarted { onNavigationEvent(it) }
        ...
    }
}</code></pre><p>Подход вполне жизнеспособный. Но, как по мне, не без недостатков. Например, чтобы расширение <code>collectWhenStarted</code> заработало нужно собственноручно отнаследоваться от <code>LifecycleOwnerScope</code>. Это не так удобно, как глобальные расширения, которые IDE услужливо сама подсказывает при вводе имени. </p><p>Давайте уже перейдём к context receivers.</p><h2>Context receivers спешат на помощь</h2><p>Context receivers — это концепт, фича языка Kotlin. Context receivers добавлены в язык как инструмент, позволяющий преодолеть ограничения extension-функций. Технически context receivers, как и extension-функции, компилируются в метод с дополнительным <code>this</code>-параметром. То есть context receivers являются очередным синтаксическим сахаром Kotlin, но, конечно же, круче и «слаще», чем extension.</p><p>Context receivers появились аж в Kotlin 1.6.20. Вместе с context receivers в язык добавили новое ключевое слово <code>context</code>. Фича в версии Kotlin 1.9.22 всё ещё экспериментальная, поэтому если попытаться сходу ей воспользоваться, то IDE выдаст следующее сообщение.</p><blockquote><p>The feature "context receivers" is experimental and should be enabled explicitly</p></blockquote><p>Для того чтобы context receivers заработали, нужно дополнить файл <code>build.gradle</code> модуля, в котором будет использован <code>context</code>.</p><pre><code class="kotlin">tasks.withType&lt;org.jetbrains.kotlin.gradle.tasks.KotlinCompile&gt;().configureEach {
    kotlinOptions {
        freeCompilerArgs = freeCompilerArgs + "-Xcontext-receivers"
    }
}</code></pre><p>Теперь можно использовать <code>context</code> в коде. Приступим. Будем модифицировать расширение поэтапно.   </p><p><strong>Шаг 1:</strong> Перенести receiver-параметр <code>ComponentActvity</code> в аргументы <code>context</code>.</p><pre><code class="kotlin">context (ComponentActvity)
inline fun &lt;T&gt; /*ComponentActvity.*/collectWhenStarted(
    data: Flow&lt;T&gt;,
    crossinline collector: (T) -&gt; Unit
) {
    lifecycleScope.launchWhenStarted { 
        data.collect {
            collector.invoke(it)
        } 
    }
}</code></pre><details class="spoiler"><summary>Байт-код</summary><div class="spoiler__content"><p>Байт-код скомпилированной функции с context receivers получается точно такой же, как и его прямой аналог, реализованный через extension. К сожалению, декомпилировать байт-код с context receivers у меня не вышло. Может, кто-нибудь из читателей подробнее расследует данный вопрос и приведёт свои примеры в комментариях.</p><p>Ниже приведена сигнатура декомпилированной из байт-кода extension-функции. Напомню, что байт-код идентичен аналогу на context receivers.</p><pre><code class="java">public static final void collectWhenStarted(
    @NotNull ComponentActivity $this$collectWhenStarted, 
    @NotNull final Flow data, 
    @NotNull final Function1 collector
) </code></pre><p></p></div></details><p>Код скомпилируется и выполнится, как и на прежней реализации через расширение. Но есть один нюанс, на котором стоит остановиться. Дело в том, что context receivers — это не extension (ваш капитан Очевидность). Функции с context receivers нельзя вызвать, как будто это метод класса. Пример в сниппете ниже.</p><pre><code class="kotlin">// Реализация через extension
activity.collectWhenStarted() // компилятор позволяет вызвать функцию-расширение `collectWhenStarted`, как будто это метод класса</code></pre><pre><code class="kotlin">// Реализация через context receivers
activity.collectWhenStarted() // Ошибка: Unresolved reference: collectWhenStarted
with(activity) { // apply, run тоже подойдут
    collectWhenStarted() // Функцию с context можно вызвать только внутри "контекста" 
}</code></pre><p><strong>Шаг 2:</strong> Перенести аргумент <code>data: Flow</code> в receiver функции</p><pre><code class="kotlin">context (ComponentActvity)
inline fun &lt;T&gt; Flow&lt;T&gt;.collectWhenStarted(
    /*data: Flow&lt;T&gt;,*/
    crossinline collector: (T) -&gt; Unit
) {
    lifecycleScope.launchWhenStarted { 
        /*data.*/collect {
            collector.invoke(it)
        } 
    }
}</code></pre><p>Вуаля! Готово. Теперь у <code>Flow</code> внутри (иначе говоря, «в контексте») <code>ComponentActvity</code> появится метод <code>collectWhenStarted</code>.</p><pre><code class="kotlin">class AuthFlowActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        ...            // powered by context receivers
        viewModel.backStack.collectWhenStarted { onBackStackChanged(it) }
        viewModel.navigationEvents.collectWhenStarted { onNavigationEvent(it) }
        ...
    }
}</code></pre><h2>We need to go deeper</h2><p>Приоткрою ещё немного информации относительно ключевого слова <code>context</code>:</p><ol><li><p><code>context</code> может принимать более одного аргумента. Например, <code>context (classA, classB)</code>.</p></li><li><p><code>context</code> можно прописывать над классами.</p></li></ol><blockquote><p>Прямо как аннотация <code>@Component(dependencies = [...])</code> в Dagger, не правда ли?</p></blockquote><p>Начну разбор этих пунктов с последнего — <code>context</code> над классом. В качестве основы для примеров возьму DI из Ozon ID SDK.</p><pre><code class="kotlin">internal class RootModule(
    val application: Application
)

context(RootModule)
internal class ChildModule {

    val dependency by lazy {
        Dependency(
            /*this@RootModule.*/application,
            ...
        )
    }
}</code></pre><p>Как видно из примера выше, внутри <code>ChildModule</code> можно обратиться к <code>application</code>, как-будто это свойство <code>ChildModule</code>. При этом к свойству можно обратиться и через <code>this</code> —  <code>this@RootModule.application</code>. Пригодится на случай конфликта имён context receivers или ради внесения ясности в вопрос «откуда взялась эта зависимость?».</p><p>Продолжим погружение. Рассмотрим, как создавать зависимости с context receivers.</p><pre><code class="kotlin">context(RootModule)
internal class SubChildModule

context(RootModule)
internal class ChildModule {
    ...

    val subChildModule by lazy {
        // with не нужен, уже в нужном контексте
        SubChildModule()
    }
}

val childModule = with(rootModule) {
    // Нужен with для создания
    ChildModule()
}</code></pre><p>Для того чтобы создать объект класса с context receivers, нужно, чтобы вызов конструктора происходил в требуемом контексте. Контекст можно задать следующим образом:</p><ol><li><p>создать экземпляр внутри класса, требуемого в context receiver;</p></li><li><p>создать экземпляр внутри независимого класса, в context receivers которого есть нужный класс. Например, как создан модуль <code>SubChildModule</code> внутри <code>ChildModule</code>;</p></li><li><p>создать внутри scope-функции c лямбдой-расширением в качестве аргумента (with, apply, run). Подойдут и собственные функции с аналогичным параметром.</p></li></ol><p>Теперь рассмотрим создание объектов с несколькими context receivers.</p><pre><code class="kotlin">context(RootModule, RepositoryModule, NetworkModule, CookieModule)
internal class MultiContextModule

val multiContextModule = with(rootModule) {
    with(repositoryModule) {
        with(networkModule) {
            with(cookieModule) {
                MultiContextModule()
            }
        }
    }
}</code></pre><p>Как можно заметить, код создания экземпляра класса с несколькими context receivers может оказаться не столь изящным, как хотелось бы. Но это поправимо, потому что <code>context</code> можно прописывать к лямбда-аргументам. Возьмём за основу код <code>with</code> из стандартной библиотеки Kotlin и обогатим его <code>context</code>.</p><pre><code class="kotlin">// Пример из stdlib
@kotlin.internal.InlineOnly
public inline fun &lt;T, R&gt; with(receiver: T, block: T.() -&gt; R): R {
    contract {
        callsInPlace(block, InvocationKind.EXACTLY_ONCE)
    }
    return receiver.block()
}</code></pre><pre><code class="kotlin">// with context receivers
@OptIn(ExperimentalContracts::class)
@Suppress("SUBTYPING_BETWEEN_CONTEXT_RECEIVERS")
internal inline fun &lt;T1, T2, R&gt; with(c1: T1, c2: T2, block: context(T1, T2) () -&gt; R): R {
    contract {                                           // ^^^^^^^
        callsInPlace(block, InvocationKind.EXACTLY_ONCE) // см. сюда
    }
    return block(c1, c2)
}</code></pre><p>В примере выше к типу лямбды <code>block</code> добавлено ключевое слово <code>context</code>. Параметры <code>context</code> — generic-типы. Подобных <code>with</code> можно написать больше под нужное количество context receivers. В свою очередь, это позволяет нам значительно уменьшить сдвиг кода вправо при создании <code>MultiContextModule</code>.</p><pre><code class="kotlin">context(RootModule, RepositoryModule, NetworkModule, CookieModule)
internal class MultiContextModule

val multiContextModule = with(
    rootModule,
    repositoryModule,
    networkModule,
    cookieModule
) {
    MultiContextModule()
}</code></pre><h2>Итоги</h2><p>Подведём черту под тем, что мы сегодня узнали. В этом нам поможет файл <a href="https://github.com/Kotlin/KEEP/blob/master/proposals/context-receivers.md">KEEP</a> по context receivers.   </p><ul><li><p>Context receivers — это механизм, призванный расширить возможности extension-функций. Причём, как мы видели на примере с <code>Flow</code>, именно расширить, а не заменить.</p></li><li><p>Context receivers можно прописывать как над функциями и свойствами, так и над классами.</p></li><li><p>Context receivers позволяют использовать более одного receiver-аргумента.</p></li><li><p>Context receivers непонятно где и когда применять. Как минимум у меня пока не сложилось чёткого мнения, чтобы я мог однозначно сказать «вот тут extension, тут передать аргументом, а тут обязательно context». В видео в конце статьи можете найти размышления на эту тему. Я же пока в данном вопросе буду придерживаться исключительно технического момента.</p></li></ul><h2>Заключение</h2><p>Про context receivers я узнал почти 2 года назад из <a href="https://www.youtube.com/watch?v=GISPalIVdQY">видео от Jetbrains</a>, не смог придумать им никакого полезного применения и отложил на дальнюю полку знаний про Kotlin. И тут понеслось: рефакторинг DI в Ozon ID SDK, доклад внутри мобильной команды и в результате эта статья. Надеюсь, что у меня получилось донести до читателей мощь context receivers и подтолкнуть их на дальнейший поиск применимости этой фичи.</p><p></p></div></div></div><!-- --><!-- --></div><!-- --><!--
