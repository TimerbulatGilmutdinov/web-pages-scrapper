target="_blank">Автор оригинала: <span>Philip Riecks</span></a></div><div class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Новый перевод от команды<a href="https://t.me/+QmrKn0wA8CdkNjAy"> Spring АйО</a> расскажет вам, что такое <code>SpringExtension</code>, как правильно пользоваться этим расширением и когда его надо (или не надо) регистрировать вручную. </p><hr/><p>В последнее время я видел много путаницы по поводу <code>SpringExtension</code>. Когда не удается правильно получить конфигурацию контекста для теста, некоторые разработчики рандомно выбрасывают <code>@ExtendWith(SpringExtension.class)</code> в свои тестовые классы, чтобы тесты (как они надеются) заработали. В этом блог-посте я хочу пролить немного света на <code>SpringExtension</code> и его использование при тестировании приложений на Spring Boot. К концу этой статьи вы поймете, когда, почему и как использовать это расширение.</p><p><code>SpringExtension</code> обеспечивает гладкую интеграцию тестов, написанных на JUnit Jupiter, с фреймворком TestContext от Spring. Чаще всего вам не надо в явном виде регистрировать <code>SpringExtesion</code>, поскольку все test slice аннотации Spring Boot делают это сами.</p><h3>Для чего используется расширение JUnit Jupiter?</h3><p>Первый вопрос, на который нам надо ответить, если мы хотим понять <code>SpringExtension</code> — это вопрос о том, что такое расширение JUnit Jupiter и зачем оно нужно. </p><p>Модель расширения JUnit Jupiter — это единая концепция (по контрасту с такими API от JUnit 4 как <code>Runner</code>и <code>Rule</code>), предназначенная для расширения функциональности тестов и перехватывания циклов жизни наших тестов программным путем. Существует довольно много различных точек внутри жизненного цикла, куда может встраиваться расширение (например, <code>BeforeAllCallback</code>); кроме того, доступны и другие функции в форме утилит (например, <code>ParameterResolver</code>). Чтобы увидеть полный список всех доступных API расширения, рекомендую посмотреть на интерфейс <code>Extension</code> и на все интерфейсы, которые его расширяют.</p><p>Использовать их можно довольно гибко. Мы можем выполнить рутинные задачи до или после теста, разрешить параметры и решить, выполнять тест или пропустить его.</p><p>Чаще всего мы реализуем перекрестные аспекты, которые имеют отношения к нескольким тестам, использующим расширение. Например, при написании тестов для веб, вместо оборачивания взаимодействия браузера в блок <code>try</code>-<code>catch</code> для каждого теста, чтобы сделать скриншоты неудачного завершения, мы можем написать для этого расширение JUnit Jupiter предлагает интерфейс <code>TestExecutionExceptionHandler</code> для обработки исключений в наших тестовых методах в одном централизованном месте. (PS:<a href="https://rieckpil.de/write-concise-web-tests-with-selenide-for-java-projects/"> Selenide</a> сразу поставляется с расширением, делающим скриншоты при неудачном завершении).</p><p>Написание кастомизированного расширения — наука несложная. Мы разрабатываем кастомизированное расширение, которое инжектирует случайные <code>UUID</code> в наши тестовые методы как часть мастер-класса <a href="https://rieckpil.de/testing-spring-boot-applications-masterclass/">Testing Spring Boot Applications Masterclass</a>. </p><p>Каждый раз, когда мы хотим активировать расширение JUnit Jupiter для нашего теста, мы должны в явном виде зарегистрировать его при помощи аннотации <code>@ExtendWith</code> над тестовым классом:</p><pre><code class="java">@ExtendWith(MyExtension.class)
class MyTest {
  @Test
  void test() {
  }
}</code></pre><p>Многие фреймворки и тестовые библиотеки поставляются с кастомизированным расширением JUnit Jupiter для удобной интеграции с окружением JUnit.</p><p>Примеры:</p><ul><li><p><code>MockitoExtension</code> от Mockito для простой настройки наших mock-ов</p></li><li><p>Тестовые контейнеры <code>TestcontainersExtension</code> (активируются при помощи <code>@Testcontainers</code>) для удобного запуска и остановки Docker контейнеров </p></li><li><p><code>SpringExtension</code> от Spring</p></li></ul><p>Чтобы узнать больше о модели расширения и ее различных API, см. <a href="https://junit.org/junit5/docs/current/user-guide/#extensions">JUnit 5 User Guide</a>.</p><h3>В чем цель SpringExtension?</h3><p>На следующем шаге давайте исследуем перекрестную функциональность, которую реализует <code>SpringExtension</code>.</p><p>Лучший способ начать наше расследование — посмотреть на исходный код <code>SpringExtension</code>, чтобы понять, какие именно API расширения он реализует:</p><pre><code class="java">public class SpringExtension implements BeforeAllCallback, 
    AfterAllCallback, 
    TestInstancePostProcessor, 
    BeforeEachCallback, 
    AfterEachCallback, 
    BeforeTestExecutionCallback, 
    AfterTestExecutionCallback, 
    ParameterResolver {
  
  // ...
 
}</code></pre><p>Их довольно много. Как можно понять из приведенного выше исходного кода, <code>SpringExtension</code> плотно взаимодействует с жизненным циклом наших тестов. </p><p>Если не погружаться слишком глубоко в детали реализации, главные зоны ответственности этого расширения следующие:</p><ul><li><p>Управлять жизненным циклом <code>TestContext</code> Spring (например, начать новый цикл получения кешированного контекста)</p></li><li><p>Поддерживать инжекцию зависимостей для параметров (например, конструктор тестового класса или тестовый метод)</p></li><li><p>Задачи по чистке и другие “хозяйственные” рутинные задачи, выполняемые после теста </p></li></ul><p><code>SpringExtension</code> работает как клей между JUnit Jupiter и Spring Test. Чаще всего <code>SpringExtension</code> делегирует свои обязанности <code>TestContextManager</code>, чтобы он выполнил самую тяжелую работу:</p><pre><code class="java">public class SpringExtension {
 
  @Override
  public void beforeAll(ExtensionContext context) throws Exception {
    getTestContextManager(context).beforeTestClass();
  }
}</code></pre><p><code>TestContextManager</code> отвечает за один <code>TestContext</code> и не зависит от тестового фреймворка. Более того, <code>TestContextManager</code> также вызывает все зарегистрированные <code>TestContextListeners</code> на основании события жизненного цикла (например, до тестового класса или после тестового метода).</p><p>Теперь посмотрим на еще один практический пример работы <code>SpringExtension</code>: разрешение (инжектирование) параметров:</p><pre><code class="java">@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class ApplicationIT {
 
  @Test
  void needsEnvironmentBeanToVerifySomething(
     @Autowired Environment environment) { // resolved by the SpringExtension
 
    assertNotNull(environment);
  }
}</code></pre><p>Приведенный выше метод инжектирует <code>Environment</code> через параметр метода. В фоновом режиме следующий фрагмент кода из <code>SpringExtension</code> берет на себя ответственность за разрешение этого параметра: </p><pre><code class="java">public class SpringExtension {
 
  // ...
 
  @Override
  public boolean supportsParameter(ParameterContext parameterContext, 
                                   ExtensionContext extensionContext) {
    // determine whether or not this extension is responsible to resolve
    // the parameter
  }
 
  @Nullable
  public Object resolveParameter(ParameterContext parameterContext, 
                                 ExtensionContext extensionContext) {
    // return the bean from the TestContext
  }
}</code></pre><p>Прежде чем <code>SpringExtension</code> попытается разрешить данный параметр, <code>supportsParameter</code> (часть интерфейса <code>ParameterResolver</code>) определяет, является ли это обязанностью данного расширения. В нашем примере аннотация <code>@Autowired</code> рядом с параметром является явным индикатором того, что <code>SpringExtension</code> должен разрешить этот параметр с помощью <code>TestContext</code>.</p><p>Однако, инжектирование полей работает через <code>DependencyInjectionTestExecutionListener</code>. Это один из многих заданных по умолчанию <code>TestExecutionListeners</code>, которые <code>TestContextManager</code> вызывает перед запуском любого теста.</p><pre><code class="java">@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class ApplicationIT {
 
  @Autowired
  // injected by the DependencyInjectionTestExecutionListener
  private CustomerService customerService; 
 
  @Test
  void needsEnvironmentBeanToVerifySomething(
    @Autowired Environment environment // resolved by the SpringExtension
  ) {
    assertNotNull(environment);
  }
}</code></pre><h3>Когда нам необходимо регистрировать SpringExtension?</h3><p>Чаще всего нам не нужно регистрировать это расширение явным образом, потому что оно уже активировано для нас. Так обстоят дела всякий раз, когда мы используем тестовые аннотации из Spring Boot.</p><p>Все<a href="https://rieckpil.de/spring-boot-test-slices-overview-and-usage/"> Spring Boot test slice аннотации</a>, а также<a href="https://rieckpil.de/guide-to-springboottest-for-spring-boot-integration-tests/"> @SpringBootTest</a>, регистрируют <code>SpringExtension</code> из коробки. И это хорошо, потому что в противном случае тесты не смогли бы стартовать, потому что им для работы был бы нужен <code>TestContext</code>. Поэтому данный подход избавляет нас от необходимости нажимать на клавиши несколько лишних раз, и кроме того мы не сталкиваемся со странными падениями тестов из-за того, что мы забыли зарегистрировать <code>SpringExtension</code>.</p><p>Посмотрев на исходный код аннотации <code>@WebMvcTest</code>, мы увидим вот это:</p><pre><code class="java">@Target({ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Inherited
@BootstrapWith(WebMvcTestContextBootstrapper.class)
@ExtendWith({SpringExtension.class})
// ... further annotations
public @interface WebMvcTest {
 
}</code></pre><p>Среди других мета-аннотаций и инструкций по поводу того, что необходимо автоматически сконфигурировать для тестов такого типа, мы видим, что этот код активирует для нас <code>SpringExtension</code>.</p><p>То же самое верно для <code>@SpringBootTest</code>:</p><pre><code class="java">@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Inherited
@BootstrapWith(SpringBootTestContextBootstrapper.class)
@ExtendWith(SpringExtension.class)
// ...
public @interface SpringBootTest {
  // ...
}</code></pre><p>Поэтому каждый раз, когда мы встречаем тестовый класс, где <code>SpringExtension</code> зарегистрирован вручную, и используется одна из test slice аннотаций Spring Boot, мы можем безопасно удалить его: </p><pre><code class="java">@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
@ExtendWith(SpringExtension.class) // not necassary and should be removed
class ApplicationIT {
 
  // ...
 
}</code></pre><p>В то время как JUnit Jupiter не будет жаловаться (то есть заваливать тест или поднимать много шума в логах), если одно и то же расширение сконфигурировано дважды, мы должны удалить его, чтобы избежать такого дублирования. Это лишний код, который может только привести к путанице для новых членов команды, если мы не добавляем <code>SpringExtension</code> стабильно везде.</p><h3>Когда нам необходимо регистрировать SpringExtension (часть вторая)?</h3><p>Существует ограниченный набор use case-ов, где мы обязаны явно зарегистрировать <code>SpringExtension</code> вручную. Один такой use case — это написание кастомизированной test slice аннотации.</p><p>Как часть мастер-класса<a href="https://rieckpil.de/testing-spring-boot-applications-masterclass/"> Testing Spring Boot Applications Masterclass</a>, мы загружаем компонент мессенджера в нашем приложении для тестовых целей. Этот практический пример демонстрирует, как загружать в наше приложение только релевантные части Amazon SQS Listener. Поскольку сейчас доступна test slice аннотация от Spring Boot, мы вручную регистрируем <code>SpringExtension</code>, чтобы работать с <code>TestContext</code> от Spring на протяжении всего теста:</p><pre><code class="java">@ExtendWith(SpringExtension.class)
@Import(BookSynchronizationListener.class)
@ImportAutoConfiguration(MessagingAutoConfiguration.class)
@Testcontainers(disabledWithoutDocker = true)
class BookSynchronizationListenerSliceTest {
  
   // test a SQS listener in isolation
  
}</code></pre><h3>Заключение</h3><p><code>SpringExtension</code> реализует несколько методов для вызова модели расширения JUnit Jupiter для гладкой интеграции между JUnit и Spring. Когда мы тестируем приложения на Spring Boot, нам как правило не нужно явно регистрировать расширение, поскольку все sliced context аннотации (например, <code>@WebMvcTest</code>) делают это за нас.</p><p>Официальная <a href="https://docs.spring.io/spring-framework/docs/current/reference/html/testing.html#testcontext-junit-jupiter-extension">документация</a> тоже является прекрасным источником информации, если вы хотите глубже погрузиться в тему. </p><p>Исходный код приведенных выше примеров <a href="https://github.com/rieckpil/blog-tutorials/tree/master/spring-boot-testing-tips-and-tricks">доступен на GitHub</a>.</p><p>Наслаждайтесь работой со SpringExtension.</p><hr/><figure class="full-width"><img data-src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/c16/4b2/4fe/c164b24fe67857325369e7d0e25ab10e.png" height="520" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/c16/4b2/4fe/c164b24fe67857325369e7d0e25ab10e.png" width="1600"/></figure><p><a href="https://www.springnow.ru">Регистрируйтесь</a> на главную конференцию про Spring на русском языке от сообщества Spring АйО!
