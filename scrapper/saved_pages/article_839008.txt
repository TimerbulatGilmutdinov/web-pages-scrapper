xmlns="http://www.w3.org/1999/xhtml"><p>Вдохновившись обновлением Telegram без маркета приложений я захотел сделать на одном из своих пет-проектов что-то подобное. Первой мыслью было - найти этот код в исходниках Telegram, но т.к. скорее всего у них обновление скачивается с серверов, я решил не играть в лотерею и не тратить время на раскопки в Java-коде, потому что я хотел сделать так, чтобы можно было скачивать с GitHub-releases.</p><p>Итак, начнем</p><h2>Зависимости</h2><p>Для работы понадобится Retforit, Hilt. Как подключать и использовать Hilt я рассказывать не буду, об этом множество статей, а вот для Retrofit нужно немного:</p><pre><code class="kotlin">implementation ("com.squareup.retrofit2:retrofit:2.11.0")
implementation ("com.squareup.retrofit2:converter-gson:2.11.0")</code></pre><p>Для того, чтобы сделать автообновление приложения, нужно знать, что обновление есть, и знать, что качать.</p><p>На просторах интернета немало информации про GitHub-Api, поэтому я расскажу вкратце. Поинт, по которому можно получить информацию о последнем релизе, выглядит следующим образом: https://api.github.com/repos/{user}/{repository}/releases/latest, где user и repository говорят сами за себя. Этот поинт предоставляет огромное количество информации (К сожалению, качество оставляет желать лучшего):</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/30c/67a/487/30c67a487d08f8fff7d640ab6624da13.png" height="885" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/30c/67a/487/30c67a487d08f8fff7d640ab6624da13.png" width="843"/></figure><p>Все, что нам понадобится из этого, - <code>tag_name</code> и название apk-файла - <code>name</code> - в assets. Получить это в Android приложении довольно просто можно через Retrofit. </p><h2>Retrofit сервис для получения данных о релизе</h2><p>Для начала добавим в <code>AndroidManifest</code> разрешение на использование интернета:</p><pre><code class="xml">&lt;uses-permission android:name="android.permission.INTERNET" /&gt;</code></pre><p>Сервис выглядит как и все стандартные сервисы Retrofit, нам нужен лишь один метод GET:</p><pre><code class="kotlin">interface GitHubDataService {
  
    @GET("repos/vafeen/UniversitySchedule/releases/latest")
    suspend fun getLatestRelease(): Response&lt;Release&gt;
  
}</code></pre><p><code>GsonConverterFactory</code> здесь нужен для автоматического конвертирования ответов в нужные данные. Мы будем конвертировать в класс <code>Release</code>, который исходя из данных на поинте будет выглядеть вот так:</p><pre><code class="kotlin">data class Release(
    val url: String,
    @SerializedName("assets_url") val assetsUrl: String,
    @SerializedName("upload_url") val uploadUrl: String,
    @SerializedName("html_url") val htmlUrl: String,

    @SerializedName("tag_name") val tagName: String,
    @SerializedName("target_commitish") val targetCommitish: String,

    val draft: Boolean,
    @SerializedName("prerelease") val preRelease: Boolean,
    @SerializedName("created_at") val createdAt: String,
    @SerializedName("published_at") val publishedAt: String,
    val assets: List&lt;Asset&gt;,
    @SerializedName("tarball_url") val tarballUrl: String,
    @SerializedName("zipball_url") val zipballUrl: String,
    val body: String
)</code></pre><p>Внутри него также используются классы <code>Author</code> и <code>Asset</code>:</p><pre><code class="kotlin">data class Author(
    val login: String,
    val id: Long,
    @SerializedName("node_id") val nodeId: String,

    val url: String,

    val uploader: Author,

    @SerializedName("created_at") val createdAt: String,

    @SerializedName("browser_download_url") val browserDownloadUrl: String

Начнем с модуля:</p><pre><code class="kotlin">@Module
@InstallIn(SingletonComponent::class)
class RetrofitDIModule {

    @Provides
    @Singleton
    fun provideGHDService(): GitHubDataService = Retrofit.Builder()
        .baseUrl("https://api.github.com/")
        .addConverterFactory(GsonConverterFactory.create())
        .build().create(GitHubDataService::class.java)
}</code></pre><p>Сам сетевой репозиторий - обычный класс-абстракция, который будет выглядеть следующим образом:</p><pre><code class="kotlin">class NetworkRepository @Inject constructor(
    private val gitHubDataService: GitHubDataService
) {

    suspend fun getLatestRelease(): Response&lt;Release&gt;? = try {
        gitHubDataService.getLatestRelease()
    } catch (e: Exception) {
        null
    }
}</code></pre><p><code>getLatestRelease</code> оборачивается в блок <code>try {} catch {}</code>, поскольку при получении данных из сети может возникнуть огромное количество ошибок, такие как: долгое ожидание, отсутствие или неожиданное отключение интернета и много другое. </p><h2>Проверка версии приложения </h2><p>На данном этапе, когда мы имеем версию последнего релиза, нужно узнать, требуется ли приложению обновление. Тэги общепринято называть по шаблону "v"+`номер версии`. В примере у последнего релиза версия 1.3, т.к. <code>tag_name == v1.3</code>, значит, чтобы приложение было актуальным, нужно, чтобы его <code>versionName</code>, который указывается в Gradle, совпадал с названием версии в последнем тэге.</p><p>Программно узнать версию приложения можно следующим образом:</p><pre><code class="kotlin">fun getVersionName(context: Context): String? =
    context.packageManager.getPackageInfo(context.packageName, 0).versionName</code></pre><p>Раньше <code>versionName</code> возвращало String, но в последних версиях Android студия заставляет указать нуллабельный тип, но если версия указывается в каждом релизе, беспокоиться не о чем, она здесь вернется. </p><p>Полная проверка с получением данных будет выглядеть так:</p><pre><code class="kotlin">val versionName = getVersionName(context = context)

Корутина {
val release = networkRepository.getLatestRelease()?.body()

if (release != null &amp;&amp; versionName != null &amp;&amp;
            release?.tag_name?.substringAfter("v") != versionName) {

//                  Обновление приложения
  
            }
}</code></pre><h2>Обновление приложения </h2><p>Предполагается, что текущая версия приложения ниже версии последнего релиза, а последняя версия добавлена в GitHub releases. </p><p>Обновление приложения будет включать в себя скачивание APK-файла и запрос на его установку пользователю. </p><h2>Скачивание APK-файла </h2><p>Для скачивания файла нужен еще один Retrofit-сервис </p><pre><code class="kotlin">interface DownloadService {
  
    @GET
    @Streaming
    fun downloadFile(@Url fileUrl: String): Call&lt;ResponseBody&gt;
  
}</code></pre><p>Метод - GET </p><p><code>Streaming</code> - эта аннотация указывает Retrofit, что ответ от сервера должен быть обработан по мере поступления данных, а не загружен полностью в память. (Далее важно будет обрабатывать прогресс, поскольку если просто подписаться на этот стрим, интерфейс зависнет от количества операций)</p><p>Класс <code>Сall</code> представляет собой HTTP запрос, который можно выполнить асинхронно или синхронно. Он предоставляет методы для выполнения запроса и обработки ответа.</p><p><code>ResponseBody</code> нужен для получения тела ответа. </p><p>Также добавим реализацию этого интерфейса  в <code>RetrofitDIModule</code> и конструктор репозитория</p><pre><code class="kotlin">@Provides
@Singleton
fun provideDownloadService(): DownloadService = Retrofit.Builder()
        .baseUrl("https://github.com/")
        .build().create(DownloadService::class.java)</code></pre><h2>Скачивание и установка файла </h2><h2>Скачивание </h2><p>Код скачивания файла я нашел в интернете по ссылке <a href="https://github.com/mt-ks/kotlin-file-download" rel="noopener noreferrer nofollow">https://github.com/mt-ks/kotlin-file-download </a>но немного упростил его и добавил обработку ошибок. </p><p>Скачивать и устанавливать файл я буду через Singleton Hilt, чтобы не потерять данные в потоках.  </p><p>Объяснять, как работает этот код долго, поэтому я оставлю комментарии:</p><pre><code class="kotlin">@Singleton 
class Downloader @Inject constructor(private val networkRepository: NetworkRepository,) {
    private val _percentageFlow = MutableSharedFlow&lt;Float&gt;()
    val percentageFlow = _percentageFlow.asSharedFlow()

    private val _isUpdateInProcessFlow = MutableSharedFlow&lt;Boolean&gt;()
    val isUpdateInProcessFlow = _isUpdateInProcessFlow.asSharedFlow()

    private fun installApk(context: Context) {
        val apkFilePath = context.pathToDownloadRelease()
        // Создаем объект File для APK файла по указанному пути
        val file = File(apkFilePath)

        // Проверяем, существует ли файл
        if (file.exists()) {
            // Создаем Intent для установки APK
            val intent = Intent(Intent.ACTION_VIEW).apply {
                // Устанавливаем URI и MIME-тип для файла
                setDataAndType(
                    FileProvider.getUriForFile(
                        context,
                        "${context.packageName}.provider", // Указываем авторитет FileProvider
                        file
                    ),
                    "application/vnd.android.package-archive" // MIME-тип для APK файлов
                )
                // Добавляем флаг для предоставления разрешения на чтение URI
                addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)
                // Добавляем флаг для запуска новой задачи
                addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
            }
            // Запускаем активность для установки APK
            context.startActivity(intent)
        } else {
            // Логируем ошибку, если файл не существует
            Log.e("InstallApk", "APK file does not exist: $apkFilePath")
        }
    }

    fun downloadApk(
        context: Context,
        url: String
    ) {
        CoroutineScope(Dispatchers.IO).launch {
            _isUpdateInProcessFlow.emit(true)
        }
        val apkFilePath = context.pathToDownloadRelease()
        // Создаем вызов для загрузки файла
        val call = networkRepository.downloadFile(url)

        // Выполняем асинхронный запрос
        call?.enqueue(object : Callback&lt;ResponseBody&gt; {
            // Обрабатываем успешный ответ
            override fun onResponse(
                call: Call&lt;ResponseBody&gt;,
                response: Response&lt;ResponseBody&gt;
            ) {
                CoroutineScope(Dispatchers.IO).launch {
                    try {
                        val body = response.body()
                        // Проверяем, успешен ли ответ
                        if (response.isSuccessful &amp;&amp; body != null) {
                            // Создаем файл для записи данных
                            val file = File(apkFilePath)
                            // Получаем поток данных из тела ответа
                            val inputStream = body.byteStream()
                            // Создаем поток для записи данных в файл
                            val outputStream = FileOutputStream(file)
                            // Буфер для чтения данных
                            val buffer = ByteArray(8 * 1024)
                            var bytesRead: Int
                            var totalBytesRead: Long = 0
                            // Получаем длину содержимого
                            val contentLength = body.contentLength()

                            // Используем потоки для чтения и записи данных
                            while (inputStream.read(buffer).also { bytesRead = it } != -1) {
                                // запись данных из буфера в выходной поток
                                outputStream.write(buffer, 0, bytesRead)
                                totalBytesRead += bytesRead
                                // Отправляем процент загрузки
                                _percentageFlow.emit(totalBytesRead.toFloat() / contentLength)
                                if (contentLength == totalBytesRead) {
                                    // отправляем окончание процесса загрузки
                                    _isUpdateInProcessFlow.emit(false)
                                    // установка
                                    installApk(context = context)
                                }
                            }
                        } else {
                            //  Отправляем сигнал о неудаче
                            _isUpdateInProcessFlow.emit(false)
                            _percentageFlow.emit(0f)
                        }
                    } catch (e: Exception) {
                        // Обрабатываем исключение и отправляем сигнал о неудаче
                        _isUpdateInProcessFlow.emit(false)
                        _percentageFlow.emit(0f)
                    }
                }
            }

            // Обрабатываем ошибку при выполнении запроса
            override fun onFailure(call: Call&lt;ResponseBody&gt;, t: Throwable) {
                Log.e("status", "Download error: ${t.message}")
            }
        })
    }

}</code></pre><p><code>percentageFlow</code> и <code>isUpdateInProcessFlow</code> нужно для того, чтобы подписываться на эти <code>Flow</code> на экранах и соответствующе обновлять интерфейс. В конце статьи я покажу на видео, как я это делал. Здесь используется <code>SharedFlow</code>, поскольку от обычного <code>Flow</code> оно отличается тем, что <code>SharedFlow</code> хранит в себе всю историю изменения и ведет запись даже при отсутствии подписчиков, а при их появлении просто транслирует все данные, а обычное <code>Flow</code> ведет запись только когда на него подписываются. </p><p>В "Отправляем процент загрузки" я отправляю класс тип данных Float и в виде числа от 0 до 1, поскольку в <code>ProgressBar</code> обычно используется именно такой диапазон для отображения прогресса, а пользователю уже отображаю в процентах. </p><p>Вернемся к моменту получения данных о релизе, где нужно вставить обновление: </p><p>Здесь начинаем скачивание и downloader уведомляет <code>isUpdateInProcessFlow</code> о начале скачивания. </p><pre><code class="kotlin">val versionName = getVersionName(context = context)
