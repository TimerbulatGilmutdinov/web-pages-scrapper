article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Сегодня любой Java разработчик сходу сможет правильно ответить на вопрос «Как смапить дату и время из колонки таблицы БД на поле в Java классе?». Или нет? </p><p>На самом деле, нюансов по ходу решения этой задачи может возникнуть немало.</p><p>В новом переводе от команды <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> рассказывается про подробности работы с современным API <code>java.time</code>, правильный маппинг данных с учётом часовых поясов, устаревших типов <code>java.util.Date</code>, <code>Calendar</code> и многое другое.</p><hr/><p>Базы данных поддерживают различные типы данных для хранения информации о дате и времени. Наиболее используемые из них:</p><ul><li><p><strong>DATE</strong> — для сохранения даты без времени,</p></li><li><p><strong>TIME</strong> — для хранения времени без даты,</p></li><li><p><strong>TIMESTAMP</strong> — для хранения информации о дате и времени.</p></li></ul><p>Вы можете маппить все эти типы с помощью JPA и Hibernate.</p><p>Однако вам нужно понять, к какому типу Java вы хотите привязать столбец базы данных. Язык Java предоставляет множество классов для представления информации о дате и времени, например:</p><ul><li><p>Типы из API даты и времени (Date and Time API):<br/>Это такие классы, как <code>java.time.LocalDate, java.time.LocalDateTime, java.time.OffsetTime, java.time.OffsetDateTime, java.time.ZonedDateTime, java.time.Duration</code>. Современные приложения должны использовать именно их вместо устаревших типов.</p></li><li><p>SQL-специфичные типы: <code>java.sql.Date</code>, <code>java.sql.Time</code> и <code>java.sql.Timestamp</code>.</p></li><li><p>А также устаревшие типы: <code>java.util.Date</code> и <code>java.util.Calendar</code>.</p></li></ul><p>Современные версии стандарта Jakarta Persistence поддерживают большинство из перечисленных типов. Кроме того, Hibernate предоставляет проприетарную поддержку практически для всех остальных.</p><p>В этой статье я продемонстрирую различные варианты маппинга:</p><ol><li><p>Мы начнем с маппинга классов из пакета <code>java.time</code>, так как они наиболее актуальны для современных приложений.</p></li><li><p>Затем я покажу, как маппить классы из пакета <code>java.util</code>, которые до сих пор используются во многих приложениях.</p></li><li><p>И завершим статью разбором маппинга классов из пакета <code>java.sql</code>.</p></li></ol><h2>Содержание</h2><ul><li><p>Маппинг классов <code>java.time</code></p></li><li><p>Работа с <code>ZonedDateTime</code></p></li><li><p>Поддержка <code>ZonedDateTime</code> в Hibernate 6</p></li><li><p>Настройка обработки часовых поясов</p></li><li><p>Типы хранения часовых поясов в Hibernate (<code>TimeZoneStorageTypes</code>)</p></li><li><p>Поддержка <code>ZonedDateTime</code> в Hibernate 5</p></li><li><p>Маппинг классов <code>java.util</code></p></li><li><p>Маппинг классов <code>java.sql</code></p></li><li><p>Заключение</p></li></ul><h2>Маппинг классов java.time</h2><p>Классы API даты и времени являются наиболее популярным способом представления значений даты и времени. href="https://thorben-janssen.com/hibernate-5-date-and-time/"><u>Hibernate поддерживает несколько больше классов</u></a>, чем JPA. Вы можете легко добавить поддержку дополнительных классов, реализовав <a href="https://thorben-janssen.com/jpa-21-how-to-implement-type-converter/"><u>AttributeConverter</u></a>. В одной из предыдущих статей я использовал этот подход для <a href="https://thorben-janssen.com/jpa-tips-map-duration-attribute/"><u>маппинга атрибута типа Duration с помощью JPA</u></a>.</p><p>В зависимости от версии Hibernate следует быть осторожным при использовании <code>ZonedDateTime</code>. В Hibernate 5 работа с часовыми поясами и маппинг в столбец типа <code>TIMESTAMP</code> имеет несколько подводных камней. Hibernate 6 предоставляет больше контроля над этим процессом, но вам все равно нужно понимать, как это работает. Подробнее об этом я рассказываю в разделе <a href="https://thorben-janssen.com/hibernate-jpa-date-and-time/#timezone"><strong><u>Работа с ZonedDateTime</u></strong></a>.</p><p>Сначала давайте рассмотрим базовый маппинг сущности и простой тестовый пример с использованием этой сущности.</p><pre><code class="java">@Entity

	private LocalDate localDate;

	private LocalDateTime localDateTime;

	private OffsetTime offsetTime;

	private OffsetDateTime offsetDateTime;

	// Hibernate-specific - not supported by JPA

	private Duration duration;

	private Instant instant;

	private ZonedDateTime zonedDateTime;

	...

}</code></pre><pre><code class="java">MyEntity e = new MyEntity();

e.setLocalDate(LocalDate.of(2019, 7, 19));

em.persist(e);</code></pre><p>Классы API даты и времени четко определяют, содержат ли они информацию о дате и/или времени. Поэтому спецификация JPA и все реализующие её фреймворки могут сопоставлять их с соответствующими SQL-типами.</p><pre><code class="java">11:52:26,305 DEBUG SQL:94 - insert into MyEntity (duration, instant, localDate, localDateTime, offsetDateTime, offsetTime, sqlDate, sqlTime, sqlTimestamp, utilCalendar, utilDate, zonedDateTime, id) values (?, ?, ?, ?, ?, ?, ?, ?)

11:52:26,313 TRACE BasicBinder:65 - binding parameter [6] as [TIME] - [15:05:30+02:00]
11:52:26,324 TRACE BasicBinder:65 - binding parameter [7] as [TIMESTAMP] - [2019-07-18T15:05:30-04:00[UTC-04:00]]
11:52:26,324 TRACE BasicBinder:65 - binding parameter [8] as [BIGINT] - [1]</code></pre><h2>Работа с ZonedDateTime</h2><p>Объект <code>ZonedDateTime</code> представляет дату с информацией о времени и часовом поясе, и тип JDBC <code>TIMESTAMP_WITH_TIMEZONE</code> кажется идеальным вариантом. К сожалению, не все реляционные базы данных поддерживают этот тип, и Hibernate приходится выполнять преобразование типа.</p><p>Как уже упоминалось, обработка <code>ZonedDateTime</code> зависит от версии Hibernate:</p><ul><li><p><strong>Hibernate 6</strong> поддерживает различные варианты маппинга, которые дают полный контроль над обработкой часовых поясов.</p></li><li><p><strong>Hibernate 5</strong> нормализует часовой пояс объекта <code>ZonedDateTime</code>, что может вызывать проблемы.</p></li></ul><h2>Поддержка ZonedDateTime в Hibernate 6</h2><p>В Hibernate 6 был введен enum <code>TimeZoneStorageType</code>. Он позволяет настроить, как Hibernate будет обрабатывать информацию о часовом поясе, и дает возможность избежать любых нормализаций часовых поясов.</p><h3>Настройка обработки часовых поясов</h3><p>Вы можете использовать значения этого enum, чтобы задать обработку часового пояса по умолчанию, установив свойство <code>hibernate.timezone.default_storage</code> в конфигурации <strong>persistence.xml</strong>.</p><pre><code class="xml">&lt;persistence&gt;
    &lt;persistence-unit name="my-persistence-unit"&gt;
        &lt;description&gt;Hibernate example configuration - thorben-janssen.com&lt;/description&gt;
        &lt;exclude-unlisted-classes&gt;false&lt;/exclude-unlisted-classes&gt;
        &lt;properties&gt;
            &lt;property name="hibernate.timezone.default_storage" value="NORMALIZE"/&gt;	
			...
        &lt;/properties&gt;
    &lt;/persistence-unit&gt;
&lt;/persistence&gt;</code></pre><p>Вы также можете аннотировать атрибут <code>ZonedDateTime</code> с помощью аннотации <code>@TimeZoneStorage</code>.</p><pre><code class="java">@Entity
public class MyEntity {
	@TimeZoneStorage(TimeZoneStorageType.NATIVE)
	private ZonedDateTime zonedDateTime;
	...
}</code></pre><h3>Типы TimeZoneStorageTypes в Hibernate</h3><p>Вот краткий обзор различных типов <code>TimeZoneStorageTypes</code>, предоставляемых Hibernate. Более подробное описание каждого типа вы можете найти в <a href="https://thorben-janssen.com/hibernate-6-offsetdatetime-and-zoneddatetime"><u>этой статье</u></a>.</p><div><div в SQL-тип <code>TIMESTAMP</code> без информации о часовом поясе.</p><p>Это достигается путем нормализации временной метки. Hibernate преобразует <code>ZonedDateTime</code> в локальный часовой пояс JVM и сохраняет его в базе данных. При чтении <code>TIMESTAMP</code> Hibernate добавляет информацию о локальном часовом поясе.</p><pre><code class="java">MyEntity e = new MyEntity();
e.setZonedDateTime(ZonedDateTime.of(2019, 7, 18, 15, 05, 30, 0, ZoneId.of("UTC-4")));
em.persist(e);</code></pre><p>Hibernate 5 отображает информацию о часовом поясе в логах.</p><pre><code class="java">09:57:08,918 DEBUG SQL:92 - insert into MyEntity (zonedDateTime, id) values (?, ?)
09:57:08,959 TRACE BasicBinder:65 - binding parameter [1] as [TIMESTAMP] - [2019-07-18T15:05:30-04:00[UTC-04:00]]
09:57:08,961 TRACE BasicBinder:65 - binding parameter [2] as [BIGINT] - [1]</code></pre><p>Но в базе данных видно, что он преобразовал часовой пояс с UTC-4 в UTC+2, который является моим локальным часовым поясом.</p><figure class=""><img data-src="https://habrastorage.org/getpro/habr/upload_files/36d/c8b/34a/36dc8b34a9c05823d5d42d752f055f7b.png" height="99" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/36d/c8b/34a/36dc8b34a9c05823d5d42d752f055f7b.png" width="442"/></figure><p>Этот маппинг работает при соблюдении следующих условий:</p><ul><li><p>используется часовой пояс без учета летнего времени,</p></li><li><p>все экземпляры вашего приложения используют один и тот же часовой пояс,</p></li><li><p>вы никогда не меняете этот часовой пояс.</p></li></ul><h2>Очевидно, что это должно...</h2><p>Вы можете избежать этих проблем, настроив часовой пояс без учета летнего времени в <a href="https://thorben-janssen.com/jpa-persistence-xml/"><u>конфигурации </u><strong><u>persistence.xml</u></strong></a>. В таком случае Hibernate 5 будет использовать указанный часовой пояс вместо локального часового пояса вашей JVM. Я рекомендую использовать часовой пояс UTC.</p><pre><code class="xml">&lt;persistence&gt;
    &lt;persistence-unit name="my-persistence-unit"&gt;

        ...

        &lt;properties&gt;
            &lt;property name="hibernate.dialect" value="org.hibernate.dialect.PostgreSQLDialect" /&gt;
            &lt;property name="hibernate.jdbc.time_zone" value="UTC"/&gt;

            ...

        &lt;/properties&gt;
    &lt;/persistence-unit&gt;
&lt;/persistence&gt;</code></pre><p>Когда вы повторно запустите тест, вы не увидите никаких изменений в логах.</p><pre><code class="java">10:06:41,070 DEBUG SQL:92 - insert into MyEntity (zonedDateTime, id) values (?, ?)
10:06:41,107 TRACE BasicBinder:65 - binding parameter [1] as [TIMESTAMP] - [2019-07-18T15:05:30-04:00[UTC-04:00]]
10:06:41,108 TRACE BasicBinder:65 - binding parameter [2] as [BIGINT] - [1]</code></pre><p>Но если вы посмотрите на таблицу в базе данных, вы увидите, что Hibernate 5 теперь преобразовал <code>ZonedDateTime</code> в часовой пояс UTC.</p><figure class=""><img data-src="https://habrastorage.org/getpro/habr/upload_files/63e/137/853/63e1378530ef77774eab164f61c9700c.png" height="98" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/63e/137/853/63e1378530ef77774eab164f61c9700c.png" width="443"/></figure><h2>Маппинг классов java.util</h2><p>До выхода Java 8 классы <code>java.util.Date</code> и <code>java.util.Calendar</code> были наиболее используемыми для представления дат с и без информации о времени.</p><p>Конечно, вы можете маппить оба этих класса с помощью JPA и Hibernate. Однако для маппинга требуется указать несколько дополнительных данных. Нужно определить, хотите ли вы связать <code>java.util.Date</code> или <code>java.util.Calendar</code> со столбцом типа <code>DATE</code>, <code>TIME</code> или <code>TIMESTAMP</code>.</p><p>Для этого необходимо <a href="https://thorben-janssen.com/hibernate-tips-map-java-util-date-database-column/"><u>аннотировать атрибут сущности с помощью аннотации @Temporal</u></a> и указать значение из enum <code>TemporalType</code>. Вы можете выбрать одно из следующих значений:</p><ul><li><p><strong>TemporalType.DATE</strong> — для маппинга в столбец SQL-типа <code>DATE</code>;</p></li><li><p><strong>TemporalType.TIME</strong> — для маппинга в столбец SQL-типа <code>TIME</code>;</p></li><li><p><strong>TemporalType.TIMESTAMP</strong> — для маппинга в столбец SQL-типа <code>TIMESTAMP</code>.</p></li></ul><p>В следующем примере кода используется аннотация <code>@Temporal</code> для маппинга атрибута типа <code>java.util.Date</code> в столбец <code>TIMESTAMP</code> и атрибута типа <code>java.util.Calendar</code> в столбец <code>DATE</code>.</p><pre><code class="java">@Entity
public class MyEntity {

	@Temporal(TemporalType.TIMESTAMP)
	private Date utilDate;

	@Temporal(TemporalType.DATE)
	private Calendar utilCalendar;

	...

}</code></pre><p>Затем вы можете использовать эти атрибуты так же, как и любые другие атрибуты сущности.</p><pre><code class="java">MyEntity e = new MyEntity();
e.setUtilDate(new Date(119, 6, 18));
e.setUtilCalendar(new GregorianCalendar(2019, 6, 18));
em.persist(e);</code></pre><p>Если вы активируете <a href="https://thorben-janssen.com/hibernate-logging-guide/"><u>логирование SQL-запросов</u></a>, то в логах найдете следующие сообщения.</p><pre><code class="java">16:04:07,185 DEBUG SQL:92 - insert into MyEntity (utilCalendar, utilDate, id) values (?, ?, ?)

Это довольно сложный способ Hibernate показать, какой объект <code>Calendar</code> привязывается к параметру типа <code>DATE</code>. Но не беспокойтесь: если вы посмотрите на базу данных, то увидите, что Hibernate записал дату в столбец типа <code>DATE</code>.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/547/983/807/5479838070ae9ef33e1378786609b595.png" height="98" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/547/983/807/5479838070ae9ef33e1378786609b595.png" width="584"/></figure><h2>Маппинг классов java.sql</h2><p>Маппинг классов <code>java.sql.Date</code>, <code>java.sql.Time</code> и <code>java.sql.Timestamp</code> выполняется просто, так как эти классы соответствуют SQL-типам данных. Это позволяет провайдеру JPA, например Hibernate, автоматически определить маппинг.</p><p>Таким образом, без дополнительных аннотаций:</p><ul><li><p><code>java.sql.Date</code> отображается в SQL-тип <code>DATE</code>,</p></li><li><p><code>java.sql.Time</code> отображается в SQL-тип <code>TIME</code>,</p></li><li><p><code>java.sql.Timestamp</code> отображается в SQL-тип <code>TIMESTAMP</code>.</p></li></ul><pre><code class="java">@Entity
public class MyEntity {

	private java.sql.Date sqlDate;
	private Time sqlTime;
	private Timestamp sqlTimestamp;

	...

}</code></pre><p>Затем вы можете использовать эти атрибуты в своей бизнес-логике для сохранения информации о дате и времени в базе данных.</p><pre><code class="java">MyEntity e = new MyEntity();
e.setSqlDate(new java.sql.Date(119, 6, 18));
e.setSqlTime(new Time(15, 05, 30));
e.setSqlTimestamp(new Timestamp(119, 6, 18, 15, 05, 30, 0));
em.persist(e);</code></pre><p>А после активации <a href="https://thorben-janssen.com/hibernate-logging-guide/"><u>логирования SQL-запросов</u></a> вы можете увидеть, что Hibernate отображает атрибуты сущности в соответствующие SQL-типы.</p><pre><code class="java">06:33:09,139 DEBUG SQL:92 - insert into MyEntity (sqlDate, sqlTime, sqlTimestamp, id) values (?, ?, ?, ?)

06:33:09,154 TRACE BasicBinder:65 - binding parameter [4] as [BIGINT] - [1]</code></pre><h2>Заключение</h2><p>JPA и Hibernate могут маппить столбцы базы данных типов <code>DATE</code>, <code>TIME</code> и <code>TIMESTAMP</code> на различные классы Java. Вы можете связать их с:</p><ul><li><p><code>java.util.Date</code> и <code>java.util.Calendar</code>;</p></li><li><p><code>java.sql.Date</code>, <code>java.sql.Time</code> и <code>java.sql.Timestamp</code>;</p></li><li><p><code>java.time.LocalDate, java.time.LocalDateTime, java.time.OffsetTime, java.time.OffsetDateTime, java.time.ZonedDateTime, java.time.Duration</code>.</p></li></ul><p>Вам нужно лишь выбрать, какой тип Java использовать в своем коде. Я рекомендую использовать классы из пакета <code>java.time</code>. Они являются частью API даты и времени, который был представлен в Java 8. Эти классы значительно удобнее в использовании как для маппинга, так и в бизнес-логике.</p><figure class="full-width"><img
