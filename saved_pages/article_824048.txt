class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/6d4/a23/0c4/6d4a230c48500ec64676b18239991c51.png" height="793" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/6d4/a23/0c4/6d4a230c48500ec64676b18239991c51.png" width="1260"/></figure><p>Я хочу поделиться своим опытом создания кроссплатформенного приложения на базе <a href="https://kotlinlang.org/docs/multiplatform.html" rel="noopener noreferrer nofollow"><u>kotlin-multiplatform</u></a> (KMP), организацией его архитектуры и настройкой для работы с различными библиотеками. Я буду рассказывать о процессе создания приложения поэтапно и опишу каждую из особенностей его работы. Статья подойдет больше разработчикам, которые уже имеют опыт с многомодульными проектами в android и начинают изучать KMP. В конце я опишу свою реализацию архитектурного паттерна MVI и его применение в проекте. Также хотелось бы отметить, что в данной статье описано мое личное видение способов организации кода в проекте и я никого не принуждаю их использовать и статья носит лишь информативный характер.</p><p>Полный код проекта вы можете посмотреть - <a href="https://github.com/BraveOwlet/simple_mvi_example" rel="noopener noreferrer nofollow"><u>тут</u></a>.</p><p><strong>О приложении</strong></p><p>В таблице указаны подходы и библиотеки, используемые мной в тестовом приложении:</p><div><div class="table"><table><tbody><tr><td><p align="left">Архитектура приложения</p></td><td><p align="left">Многомодульная архитектура</p></td></tr><tr><td><p align="left">Архитектура внутри модуля</p></td><td><p align="left">Clean</p></td></tr><tr><td><p align="left">Архитектура UI</p></td><td><p align="left">MVI</p></td></tr><tr><td><p align="left">Локальное хранение информации</p></td><td><p align="left"><a href="https://developer.android.com/kotlin/multiplatform/room" rel="noopener noreferrer nofollow"><u>Room</u></a></p></td></tr><tr><td><p align="left">Работа в сети</p></td><td><p align="left"><a href="https://ktor.io/docs/client-create-multiplatform-application.html" rel="noopener noreferrer nofollow"><u>Ktor</u></a></p></td></tr><tr><td><p align="left">DI фреймворк</p></td><td><p align="left"><a href="https://insert-koin.io/docs/reference/koin-mp/kmp/" rel="noopener noreferrer nofollow"><u>Koin</u></a></p></td></tr><tr><td><p align="left">Способ отрисовки UI</p></td><td><p align="left"><a href="https://www.jetbrains.com/lp/compose-multiplatform/" rel="noopener noreferrer nofollow"><u>Compose Multiplatform</u></a></p></td></tr><tr><td><p align="left">Навигация</p></td><td><p align="left"><a href="https://github.com/adrielcafe/voyager" rel="noopener noreferrer nofollow"><u>Voyager</u></a></p></td></tr><tr><td><p align="left">Работа с ресурсами</p></td><td><p align="left"><a href="https://www.jetbrains.com/lp/compose-multiplatform/" rel="noopener noreferrer nofollow"><u>Compose Multiplatform</u></a></p></td></tr></tbody></table></div></div><p>Тестовое приложение будет использовать открытое REST-API <a href="https://dog.ceo/dog-api/" rel="noopener noreferrer nofollow"><u>DogAPI</u></a> для получения изображения собачек. После получения из сети ссылки на изображение будет возможность сохранить ее в БД для последующего отображения. Тестовое приложение будет разделено на 3 экрана.</p><ul><li><p>MainScreen — главный экран приложения с кнопками для перехода к следующим экранам,</p></li><li><p>DogsScreen — экран для загрузки картинки и сохранения ссылки на нее в БД, </p></li><li><p>SavedDogsScreen — экран со списком сохраненных картинок.</p></li></ul><figure class="full-width"><img alt="Экраны тестового приложения" data-src="https://habrastorage.org/getpro/habr/upload_files/dbf/260/47a/dbf26047a2558f594323a543fedc6ba9.png" height="821" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/dbf/260/47a/dbf26047a2558f594323a543fedc6ba9.png" title="Экраны тестового приложения" width="1217"/><div><figcaption>Экраны тестового приложения</figcaption></div></figure><ol><li><p><strong>Создание проекта</strong></p></li></ol><p>Для создания проекта удобнее всего использовать <a href="https://kmp.jetbrains.com/" rel="noopener noreferrer nofollow"><u>Kotlin Multiplatform Wizard</u></a>. С помощью него указывается основное имя проекта, id и выбираются платформы, под которые будет настроен проект. В моем случае проект будет работать на android и iOS платформах. Далее после создания проекта его можно открыть с помощью Android Studio. В итоге мы получим структуру проекта, выполненную в одном модуле.</p><figure class="full-width"><img alt="Генерация проекта через Kotlin Multiplatform Wizard" data-src="https://habrastorage.org/getpro/habr/upload_files/0fa/9c0/17f/0fa9c017f0bc83c78195d6c7722ac935.png" height="865" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/0fa/9c0/17f/0fa9c017f0bc83c78195d6c7722ac935.png" title="Генерация проекта через Kotlin Multiplatform Wizard" width="1156"/><div><figcaption>Генерация проекта через <a href="https://kmp.jetbrains.com/" rel="noopener noreferrer nofollow"><u>Kotlin Multiplatform Wizard</u></a></figcaption></div></figure><ol start="2"><li><p><strong>Принцип разделения логики для платформ в KMP</strong></p></li></ol><p>В основе кроссплатформенного модуля лежит несколько подмодулей: </p><ul><li><p>common-подмодуль, где реализуется общая для всех платформ логика,</p></li><li><p>platform-подмодули, где реализуется логика для каждой из платформ отдельно.</p></li></ul><p>Для подключения той или иной платформы необходимо прописать их в build.gradle.kts файле модуля, также в нем описываются зависимости для каждой из платформ.</p><figure class="full-width"><img alt="Настройка  build.gradle.kts" data-src="https://habrastorage.org/getpro/habr/upload_files/031/082/c48/031082c486a7b44410f4e3db569327f4.png" height="764" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/031/082/c48/031082c486a7b44410f4e3db569327f4.png" title="Настройка  build.gradle.kts" width="1308"/><div><figcaption>Настройка  build.gradle.kts</figcaption></div></figure><p>Для доступа к платформозависящей логики в common-подмодуле необходимо ее объявить и пометить ключевым словом expect, а далее в каждом подмодуле платформы реализовать данный код с ключевым словом actual. Например, в сгенерированном проекте можно увидеть следующий код:</p><figure class="full-width"><img alt="Реализация кода для разных платформ" data-src="https://habrastorage.org/getpro/habr/upload_files/09c/91a/513/09c91a5136f7e65d0d5f75a27a642c3b.png" height="662" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/09c/91a/513/09c91a5136f7e65d0d5f75a27a642c3b.png" title="Реализация кода для разных платформ" width="1508"/><div><figcaption>Реализация кода для разных платформ</figcaption></div></figure><ol start="3"><li><p><strong>Многомодульная архитектура</strong></p></li></ol><p>В сгенерированном проекте имеется всего один заглавный модуль — composeApp. Для реализации многомодульности необходимо определить архитектуру, в которой она будет выполнена. </p><p>В моем варианте она будет делится на common, core, component, feature и app модули. </p><div><div class="table"><table><tbody><tr><td data-colwidth="123" width="123"><p align="left">Common</p></td><td><p align="left">Модули, которые не зависят от проекта, их можно с легкостью переиспользовать в любых проектах. Желательно независящие от других модулей. Своего рода локальные библиотеки.</p></td></tr><tr><td data-colwidth="123" width="123"><p align="left">Core</p></td><td><p align="left">Модули, которые зависят от проекта, в них находится общая логика, которую используют компоненты и фичи. Могут зависеть от Common и других Core модулей.</p></td></tr><tr><td data-colwidth="123" width="123"><p align="left">Component</p></td><td><p align="left">Модули, в которых находится логика работы с каким-либо компонентом. Могут зависеть друг от друга, от Common и Core модулей.</p></td></tr><tr><td data-colwidth="123" width="123"><p align="left">Feature</p></td><td><p align="left">Модули, в которых находится логика работы с каким-либо экраном или группой экранов. Как правило, делятся на API и IMPL части для возможности ссылаться друг на друга. Могут зависеть друг от друга, от Component, Common и Core модулей.</p></td></tr><tr><td data-colwidth="123" width="123"><p align="left">App</p></td><td><p align="left">Главный модуль приложения, который собирает в себе все остальные модули и проводит начальную инициализацию проекта.</p></td></tr></tbody></table></div></div><figure class="full-width"><img alt="Общая архитектура проекта" data-src="https://habrastorage.org/getpro/habr/upload_files/174/2bd/e46/1742bde46b6b922b9b10da7a7f2b796e.png" height="965" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/174/2bd/e46/1742bde46b6b922b9b10da7a7f2b796e.png" title="Общая архитектура проекта" width="1035"/><div><figcaption>Общая архитектура проекта</figcaption></div></figure><p>После завершения создания приложения в структуре моего проекта будут следующие модули:</p><figure class="full-width"><img alt="Структура модулей в проекте" data-src="https://habrastorage.org/getpro/habr/upload_files/0ab/17e/9c0/0ab17e9c0c4f0e91abf2120dab3fca96.png" height="895" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/0ab/17e/9c0/0ab17e9c0c4f0e91abf2120dab3fca96.png" title="Структура модулей в проекте" width="1116"/><div><figcaption>Структура модулей в проекте</figcaption></div></figure><p>Далее я расскажу о особенностях каждого модуля.</p><ol start="4"><li><p><strong>app-модуль</strong></p></li></ol><p>Главным модулем приложения является app-модуль, он зависит от всех остальных модулей и инициализирует приложение.</p><p>Функции app модуля:</p><ul><li><p>точка входа в приложение;</p></li><li><p>держатель DI графа;</p></li><li><p>старт compose-ui и инициализация навигации;</p></li><li><p>инициализация.</p></li></ul><p><strong>Точка входа.</strong></p><p>Для каждой из платформ необходима своя точка входа в приложение, которая располагается в app модуле в своем подмодуле платформы:</p><ul><li><p>для android - Application; </p></li><li><p>для iOS - MainViewController (используется в xcode проекте). </p></li></ul><p><strong>Держатель DI графа.</strong></p><p>В этом приложении для реализации инверсии зависимостей я использую koin. Для подключения DI-графа к app-модулю необходимо вызвать метод startKoin и указать в его лямбде необходимые koin-модули. </p><ul><li><p>Для android данный метод вызывается в методе onCreate класса Application;</p></li><li><p>Для iOS необходимо вызвать метод initKoin из iOSApp.swift проекта xcode. </p></li></ul><p>Для сбора всех koin-модулей используется список appModules, в котором находятся все koin-модули от других модулей проекта. src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/ba3/b2b/e88/ba3b2be887797a3caee5dc69301a90a8.png" title="Буфер состояний" width="599"/><div><figcaption>Буфер состояний</figcaption></div></figure><p>Далее определен eventFlow с MviEvent, который лениво получается из канала eventChannel</p><figure class="full-width"><img alt="Flow с Event" data-src="https://habrastorage.org/getpro/habr/upload_files/bf0/2c3/eb8/bf02c3eb85e2006d40e4ee3841dd9b39.png" height="594" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/bf0/2c3/eb8/bf02c3eb85e2006d40e4ee3841dd9b39.png" title="Flow с Event" width="1600"/><div><figcaption>Flow с Event</figcaption></div></figure><p>и stateFlow для MviState, который лениво получается из bufferStateFlow.</p><p>При страте подписки на bufferStateFlow происходит:</p><ol><li><p>подписка на канал actionChannel, где каждый MviAction отправляется в Actor;</p></li><li><p>подписка на канал effectChannel, где каждый MviEffect преобразуется в MviState c помощью Reducer и отправляется в bufferStateFlow;</p></li><li><p>происходит вызов Bootstrap.</p></li></ol><figure class="full-width"><img alt="Получение Flow с состоянием" data-src="https://habrastorage.org/getpro/habr/upload_files/d92/f42/da5/d92f42da5fb86d1f45f9d669a08bea4e.png" height="970" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d92/f42/da5/d92f42da5fb86d1f45f9d669a08bea4e.png" title="Получение Flow с состоянием" width="1030"/><div><figcaption>Получение Flow с состоянием</figcaption></div></figure><p>Схема работы MVI выглядит следующим образом:</p><figure class="full-width"><img alt="Итоговая схема работы MVI" MVI</figcaption></div></figure><p><strong>Вторая часть — mvi-koin-voyager</strong></p><p>Тут описан абстрактный класс MviModel и интерфейс MviView. По факту, это модуль с конкретной реализацией Model и View на основе библиотек koin и voyager. Данную реализацию можно заменить на любую другую, например, с использованием стандартной VewModel и hilt в случае использования только на android. </p><p>В текущем примере, MviModel — реализует интерфейс ScreenModel из библиотеки Voyager и ранее упомянутый интерфейс Mvi, а также инкапсулирует объект mvi реализующий интерфейс Mvi и объявляет методы bootstrap, actor и reducer для передачи их в реализацию объекта mvi. Можно заметить, что почти все описание MviModel можно вынести в general часть, кроме определения coroutineScope, в котором работает MVI. Однако я не стал этого делать, чтобы иметь возможность унаследовать MviModel от ViewModel android, которая является абстрактным классом.</p><figure class="full-width"><img alt="MviModel на основе ScreenModel от voyager" data-src="https://habrastorage.org/getpro/habr/upload_files/793/fc6/284/793fc6284b8ff26c8c879dad7c09c252.png" height="875" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/793/fc6/284/793fc6284b8ff26c8c879dad7c09c252.png" title="MviModel на основе ScreenModel от voyager" width="1142"/><div><figcaption>MviModel на основе ScreenModel от voyager</figcaption></div></figure><p>Дальнейшая реализация MviModel в фиче-модуле происходит посредством koin с тегом названия класса, реализующего интерфейс MviView. Для этого создан вспомогательный метод, возвращающий KoinDefinition. В него с помощью дженериков передается класс реализующий MviView и имя данного класса используется в качестве ключа для создания фабрики MviModel.</p><figure class="full-width"><img alt="Предоставление MviModel через DI" data-src="https://habrastorage.org/getpro/habr/upload_files/bd9/44a/248/bd944a248104181279ae64c6122e7a63.png" height="255" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/bd9/44a/248/bd944a248104181279ae64c6122e7a63.png" title="Предоставление MviModel через DI" width="1600"/><div><figcaption>Предоставление MviModel через DI</figcaption></div></figure><p>MviView является интерфейсом, который наследуется от интерфейса Screen из библиотеки Voyager. В нем мы определяем метод Content, в котором по имени текущей реализации данного интерфейса находится MviModel, и вызываем метод для отрисовки compose-ui - content, который будет определен конкретной реализаций. В данный метод передается State, Flow с MviEvent и лямбда для выполнения MviAction. </p><figure class="full-width"><img alt="MviView на основе Screen из voyager" data-src="https://habrastorage.org/getpro/habr/upload_files/c3b/bc9/090/c3bbc9090ba75e2c6038f3c84ec77791.png" height="873" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/c3b/bc9/090/c3bbc9090ba75e2c6038f3c84ec77791.png" title="MviView на основе Screen из voyager" width="1144"/><div><figcaption>MviView на основе Screen из voyager</figcaption></div></figure><p>Также имеется метод, упрощающий подписку на Flow с MviEvent.</p><figure class="full-width"><img alt="Подписка на Flow с Event" data-src="https://habrastorage.org/getpro/habr/upload_files/f83/873/e40/f83873e404bc155bec8ecca51c52df91.png" height="346" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/f83/873/e40/f83873e404bc155bec8ecca51c52df91.png" title="Подписка на Flow с Event" width="1600"/><div><figcaption>Подписка на Flow с Event</figcaption></div></figure><p>Далее возможно создавать фичи-модули с использованием вышеупомянутых файлов и реализовывать в них логику с помощью реализаций MviAction, MviEffect, MviEvent и MviState.</p><ol start="11"><li><p><strong>Пример реализации фичи</strong></p></li></ol><p>Модуль фичи разделен на Api- и Impl-модули для возможности ссылаться друг на друга. </p><ul><li><p>в Api-модуле представлен интерфейс с функционалом данного модуля. В данном случае это два метода для вызова экранов;</p></li><li><p>в Impl-модуле находится реализация интерфейса из Api-модуля и реализация самой фичи (экранов).</p></li></ul><figure class="full-width"><img alt="Api и Impl части модуля фичи" data-src="https://habrastorage.org/getpro/habr/upload_files/2b8/5fa/0ac/2b85fa0ac708300dc3993f05f5f33ef5.png" height="454" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/2b8/5fa/0ac/2b85fa0ac708300dc3993f05f5f33ef5.png" title="Api и Impl части модуля фичи" width="2201"/><div><figcaption>Api и Impl части модуля фичи</figcaption></div></figure><p>Реализация предоставляется посредством DI, а именно koin-модуля, также в нем предоставляются MviModel экранов с помощью экстеншена упомянутого ранее. Данный koin-модуль в дальнейшем добавляется в список модулей проекта в app модуле.</p><figure class="full-width"><img alt="DI модуля фичи" data-src="https://habrastorage.org/getpro/habr/upload_files/70a/682/c0d/70a682c0de7a724bcec74a2fc37242ab.png" height="360" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/70a/682/c0d/70a682c0de7a724bcec74a2fc37242ab.png" title="DI модуля фичи" width="1600"/><div><figcaption>DI модуля фичи</figcaption></div></figure><p>Сам экран описывается четырьмя основными элементами:</p><ul><li><p>экран - реализует MviScreen;</p></li><li><p>модель - реализует MviModel;</p></li><li><p>compose-контент;</p></li><li><p>модели реализующие MviAction, MviEffect, MviEvent и MviState.</p></li></ul><figure class="full-width"><img alt="Структура модуля фичи" data-src="https://habrastorage.org/getpro/habr/upload_files/a3d/328/ddd/a3d328ddd8c9f626bb543eadadabcc0a.png" height="1368" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/a3d/328/ddd/a3d328ddd8c9f626bb543eadadabcc0a.png" title="Структура модуля фичи" width="730"/><div><figcaption>Структура модуля фичи</figcaption></div></figure><p>Модели для экрана с сохраненными собачками выглядят следующим образом:</p><ul><li><p>SavedDogsScreenAction — события на экране, в данном случае имеется событие class="full-width"><img alt="Объекты MVI в модуле фичи" data-src="https://habrastorage.org/getpro/habr/upload_files/76e/e47/c89/76ee47c89c8a9bbf2bc394c3a0d8504b.png" height="914" Вывод лога формируется следующим образом: </p><figure class="full-width"><img alt="Структура сообщения из логов" data-src="https://habrastorage.org/getpro/habr/upload_files/a15/64d/295/a1564d2958ecae6624814ed0412836d8.png" height="446" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/a15/64d/295/a1564d2958ecae6624814ed0412836d8.png" title="Структура сообщения из логов" width="2240"/><div><figcaption>Структура сообщения из логов</figcaption></div></figure><p>Логирование происходи на всех этапах: обновление состояния, получение MviEvent, получение MviAction, получение MviEffect.</p><figure class="full-width"><img alt="Вывод логов в LogCat" data-src="https://habrastorage.org/getpro/habr/upload_files/fac/246/7fa/fac2467fa7ee34a8cfaa3a37afb0529b.png" height="277" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/fac/246/7fa/fac2467fa7ee34a8cfaa3a37afb0529b.png" title="Вывод логов в LogCat" width="1600"/><div><figcaption>Вывод логов в LogCat</figcaption></div></figure><p>Данный функционал позволяет полностью отслеживать процесс работы того или иного экрана, а также возможноcть расширить функционал логирования и добавить запись логов в файл или БД, с последующей отправкой на удаленный сервер.</p><p><strong>Итог</strong></p><p>Описанный выше подход к архитектурному решению для кроссплатформенного приложения является, на мой взгляд, наиболее удобным и практичным с точки зрения расширяемости. Также хочется отметить что реализацию MVI возможно с легкостью менять используя другие библиотеки для навигации и инверсии зависимостей.</p><p>В дальнейшем планирую покрыть данное приложение тестами, а именно хотелось бы протестировать работоспособность MVI реализации при большом объеме данных. </p></div></div></div><!-- --><!-- --></div><!-- --><!-- --></div><!--]--><!-- --><div
