article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Привет, хабр! Сегодня я хочу рассказать про свой опыт взаимодействия с языком kotlin.</p><p>Представлюсь – я java разработчик, работаю в крупном банке, создаю (и поддерживаю существующие) микросервисы.</p><p>Небольшая ремарка: <strong>я не собираюсь становиться Android разработчиком</strong>, ни сейчас, ни в будущем, поэтому, когда я заинтересовался новым языком, не принимал в расчет аргументы про различные удобства мобильной разработки на нем, и руководствовался только удобством языка в целом для бэкэнда.</p><p>Итак, почему я решил изучить kotlin. Ну, во-первых, прожужали все уши, мол сокращение объема код, лаконичность, читаемость и сахар.</p><p>Во-вторых, Kotlin полностью совместим с Java. Для понимания, ниже скриншот того, как выглядит содержание одного из пакетов intellij плагина git4idea:</p><figure class=""><img alt="кусок структуры исходников плагина git4idea" data-src="https://habrastorage.org/getpro/habr/upload_files/089/98c/4fa/08998c4fad8201dbcca8f7444c8b1664.png" height="301" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/089/98c/4fa/08998c4fad8201dbcca8f7444c8b1664.png" title="кусок структуры исходников плагина git4idea" width="425"/><div><figcaption>кусок структуры исходников плагина git4idea</figcaption></div></figure><p>(Java и Kotlin классы идут вперемешку, код при этом остается читаемым (<em>местами</em>))</p><p>Да, проекты можно мигрировать с Java на Kotlin постепенно, как по мне – киллер-фича.</p><p>Так что, я подумал, почему бы и нет.</p><p>Вот вещи, к которым я привык за свои полгода пет-проектирования на kotlin, и которыми лично мне удобно пользоваться:</p><ol><li><p>Закрытые для расширения классы и методы. Все по умолчанию final, в отличии от Java, где, наоборот, любой класс открыт для расширения. Kotlin же придерживается принципа, описанного еще в Effective Java Дж.Блоха, который звучит так: </p><blockquote><p><em>"классы и методы должны быть закрыты и не подлежат расширению или переопределению, если только у нас нет веской причины для их расширения или переопределения. И когда мы решаем, что наш класс должен быть открыт для расширения, мы должны документировать последствия переопределения любого метода.» </em><a href="https://www.baeldung.com/kotlin/open-keyword" rel="noopener noreferrer nofollow"><em>Источник</em></a></p></blockquote><p>Для того, чтобы сделать класс в kotlin расширяемым, мы должны использовать ключевое слово open, которое <u>не обладает транзитивным свойством </u>– это означает, что когда Class2 наследуется от open class Class1, мы не сможем унаследовать Class3 от Class2, без навешивания на него ключевого слова open.</p><p></p></li><li><p>Null-safe. Возможность на уровне компиляции разрешить/запретить(по умолчанию) присвоение null в переменную – лайк.</p><pre><code class="kotlin">var a: String = "abc"
a = null                // compile error

var b: String? = "xyz"
b = null                // no problem</code></pre><p></p></li><li><p>Геттеры и сеттеры по умолчанию. Любое поле не извлекается напрямую, более того под капотом оно является private(по умолчанию) и может быть доступно только геттеру. Когда вы пытаетесь получить поле вашего объекта, вы на самом деле вызываете метод get</p><p> <code>val isEmpty: Boolean</code></p><p>эквивалентно следующему Java коду:   </p><pre><code class="java">private final Boolean isEmpty;

public Boolean isEmpty() {
    return isEmpty;
}</code></pre><p>Ну, и:</p><p><code>var someProperty: String = "defaultValue"</code></p><p>также эквивалентно:</p><pre><code class="kotlin"> var someProperty: String = "defaultValue"
    get() = field
    set(value) { field = value }</code></pre><p>При желании, разумеется, можно ограничить доступ к геттерам или сеттерам:</p><pre><code class="kotlin"> var isEmpty: Boolean = true
    private set </code></pre><p><em>Примеры </em><a href="https://stackoverflow.com/questions/37906607/getters-and-setters-in-kotlin" rel="noopener noreferrer nofollow"><em>скоммуниздил</em></a></p><p></p></li><li><p>Однострочные методы. Буду краток : при объявлении метода вида:</p><pre><code class="kotlin"> fun doSomething(): String{
    return “doing something”
} </code></pre><p>Kotlin позволяет убрать явное объявление возвращаемого типа, фигурные скобки и ключевое слово return</p><p>Выглядеть это начинает следующим образом:</p><pre><code class="kotlin"> fun doSomething() = “doing something”</code></pre><p>Сначала было непривычно (особенно когда такие объявления идут после блока инициализации полей класса), но потом привык</p><p></p></li><li><p> Упрощена большая вложенность при сравнении строк и любых других сложных типов в <strong>If</strong> - чуваки <a href="https://kotlinlang.org/docs/equality.html#structural-equality" rel="noopener noreferrer nofollow">зашили</a> <strong>equals</strong> в  <strong>==</strong> (начинает работать, если переопределить метод equals явно, либо если пометить класс ключевым словом data.)</p><p>Добавлю, что если уж вам нужно сравнить именно ссылки на объекты, в kotlin есть отдельный оператор для этого: тройное равно <strong>===</strong></p><p>Java:</p><pre><code class="java"> if (str1.equals(str2.equals(errString)? "default":str2)){
     //... 
 }</code></pre><p>Kotlin:</p><pre><code class="kotlin"> if (str1 == if (str2 == errString) "default" else str2) {
   //... 
 }</code></pre><p>Сравнение объектов на равенство:</p><p>Java:</p><pre><code class="java">@EqualsAndHashCode
@AllArgsConstructor
public class MyClass {
   private String name;
   private int value;
 }

 
public static void main(String[] args) {
  MyClass first = new MyClass("name", 5);
  MyClass second = new MyClass("name", 5);
  System.out.println(first.equals(second));//true
  System.out.println(first == second);//false 
}</code></pre><p>Kotlin:</p><pre><code class="kotlin">data class MyClassK(var name: String?, var value: Int)

fun main() {
  val first = MyClassK("name",10)
  val second = MyClassK("name",10)
  println(first == second)//true
  println(first === second)//false
}</code></pre><p>               </p></li><li><p>Именованные аргументы. В сочетании с аргументами по умолчанию именованные аргументы избавляют от необходимости использовать Строителей (<a href="https://radioprog.ru/post/1465" rel="noopener noreferrer nofollow">паттерн Builder</a>)</p><p>Вместо следующего Java-кода:</p><pre><code class="java">@Builder //lombok annotation
public class MyClass {
  private String name;
  private int value;
  private double rating;
}

MyClass myClass = MyClass.builder()
   .name("MyJavaObj")
   .value(10)
   .rating(4.99f)
   .build();</code></pre><p>На kotlin мы можем сделать так:</p><pre><code class="kotlin">class MyClassK(name: String? = null, value: Int = 0, rating:Double = 0.0)

fun main() {
  var myClassK = MyClassK(value = 10, name = "MyKotlinObj", rating = 4.99)
} </code></pre><p></p></li><li><p>Перегрузка операторов (Operator Overloading)</p><p>В kotlin заранее определен набор операторов, которые можно перегружать для улучшения читабельности:</p><pre><code class="kotlin">data class Vec(val x: Float, val y: Float) {
    operator fun plus(v: Vec) = Vec(x + v.x, y + v.y)
}

val v = Vec(2f, 3f) + Vec(4f, 1f)</code></pre><p> <em>пример позаимствован </em><a href="https://habr.com/ru/companies/vk/articles/329294/" rel="noopener noreferrer nofollow"><em>отсюда </em></a></p><p></p></li><li><p>Функции-расширения (Extension Functions)</p><p>Если коротко: мы можем дополнить существующие классы нашим функционалом без наследования, и относиться к этому функционалу, как к родному на протяжении всей дальнейшей программы. </p><pre><code class="kotlin">fun String.format() = this.replace(' ', '_')

val str = "hello world"
val formatted = str.format()</code></pre><p>или, например:</p><pre><code class="kotlin"> val result = str.removeSuffix(".txt")</code></pre><p>(В java я бы вынес это в статический метод какого-нибудь класса StringUtils)</p><p>Удобно, не так ли? Тем более что IDE подсказывает нашу функцию среди прочих всплывающих.</p><p></p></li><li><p> Работа с NPE в цепочке вызовов. Если мы хотим извлечь из сложного, составного объекта, какую-то маленькую часть, но при этом нет гарантий, что на этом пути нас не ждет null, нам приходится делать следующее:</p><p>Java:</p><pre><code class="java"> try{
   Status status = journal.getLog().getRow(0).getStatus();
   if(status == null)
     throw new NullPointerException(“null status detected in log”);
 } catch(NullPointerException e){
   status == Status.ERROR;
   //logger.error(“Journal is not correct”);
 }</code></pre><p>Kotlin:</p><pre><code class="kotlin">var status: Status? = journal?.log?.row?.status
status  =  status ?: Status.ERROR</code></pre><p>    либо</p><pre><code class="kotlin">var status: Status? = journal?.log?.row?.status
if(status.isNull()){
  status  = Status.ERROR
  logger.error(“Journal is not correct”)
}</code></pre><p>Как же это прекрасно, что теперь я могу не бояться NPE в цепочке вызовов. В Java, как можно видеть, приходится оборачивать небезопасную цепочку вызовов в громоздкий try-cath, но черт побери, иногда я хочу, чтобы при возврате null где-то в цепочке, в переменную просто присваивался null. Именно этот функционал мне и дает kotlin.</p></li></ol><blockquote><p>Не затронул здесь корутины, поскольку я недостаточно с ними знаком, да и в целом у меня нет большого опыта работы с многопоточкой, чтобы я мог высказывать свое экспертное мнение. Но вообще, посмотрел <a href="https://youtu.be/rB5Q3y73FTo?si=piObKnscuv1S9vtg" rel="noopener noreferrer nofollow">доклад</a> и вам советую. Может соберусь, изучу тему и напишу что-то про это.</p></blockquote><p><em>Лирическое завершение:</em></p><p>Вообще нахожусь в больших раздумьях насчет языка: я очень долго программирую на Java и она мне, как то роднее, что ли. <br/><em>(Когда вижу в проекте файлы с функциями, отдельно от классов, периодически дергаюсь.)</em></p><p>Много нюансов и изменений, как ни крути, хотя и совместимость.<br/>Слушал недавно несколько подкастов с гостями – разработчиками и продактами JetBrains, ребята так увлеченно рассказывали про язык, его перспективы и горизонты, мол как Java, только лучше, Kotlin Multiplatform и тд. <br/> <br/>Но все-таки остается фактом, что массово kotlin в бэкэнд не пошел, а занял нишу мобилок, хотя может это я чего-то не знаю. Может за моей спиной в проде все тайно уже переписали свои жаба проекты на kotlin, а мне не сказали.</p><p>Да и вообще есть ли такое понятие, как основной язык? Может не надо так к этому относиться, а придерживаться принципа «под каждую задачу свой инструмент». Но тогда под какие задачи бэкэнда kotlin, а под какие java?
