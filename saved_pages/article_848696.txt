xmlns="http://www.w3.org/1999/xhtml"><p>В новой статье от команды <a href="https://amplicode.ru">Amplicode</a> я расскажу, как можно создать несколько Docker Compose файлов для разных нужд. Например, для продакшена и разработки, и при этом не утонуть в копипасте.</p><p>Статья также доступна в формате видео, так что можно и смотреть, и читать — как вам удобнее!</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/6706430e28dfc09d9e99bf8d" data-style="" id="6706430e28dfc09d9e99bf8d" width=""></div><p><a href="https://rutube.ru/video/45abc6177f95521673ea397a39258ed7/">Смотреть на Rutube</a> || <a href="https://youtu.be/IOQ7MzXoDnE">Смотреть на YouTube</a></p><hr/><p>Итак, сегодня мы хотим получить два Docker Compose файла. Один будет полезен для продакшена и будет содержать три сервиса: наше Spring Boot приложение, PostgreSQL и Kafka, а второй будет использоваться во время разработки и точно так же будет включать в себя PostgreSQL и Kafka, а также инструменты для работы с ними, а именно pgAdmin и Kafka UI.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/f28/84c/969/f2884c9690f36ccf66766715c3bb339d.png" height="669" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/f28/84c/969/f2884c9690f36ccf66766715c3bb339d.png" width="1600"/></figure><details class="spoiler"><summary>Docker Compose для продакшена!?</summary><div class="spoiler__content"><p>Отметим, что касательно использования Docker Compose в продакшене однозначного мнения нет. Одни считают, что Docker Compose не подходит для продакшена, и лучше использовать продвинутые системы оркестрации, такие как Kubernetes. Другие считают, что это <a href="https://www.reddit.com/r/docker/comments/mqwm64/comment/guj6uf2/?utm_source=share&amp;utm_medium=web3x&amp;utm_name=web3xcss&amp;utm_term=1&amp;utm_content=share_button"><u>вполне допустимо при определенных условиях</u></a>. В общем, как часто бывает в программировании, ответ зависит от контекста.</p></div></details><h2>Способы решения поставленной задачи </h2><p>Возвращаясь к поставленной задаче: как вы можете заметить, Kafka и PostgreSQL должны присутствовать в обоих Docker Compose файлах, и есть несколько способов решить эту задачу.</p><h3>Использование одного Docker Compose файла с профилями</h3><p>Первый способ, которым мы можем решить поставленную задачу — это вообще отказаться от двух Docker Compose файлов, а создать только один, в котором описать все сервисы и использовать профили, чтобы указать, какие сервисы нужны во время разработки, а какие пригодятся только в продакшене.</p><pre><code class="yaml">#многие свойства сервисов не указаны для простоты восприятия 
services: 
  spring-petclinic: 
    image: spring-petclinic:latest 
    profiles: 
      - prod 
  postgres: 
    image: postgres:16.3 
    profiles: 
      - prod 
      - dev 
  kafka: 
    image: confluentinc/cp-kafka:7.6.1 
    profiles: 
      - prod 
      - dev 
  kafkaui: 
    image: provectuslabs/kafka-ui:v0.7.2 
    profiles: 
      - dev 
  pgadmin: 
    image: dpage/pgadmin4:8.12.0 
    profiles: 
      - dev </code></pre><p>Однако, этот вариант немного ограничен с точки зрения использования. Например, я не смогу указать для разных профилей, что для них должны использоваться разные порты или скрипты инициализации.</p><h3>Создание отдельных Docker Compose файлов для каждого окружения </h3><p>Второй способ, который я могу выбрать для решения этой задачи — это просто скопировать нужные мне сервисы в разные Docker Compose файлы. Однако, если придерживаться этого варианта, можно наткнуться на все типичные проблемы, которые связаны с копированием и вставкой содержимого, например, изменить версию сервиса в одном Docker Compose файле, но забыть сделать это в другом. И если первоначально у меня было всего лишь два Docker Compose файла, в дальнейшем их может неожиданно стать четыре, потом восемь, и так далее. Поддерживать все это в рабочем состоянии будет все сложнее и сложнее.</p><h3>Использование include и extends для переиспользования сервисов </h3><p>Наконец, есть третий способ, про который, возможно, не все знают. Этот способ подразумевает использование конструкций <a href="https://docs.docker.com/compose/how-tos/multiple-compose-files/include/"><u>include</u></a> и <a href="https://docs.docker.com/compose/how-tos/multiple-compose-files/extends/"><u>extends</u></a> в Docker Compose для переиспользования сервисов.</p><p>Конструкция <code>include</code> позволяет включить один Docker Compose файл в другой, по сути предоставляя аналог запуска сразу нескольких Docker Compose файлов из терминала.</p><p>Файл <code>services.yaml</code>:</p><pre><code class="yaml">#многие свойства сервисов не указаны для простоты восприятия 
services: 
  postgres: 
    image: postgres:16.3 
  kafka: 
    image: confluentinc/cp-kafka:7.6.1 </code></pre><p> Файл <code>app-compose.yaml</code>:</p><pre><code class="yaml">#многие свойства сервисов не указаны для простоты восприятия 
include: 
  - services.yaml 
 
services: 
  spring-petclinic: 
    image: spring-petclinic:latest</code></pre><p>Этот подход действительно довольно удобен, если вам нужно просто переиспользовать одни и те же сервисы для разных Docker Compose файлов без какой‑либо дополнительной настройки.</p><p>И этот подход в том числе поддерживается Amplicode с точки зрения визуального отображения:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/dca/f0d/c9a/dcaf0dc9a447409c3fb4b0216f23936c.png" height="962" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/dca/f0d/c9a/dcaf0dc9a447409c3fb4b0216f23936c.png" width="1600"/></figure><p>Но если вам нужно произвести тонкую настройку сервиса под свои нужды, то здесь больше подойдет ключевое слово <code>extends</code>.</p><p>С его помощью, точно так же, как и с помощью <code>include</code>, можно включить сервисы из другого Docker Compose файла в текущий. При этом у нас появляется возможность конфигурировать его свойства.</p><p>Файл <code>services.yaml</code>:</p><pre><code class="yaml">#многие свойства сервисов не указаны для простоты восприятия 
services: 
  postgres: 
    image: postgres:16.3 
  kafka: 
    image: confluentinc/cp-kafka:7.6.1 </code></pre><p>Файл <code>app-compose.yaml</code>:</p><pre><code class="yaml">services: 
  spring-petclinic: 
    image: spring-petclinic:latest 
  postgres: 
    extends: 
      service: postgres 
      file: services.yaml 

  kafka: 
    extends: 
      service: kafka 
      file: services.yaml 

    #так как это расширение указанного сервиса, 
    #мы можем доконфигурировать его свойства  
    ports: 
      - "9092:9092" </code></pre><p>В данной ситуации вариант с <code>extends</code> подойдет лучше всего. Используя этот вариант, можно избежать дублирования кода и при этом сохранить гибкость конфигурирования. Кстати, схожий подход использует и JHipster, генератор Spring Boot приложений. Отличный доклад про этот генератор сделал Илья Кучмин на последнем JPoint. Рекомендую его посмотреть, доклад получился очень интересный</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/6706430e87bb36d4b3fac769" data-style="" id="6706430e87bb36d4b3fac769" width=""></div><h2>Решим задачу с extends и include</h2><p>По легенде мы занимаемся доработкой существующего приложения, в котором уже был реализован следующий Docker Compose файл:</p><pre><code class="yaml">services: 
  spring-petclinic: 
    image: spring-petclinic:latest 
    build: 
      context: . 
      args: 
        DOCKER_BUILDKIT: 1 
    restart: always 
    ports: 
      - "8080:8080" 
    environment: 
      POSTGRES_HOST: postgres 
      POSTGRES_DB: spring-petclinic 
      POSTGRES_USER: root 
      POSTGRES_PASSWORD: root 
      KAFKA_BOOTSTRAP_SERVERS: kafka:29092 
    healthcheck: 
      test: wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1 
      interval: 30s 
      timeout: 5s 
      start_period: 30s 
      retries: 5 
    depends_on: 
      - postgres 
  postgres: 
    image: postgres:16.3 
    restart: always 
    ports: 
      - "5432:5432" 
    volumes: 
      - postgres_data:/var/lib/postgresql/data 
    environment: 
      POSTGRES_USER: root 
      POSTGRES_PASSWORD: root 
      POSTGRES_DB: spring-petclinic 
    healthcheck: 
      test: pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB 
      interval: 10s 
      timeout: 5s 
      start_period: 10s 
      retries: 5 
  kafka: 
    image: confluentinc/cp-kafka:7.6.1 
    restart: always 
    ports: 
      - "29092:29092" 
      - "9092:9092" 
    volumes: 
      - kafka_data:/var/lib/kafka/data 
    environment: 
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://localhost:9092 
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT:PLAINTEXT,PLAINTEXT_HOST:PLAINTEXT,CONTROLLER:PLAINTEXT 
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1 
      KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT 
      KAFKA_NODE_ID: 1 
      CLUSTER_ID: 8GyRIS62T8aMSkDJs-AH5Q 
      KAFKA_PROCESS_ROLES: controller,broker
