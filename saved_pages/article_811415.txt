id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-blurred="true" data-src="https://habrastorage.org/getpro/habr/upload_files/ed0/e0e/beb/ed0e0ebeb59a5472c54c7ebaed240e72.jpg" height="1024" который содержал несколько специфичных структур данных, переписанных на Kotlin, таких как <a href="https://developer.android.com/reference/androidx/collection/LongSparseArray" rel="noopener noreferrer nofollow">LongSparseArray</a>, <a href="https://developer.android.com/reference/androidx/collection/SimpleArrayMap" rel="noopener noreferrer nofollow">SimpleArrayMap</a> и<strong> </strong><a href="https://developer.android.com/reference/androidx/collection/SparseArrayCompat" rel="noopener noreferrer nofollow">SparseArrayCompat</a>.</p><p>На тот период Kotlin только начинал набирать обороты в Android разработке и добавление новых более эффективных коллекций, полностью написанных на нём было одним из шагов по внедрению языка.</p><p>С тех пор прошло более 6 лет и в январе текущего года был выпущен новый релиз с мощной заменой <a href="https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html" rel="noopener noreferrer nofollow">HashMap</a>, о которой я расскажу чуть позже:</p><pre><code class="kotlin">dependencies {
    implementation("androidx.collection:collection:1.4.0")
}</code></pre><p>Список релизов и изменений можете глянуть на <a href="https://developer.android.com/jetpack/androidx/releases/collection" rel="noopener noreferrer nofollow">официальном сайте</a>.</p><p>Зачем вообще нужно было строгать новые коллекции и переписывать старые?</p><p>На это есть как минимум три причины:</p><ol><li><p>Эффективный расход памяти - думаю не секрет что даже при наличии 8Gb ОЗУ на вашем телефоне память не бесконечна, поэтому новые коллекции были написаны, придерживаясь принципа "минимум объектов". </p></li><li><p>Эффективная реализация алгоритмов - старые реализации могут содержать не очень эффективные алгоритмы и устаревшие решения, требующие рефакторинга.</p></li><li><p>Kotlin Multiplatform - при написании общего кода на Kotlin под разные платформы требуется минимальное количество зависимостей от платформенных структур данных, например таких как android.util.SparseArray.</p></li></ol><p>А теперь перейдём к самой вкусной части статьи, разберёмся что за магические штуки наколдовали Google кодеры и самое главное как они работают под капотом.</p><p>Полетели!</p><h2>Списки или динамические массивы</h2><p>Было добавлено несколько реализаций:</p><pre><code class="kotlin">// обобщённая реализация
val games = objectListOf("Atomic Heart", "BioShock", "Mafia II")
val movies = objectListOf("Dune: Part Two", "1 + 1")

val moviesAndGames = mutableObjectListOf&lt;String&gt;()
// переопределённые операторы plusAsssign
moviesAndGames += movies
moviesAndGames += games
moviesAndGames += "Far Cry 3"
moviesAndGames += "The Shawshank Redemption"

// изменяемая версия mutableIntListOf()
val fibonacciIntegerNumbers = intListOf(1, 1, 2, 3, 5)

// изменяемая версия mutableLongListOf()
val fibonacciLongNumbers = longListOf(1, 1, 2, 3, 5)

// изменяемая версия mutableFloatListOf()
val fibonacciFloatingNumbers = floatListOf(1f, 1f, 2f, 3f, 5f)

/* P.S. реализации для примитивных типов были созданы под копирку 
на основе обобщённой objectListOf() */</code></pre><p>Под капотом обычный массив, который увеличивается в полтора раза при превышении размера, а при удалении значений из середины происходит сдвиг массива, можно сказать что это аналогия <a href="https://docs.oracle.com/javase/8/docs/api/java/util/ArrayList.html" rel="noopener noreferrer nofollow">ArrayList</a> из Java мира.</p><p>Основные преимущества:</p><ol><li><p>Методы forEach, forEachIndexed и аналогичные реализованы без использования итераторов + они помечены ключевым словом inline для уменьшения загрузки <a href="https://ru.wikipedia.org/wiki/%D0%A1%D1%82%D0%B5%D0%BA_%D0%B2%D1%8B%D0%B7%D0%BE%D0%B2%D0%BE%D0%B2" rel="noopener noreferrer nofollow">стэка вызовов</a>.</p></li><li><p>Есть отдельные реализации для примитивных типов Int, Long и Float без <a href="https://docs.oracle.com/javase/tutorial/java/data/autoboxing.html" rel="noopener noreferrer nofollow">autoboxing / unboxing</a> механизма, который характерен для ArrayList&lt;Integer&gt; например.</p></li><li><p>Перегрузка операторов += и -= для более лаконичного кода, также классы коллекций помечены ключевым словом <a href="https://kotlinlang.org/docs/sealed-classes.html" rel="noopener noreferrer nofollow">sealed</a>, поэтому за пределами androidx.collection библиотеки не удастся создать новых наследников и нарушить четко определённую логику.</p></li></ol><p>Сводка классов, если захотите залезть в исходники:</p><ol><li><p><a href="https://developer.android.com/reference/androidx/collection/ObjectList" rel="noopener noreferrer nofollow">ObjectList</a> / <a href="https://developer.android.com/reference/androidx/collection/MutableObjectList" rel="noopener noreferrer nofollow">MutableObjectList</a> для обобщенных списков</p></li><li><p><a href="https://developer.android.com/reference/androidx/collection/IntList" rel="noopener noreferrer nofollow">IntList</a> / <a href="https://developer.android.com/reference/androidx/collection/MutableIntList" rel="noopener noreferrer nofollow">MutableIntList</a> для примитивного типа Int</p></li><li><p><a href="https://developer.android.com/reference/androidx/collection/FloatList" rel="noopener noreferrer nofollow">FloatList</a> / <a href="https://developer.android.com/reference/androidx/collection/MutableFloatList" rel="noopener noreferrer nofollow">MutableFloatList</a> для примитивного типа Float</p></li><li><p><a href="https://developer.android.com/reference/androidx/collection/LongList" rel="noopener noreferrer nofollow">LongList</a> / <a href="https://developer.android.com/reference/androidx/collection/MutableLongList" rel="noopener noreferrer nofollow">MutableLongList</a> для примитивного типа Long</p></li></ol><h2>Пары значений </h2><p>Библиотека androidx.collection включает три реализации пар значений для примитивных типов, возможно на момент чтения этой статьи что-то изменилось, но базовые идеи реализации редко меняются.</p><p>1. <a href="https://developer.android.com/reference/androidx/collection/IntIntPair" rel="noopener noreferrer nofollow">IntIntPair</a> хранит два значения типа Int в виде одного числа Long благодаря простейшей битовой математики: первое число хранится в старших 32 битах Long, а второе в младших:</p><pre><code class="kotlin">@JvmInline
value class IntIntPair(val packedValue: Long) {
  
    constructor(first: Int, second: Int) : this(packInts(first, second))
    
    // первое значение хранится в старших битах числа Long
    val first: Int
        get() = (packedValue shr 32).toInt()

    // второе значение хранится в младших битах числа Long
    val second: Int
        get() = (packedValue and 0xFFFFFFFF).toInt()
}

inline fun packInts(val1: Int, val2: Int): Long {
    /* чтобы положить два 32-битных числа в одно 64-битное 
    первое число преобразуем к 64-битному и делаем сдвиг влево на 32 бита, 
    второе также расширяем до 64 бит с занулением старших 32 бит, 
    если такие случайно там оказались, после этого совмещаем биты обеих чисел 
    с помощью логического OR */
    return (val1.toLong() shl 32) or (val2.toLong() and 0xFFFFFFFF)
}</code></pre><p>Обратите внимание, что IntIntPair является <a href="https://kotlinlang.org/docs/inline-classes.html" rel="noopener noreferrer nofollow">inline value классом</a>, а это значит что поле <em>packedValue</em> будет встраиваться в место вызова без создания объекта класса, что избавляет <a href="https://ru.wikipedia.org/wiki/%D0%A1%D0%B1%D0%BE%D1%80%D0%BA%D0%B0_%D0%BC%D1%83%D1%81%D0%BE%D1%80%D0%B0" rel="noopener noreferrer nofollow">GC</a> от ненужной работы.</p><p>2. <a href="https://developer.android.com/reference/androidx/collection/FloatFloatPair" rel="noopener noreferrer nofollow">FloatFloatPair</a> аналогично IntIntPair хранит два значения типа Float в одном значении типа Long, отличие заключается только в дополнительной трансформации числа Float в битовое представление типа Int:</p><pre><code class="kotlin">/* сначала извлекается битовое представление типа Int из старших 32 бит числа, 
затем трансформируется во Float тип */
inline val first: Float
    get() = floatFromBits((packedValue shr 32).toInt())

/* алгоритм аналогичен извлечению первого значения, 
только битовое представление извлекается из младших 32 бит */
inline val second: Float
    get() = floatFromBits((packedValue and 0xFFFFFFFF).toInt())

inline fun packFloats(val1: Float, val2: Float): Long {
    /* превращаем каждое значение Float в битовое представление типа Int, 
    для дальнейших битовых операций переводим в Long */
    val v1 = val1.toRawBits().toLong()
    val v2 = val2.toRawBits().toLong()
    // также как в IntIntPair запаковываем два числа типа Int в один Long
    return (v1 shl 32) or (v2 and 0xFFFFFFFF)
}

inline fun floatFromBits(bits: Int): Float = java.lang.Float.intBitsToFloat(bits)
inline fun Float.toRawBits(): Int = java.lang.Float.floatToRawIntBits(this)</code></pre><p>Более детальное описание алгоритма превращения числа Float в битовое представление Int и обратно содержится в документации по методам <a href="https://docs.oracle.com/javase%2F7%2Fdocs%2Fapi%2F%2F/java/lang/Float.html#intBitsToFloat(int)" rel="noopener noreferrer nofollow">Float.intBitsToFloat</a> и <a href="https://docs.oracle.com/javase%2F7%2Fdocs%2Fapi%2F%2F/java/lang/Float.html#floatToRawIntBits(float)" rel="noopener noreferrer nofollow">Float.floatToRawIntBits</a>.</p><p><em>3.</em> <a href="https://developer.android.com/reference/androidx/collection/LongLongPair" rel="noopener noreferrer nofollow">LongLongPair</a> - простейшая реализация для двух чисел типа Long:</p><pre><code class="kotlin">class LongLongPair(val first: Long, val second: Long) { ... }</code></pre><p>Выигрывает перед <a href="https://developer.android.com/reference/android/util/Pair" rel="noopener noreferrer nofollow">Pair</a>&lt;Long, Long&gt; в памяти, так как для всех обобщенных типов используются <a href="https://docs.oracle.com/javase/tutorial/java/data/numberclasses.html" rel="noopener noreferrer nofollow">классы обертки</a>, а не примитивные типы.</p><p>Во всех вышеперечисленных реализациях пар значений есть переопределение операторов component1 и component2 для конструкций<a href="https://kotlinlang.org/docs/destructuring-declarations.html" rel="noopener noreferrer nofollow"> деструктуризации</a>:</p><pre><code class="kotlin">val (firstInt, secondInt) = IntIntPair(3, 7)
val (firstLong, secondLong) = LongLongPair(2021, 2023)
val (firstFloat, secondFloat) = FloatFloatPair(0.33f, 0.45f)</code></pre><h2>Хэш-таблицы</h2><p>Настал момент заглянуть в самую сложную алгоритмическую часть этой статьи, готов представить вам, новая реализация <a href="https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html" rel="noopener noreferrer nofollow">HashMap</a>:</p><pre><code class="kotlin">// обобщённый вариант хэш-таблицы
val scatterMap: ScatterMap&lt;String, String&gt; = mutableScatterMapOf()

// реализации хэш-таблиц для примитивных типов, аналогичные варианты есть для Long и Float
intIntMapOf()
intFloatMapOf()
intLongMapOf()
intObjectMapOf&lt;String&gt;()

// варианты для случаев, когда в качестве ключа один из примитивных типов: Int, Long или Float
intObjectMapOf&lt;String&gt;()
longObjectMapOf&lt;String&gt;()
floatObjectMapOf&lt;String&gt;()

/* P.S. похожие штуки на intIntMapOf(), intLongMapOf() и intObjectMapOf() 
есть в android.util пакете - SparseIntArray, SparseLongArray 
и SparseArray соответственно */</code></pre><p>Полагаю вы уже догадались, что все эти реализации были созданы, также как и ранее рассмотренные списки, под копирку на основе обобщённой версии:</p><pre><code class="kotlin">sealed class ScatterMap&lt;K, V&gt; {
    // меняются буквально только типы для массивов ключей и значений
    var keys: Array&lt;Any?&gt; = EMPTY_OBJECTS
    var values: Array&lt;Any?&gt; = EMPTY_OBJECTS
    // остальная логика остаётся неизменной
}

sealed class IntIntMap {
    // типы были изменены на примитивные
    var keys: IntArray = EmptyIntArray
    var values: IntArray = EmptyIntArray

}

/* P.S. Такой подход может показаться избыточным так как 
одинаковая логика копируется прямо в лоб, на самом деле особого проигрыша 
в этом нет и любые неиспользуемые классы легко выпиливаются 
современными системами сборки. */</code></pre><p>Следовательно мы можем взять реализацию <a href="https://developer.android.com/reference/kotlin/androidx/collection/ScatterMap" rel="noopener noreferrer nofollow">ScatterMap</a> / <a href="https://developer.android.com/reference/kotlin/androidx/collection/MutableScatterMap" rel="noopener noreferrer nofollow">MutableScatterMap</a> и рассмотреть её под капотом, логика остальных ничем не будет отличаться.</p><p>Приступим к изучению!</p><h4>Как ScatterMap хранит информацию о парах ключ / значение</h4><p>Механизм хранения информации о записях ключ / значение отличается от HashMap, где все лежит в одном массив<br/><br/><br/>е Map.Entry объектов, в ScatterMap для этого используется отдельный массив метаданных:</p><pre><code class="kotlin">internal var metadata: LongArray = EmptyGroup</code></pre><p>где единицей хранимой информации является 8 бит, которыми можно сохранить одно состояние ровно для одной записи хэш-таблицы.</p><p>Список возможных состояний и их 8-ми битовая последовательность:</p><ol><li><p>Пустое значение - 10000000</p></li><li><p>Конец таблицы - 11111111, используется при итерации по таблице, чтобы остановиться на последней записи</p></li><li><p>Удалённое значение - 11111110</p></li><li><p>Заполненное значение - хранит младшие 7 бит хэша ключа, вернёмся к этому чуть позже</p></li></ol><p>Думаю не секрет, что Long является 64 битным числом и было бы печально хранить в нём только 8 бит, поэтому была введена концепция группы:</p><p><em>Группа - набор из 8-ми битных состояний или слотов, как это принято в терминологии ScatterMap, в данном случае это число Long, которое может хранить 8 слотов, а это значит что размер группы равен 8.</em></p><p>Есть интересный момент насчет групп в исходном коде ScatterMap:</p><pre><code class="kotlin">// размер группы, который мы уже вычислили
const val GroupWidth = 8

/* при чтении кода сложно было абстрагироваться от числа Long из-за 
битовой арифметики, поэтому этот typealias ни раз меня сбивал с толку */
typealias Group = Long</code></pre><p>При последовательном проходе по массиву метаданных за одну итерацию обычно извлекается одна группа, то есть число типа Long, которое не всегда может быть получено по единому индексу из-за некоторого смещения, о котором речь пойдёт ниже.</p><h4>Метод поиска индекса ключа</h4><p>Для более глубокого понимания как работает ScatterMap рассмотрим основной метод для поиска индекса:</p><pre><code class="kotlin">/* метод возвращает индекс ключа, по которому также можно получить значение, 
так как массивы keys и values сопоставимы по индексам */
inline fun findKeyIndex(key: K): Int {
    // хэш-функция будет рассмотрена ниже
    val hash = hash(key)
    /* получаем младшие 7 бит хэша, которые используются 
    для поиска нужного слота в массиве метаданных */
    val hash2 = h2(hash)

    /* в качестве маски для вычисления смещения будет использоваться 
    размер хэш-таблицы */
     val probeMask = _capacity
    // берём старшие 25 бита хэша и рассчитываем смещение на основе маски
    var probeOffset = h1(hash) and probeMask
    var probeIndex = 0

    while (true) {
        /* получаем группу по смещению, это не совсем тривиальный алгоритм, 
        так как смещение не всегда кратно числу 8, в таком случае часть байтов берётся из одного числа Long, 
        другая из другого, например для массива metadata, который состоит из Long0 и Long1 
        группа по смещению 1 будет содержать 7 старших байтов из Long0 и 1 младший байт из Long1 */
        val g = group(metadata, probeOffset)
        /* ищем в группе нужный слот (состояние записи ключ / значение), 
        ранее я уже приводил 4 возможных состояния и там было указано 
        что в случае заполненного значения слот должен содержать младшие 7 бит хэша ключа */
        var m = g.match(hash2)
        // если нужный слот был найден прыгаем в цикл
        while (m.hasNext()) {
            /* в отличии от HashMap, где в случае совпадения хэшей 
            у нескольких ключей создаётся связанный список, в ScatterMap для этого 
            был введён специальный алгоритм (quadratic probing), суть которого состоит в том, 
            что при совпадении хэшей вычисляется новый индекс умножением текущего на степень двойки, 
            если там уже есть значение вычисление будет продолжаться до тех пор пока не найдется пустое местечко */
            val index = (probeOffset + m.get()) and probeMask
            // реальное совпадение ключа означает равенство двух методов hashCode() и equals(), никогда не забывайте об этом
            if (keys[index] == key) {
                return index
            }
            m = m.next()
        }
        // если такого хэша нет в группе, значит ключ не был добавлен в хэш-таблицу
        if (g.maskEmpty() != 0L) {
            break
        }

        probeIndex += GroupWidth
        probeOffset = (probeOffset + probeIndex) and probeMask
    }

    return -1
}</code></pre><p>Обобщая всё вышесказанное можно сказать, что основной механизм работы ScatterMap основан на хранении состояния (слота) записи ключ / значение в отдельном массиве metadata типа Long, доступ к которому основан на битовой арифметике, а для решения коллизий используется алгоритм <a href="https://en.wikipedia.org/wiki/Quadratic_probing" rel="noopener noreferrer nofollow">quadratic probing</a>, основанный на вычислении следующего свободного индекса в массиве.</p><h4>Хэш-функция и итоги</h4><p>Теперь рассмотрим хэш-функцию:</p><pre><code class="kotlin">inline fun hash(k: Any?): Int {
    // константа MurmurHashC1 была добавлена для сокращения коллизий
    val hash = k.hashCode() * MurmurHashC1
    /* к старшим битам подмешиваются младшие для более 
    эффективной работы алгоритма quadratic probing */
    return hash xor (hash shl 16)
}

// константа была взята из MurmurHash алгоритма: 
// https://en.wikipedia.org/wiki/MurmurHash#Algorithm
const val MurmurHashC1: Int = 0xcc9e2d51.toInt()</code></pre><p>Хочу добавить, если не проговорил это явно, все данные: ключи и значения лежат в двух массивах keys и values без какой-либо вложенности как это было в HashMap, где Map.Entry мог быть узлом связанного списка или дерева. </p><p>Итоговые замечания:</p><ol><li><p>Все остальные операции: вставка и удаление работают практически также, как и метод поиска индекса ключа с небольшой лишь разницей: происходит не только чтение массивов metadata, keys и values но и запись нового значения либо его зануление + при превышении размера хэш-таблицы размеры массивов увеличиваются.</p></li><li><p>ScatterMap не сохраняет порядок добавления новых значений, как это делает <a href="https://docs.oracle.com/javase/8/docs/api/java/util/LinkedHashMap.html" rel="noopener noreferrer nofollow">LinkedHashMap</a> например.</p></li><li><p>forEach, forEachIndexed также как и в предыдущих коллекциях являются inline функциями и работают без создания итераторов.</p></li></ol><h4>Множества</h4><p>Реализация практически такая же как у хэш-таблиц, только вместо двух массивов keys и values под капотом лежит один:</p><pre><code class="kotlin">sealed class ScatterSet&lt;E&gt; {
