оригинала: <span>Nataliia Dziubenko</span></a></div><div class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Новый перевод от команды<a href="https://t.me/+QmrKn0wA8CdkNjAy"> Spring АйО</a> расскажет вам (с примерами кода), как JVM обрабатывает исключения на низком уровне, что такое таблица исключений и какие сценарии используются для вариантов <code>try</code>-<code>catch</code> и <code>try</code>-<code>finally</code>.  </p><hr/><p><strong>То, как JVM обрабатывает инструкции на байткоде, уже само по себе довольно интересно. Но знаете ли вы, что происходит, когда выбрасывается исключение? Как JVM справляется с передачей управления? Как это выглядит в байткоде? </strong></p><p>Сейчас я вам по-быстренькому это покажу! </p><h4>Пример</h4><p>Вот небольшой кусочек кода на Java, в котором присутствуют все важные действующие лица, участвующие в игре в исключения (<code>try</code>-<code>catch</code>-<code>finally</code>):</p><pre><code class="java">int a = 0;
try {
    if (a == 0) { // to make it less boring, let's have some branching logic
        throw new Exception("Exception message!");
    }
} catch (Exception e) {
    doSomethingOnException();
} finally {
    doSomethingFinally();
}</code></pre><h4>Таблица исключений</h4><p>Получившийся байткод включает в себя интересный фрагмент внутри атрибута Code, который называется таблицей исключений. Каждый метод может иметь свою собственную таблицу исключений, но она включается в код только тогда, когда ее присутствие имеет смысл. Если в методе отсутствует логика обработки исключений, то и таблицы исключений тоже не будет. </p><p>В нашем примере таблица исключений выглядит вот так:</p><pre><code>Exception table:
   from    to  target type
       2    16    22   Class java/lang/Exception
       2    16    32   any
      22    26    32   any</code></pre><p>Числа указывают на адреса байткод-инструкций. Каждая строка в таблице показывает диапазон байткод-инструкций (<code>from</code> и <code>to</code>), за которым следит обработчик исключений. Сам обработчик — это тоже всего лишь набор байткод-инструкций, а колонка <code>target</code> указывает на адрес, где начинается код обработки. <code>type</code> просто задает тип исключения, который обрабатывается тем или иным обработчиком. </p><h4>Байткод-инструкции </h4><p>Чтобы точно понять, куда ведут эти адреса, давайте посмотрим также на набор байткод-инструкций для данного метода (объяснение приведено ниже):</p><pre><code class="assembly"> 0: iconst_0
 1: istore_0
 2: iload_0
 3: ifne          16
 6: new           #12          // class java/lang/Exception
 9: dup
10: ldc           #14           // String Exception message!
12: invokespecial #16     // Method java/lang/Exception."&lt;init&gt;":(Ljava/lang/String;)V
15: athrow
16: invokestatic  #19      // Method doSomethingFinally:()V
19: goto          38
22: astore_1
23: invokestatic  #22      // Method doSomethingOnException:()V
26: invokestatic  #19      // Method doSomethingFinally:()V
29: goto          38
32: astore_2
33: invokestatic  #19      // Method doSomethingFinally:()V
36: aload_2
37: athrow
38: return</code></pre><p>Давайте пройдемся по этому коду.</p><p>Инструкции <code>0</code> - <code>2</code> отвечают за создание переменной <code>int a = 0;</code>.</p><p>Дальше идет инструкция <code>ifne</code>, которая представляет собой условный переход, а <code>16</code> — это адрес, на который надо перейти. Если <code>a</code> не равна <code>0</code>, мы переходим на адрес <code>16</code>. Там мы находим содержимое нашего блока <code>finally</code>, за которым идет инструкция <code>goto</code>, ведущая в конец метода.</p><p>Если условие принимает значение <code>false</code>, мы переходим к следующим инструкциям. Инструкции <code>6</code> - <code>12</code> создают экземпляр класса <code>Exception</code>. Заметьте, что ссылка на этот экземпляр дублируется, поэтому к концу этого набора инструкций она появится в стеке операндов дважды.</p><p>Инструкция с адресом <code>15</code>, <code>athrow</code>, выбрасывает само исключение! Она “съедает” одну из ссылок на экземпляр <code>Exception</code> из стека операндов и таким образом узнает, какое именно исключение следует выбросить. По факту, она подготавливает стековый фрейм для обработки исключения (очищая его), оставляя только одну ссылку на исключение сверху.</p><p>Когда JVM встречает инструкцию <code>athrow</code>, она проверяет таблицу исключений метода, чтобы найти подходящее местоположение, с которого следует для продолжить выполнение. </p><h4>Сценарий try-catch-finally </h4><p>Давайте посмотрим на таблицу исключений еще раз, ведь теперь у нас есть больше контекста.</p><pre><code>Exception table:
   from    to  target type
       2    16    22   Class java/lang/Exception
       2    16    32   any
      22    26    32   any</code></pre><p>Мы обнаружили <code>athrow</code> по адресу <code>15</code>. Этот адрес покрывается первыми двумя дорожками таблицы, поскольку попадает в диапазон <code>[2, 16)</code>.</p><p>Зачем нужны два обработчика? Один — это блок <code>catch</code> (<code>target</code> показывает на <code>22</code>, откуда начинается логика <code>catch</code>), а второй — это блок <code>finally</code> (<code>target 32</code> указывает на логику <code>finally</code>). Третья дорожка в таблице закрывает диапазон инструкций, содержащих нашу логику <code>catch</code>, а <code>target</code> указывает на блок <code>finally</code>. Это означает, что если во время выполнения блока <code>catch</code> что-то случится, мы все еще хотим оказаться в блоке <code>finally</code>.</p><p>Когда мы находимся в блоке <code>catch</code> (адрес <code>22</code>), какие-то данные из стека операндов сохраняются в переменной. Помните, у нас была дополнительная ссылка на экземпляр <code>Exception</code> в нашем стеке? Здесь инструкция <code>astore</code> сохраняет ее как переменную. Внутри JVM переменная не имеет имени, но по логике вещей это наша переменная <code>e</code>, которую мы можем использовать в блоке <code>catch</code> для логирования или других действий.</p><h4>Менее удачный сценарий</h4><p>Описанный выше сценарий — самый удачный. JVM поискала по таблице исключений и нашла обработчик, который знал, что делать с <code>Exception</code>. Если бы существовал только обработчик <code>finally</code>, этого было бы недостаточно.</p><p>Как JVM узнает, относится ли найденный обработчик к <code>catch</code> или к <code>finally</code>? Мы видим, что колонка <code>type</code> для обработчика <code>finally</code> содержит тип <code>any</code>, что означает, что он должен быть выполнен в любом случае, независимо от того, было выброшено исключение или нет. Если вместо этого значения задать реальный тип, это будет обработчик для <code>catch</code>. Тип, который ищет JVM, является точным совпадением с исключением или его супертипом.</p><p>Но что если подходящего обработчика нет? JVM завершает текущий метод и передаёт исключение вызывающему методу. Если в стеке вызовов нет подходящего обработчика, поток завершает работу с трассировкой стека.</p><h4>Обобщенный сценарий</h4><p>Механизм <code>try</code>-<code>catch</code>-<code>finally</code> можно обобщить следующим образом:</p><ul><li><p>Инструкция <code>athrow</code> говорит JVM, что выброшено исключение. Какое исключение? То, что находится сверху в стеке операндов.</p></li><li><p>Когда это происходит, JVM проверяет таблицу исключений для текущего метода. Получив адрес инструкции <code>athrow</code>, она может найти адреса подходящих обработчиков (или одного обработчика).</p></li><li><p>Если обнаружен обработчик <code>catch</code>, (обработчик, который имеет правильный тип исключения в колонке <code>type</code>), ссылка на исключение сохраняется из стека в переменную. Затем выполняется логика <code>catch</code>.</p></li><li><p>Если найден обработчик <code>finally</code>, он выполняется.</p></li><li><p>Если не найден подходящий обработчик <code>catch</code>, JVM прерывает выполнение текущего фрейма и ищет обработчик в предыдущем фрейме.</p></li><li><p>JVM проходит через все фреймы, пока не найдет обработчик <code>catch</code>, выполняя все обработчики <code>finally</code>, обнаруженные по пути.</p></li><li><p>Если не найден ни один обработчик, выполнение программы прекращается.</p></li></ul><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/a40/da0/b40/a40da0b40cd0298af40ddda054e2b06d.png" height="769" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/a40/da0/b40/a40da0b40cd0298af40ddda054e2b06d.png" width="1368"/></figure><hr/><figure class="full-width"><img
