Drosopoulou</span></a></div><div class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>В новом переводе от команды <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> рассмотрим, как sealed классы и интерфейсы позволяют строго контролировать иерархию классов, обеспечивая тем самым безопасность и простоту поддержки кода. </p><p>Мы познакомимся с ключевыми особенностями sealed классов и интерфейсов, их влиянием на архитектуру приложений и практическими примерами их использования.</p><hr/><p>Когда в язык Java были введены sealed классы и интерфейсы (термин “sealed” в различных источниках может переводиться как “запечатанные”, “закрытые” или “изолированные”, либо не переводиться вовсе — прим. пер.), это стало существенным шагом в сторону улучшения безопасности типов и предсказуемости кода. Эти возможности языка предоставляют мощный механизм для ограничения иерархий наследования и для гарантии того, что только явно определенные подклассы или имплементации могли расширять или имплементировать заданный класс или интерфейс. Sealed классы и интерфейсы помогают создать более устойчивую, легко поддерживаемую и понятную кодовую базу, ограничивая возможное множество типов, которые могут наследоваться от них.</p><p>В этой статье рассматриваются тонкости sealed классов и интерфейсов, подробно изучаются их синтаксис, примеры применения и преимущества, которые они предлагают для современной разработки на Java.</p><h3>1. Понимание sealed классов</h3><h4>Что такое sealed классы?</h4><p>Представьте себе sealed класс как класс, который сам решает, кто может от него наследоваться. В отличие от обычных классов, которые может расширять любой класс, sealed классы в явном виде перечисляют классы, которым разрешено быть их дочерними классами. Такой контроль дает нам больше уверенности в том, как именно тот или иной класс будет использоваться в нашем коде.</p><h4>Как создать sealed класс</h4><p>Чтобы создать sealed класс, мы используем ключевое слово sealed, за которым следует ключевое слово permits для задания списка разрешенных подклассов:</p><pre><code class="java">sealed class Shape permits Circle, Rectangle, Triangle {
    // ...
}</code></pre><p>В этом примере только Circle, Rectangle, и Triangle могут расширять класс Shape.</p><p><strong>Non-sealed и final классы</strong></p><p>Чтобы прояснить ситуацию, упомянем два других типа классов, относящихся к наследованию:</p><ul><li><p><strong>Non-sealed классы:</strong> это тип класса, используемый по умолчанию, любой класс может расширять их.</p></li><li><p><strong>Final классы:</strong> эти классы вообще нельзя расширять.</p></li></ul><h4>Зачем использовать sealed классы?</h4><p>Sealed классы имеют несколько преимуществ:</p><ul><li><p><strong>Предсказуемость:</strong> когда мы точно знаем, какие классы могут расширять sealed класс, код становится проще понимать и поддерживать.</p></li><li><p><strong>Безопасность типов:</strong> ограничения на возможные подтипы помогает предотвращать неожиданное поведение и ошибки.</p></li><li><p><strong>Паттерны проектирования:</strong> sealed классы могут эффективно использоваться для реализации паттернов проектирования, таких как State (состояние) или Visitor (посетитель).</p></li></ul><h4>Пример: иерархия класса Shape </h4><p>Для иллюстрации сказанного давайте создадим простую иерархию, используя sealed классы:</p><pre><code class="java">sealed class Shape permits Circle, Rectangle, Triangle {
    abstract double area();
}
 
final class Circle extends Shape {
    // ...
}
 
final class Rectangle extends Shape {
    // ...
}
 
final class Triangle extends Shape {
    // ...
}</code></pre><p>В этом примере класс Shape является sealed, и только Circle, Rectangle, и Triangle могут наследовать от него. Это гарантирует нам, что любая соответствующая форме предмета переменная shape, с которой мы можем столкнуться, будет принадлежать только к одному из трех типов.</p><h3>2. Случаи использования sealed классов</h3><h4>Моделирование конечных состояний</h4><p>Одно из наиболее распространенных применений для sealed классов — это представление конечного множества состояний. Например:  </p><ul><li><p><strong>Состояния UI:</strong> загрузка, ошибка, успех, бездействие.</p></li><li><p><strong>Состояния сетевого запроса:</strong> ожидает, в процессе, успех, неудача.</p></li><li><p><strong>Состояния игры:</strong> играет, на паузе, окончена.</p></li></ul><p>Используя sealed класс, мы можем гарантировать, что все возможные состояния определены в явном виде и корректно обрабатываются, предотвращая ошибки в runtime:</p><pre><code class="kotlin">sealed class UIState {
    data class Loading(val isLoading: Boolean) : UIState()
    data class Error(val message: String) : UIState()
    data class Success(val data: Any) : UIState()
    object Idle : UIState()
}</code></pre><details class="spoiler"><summary>Скрытый текст</summary><div class="spoiler__content"><pre><code class="java">import lombok.Getter;
import lombok.RequiredArgsConstructor;

public sealed class UIState permits UIState.Loading, UIState.Error, UIState.Success, UIState.Idle {

    @Getter
    @RequiredArgsConstructor
    public static final class Loading extends UIState {
        private final boolean isLoading;
    }

    @Getter
    @RequiredArgsConstructor
    public static final class Error extends UIState {
        private final String message;
    }

    @Getter
    @RequiredArgsConstructor
    public static final class Success extends UIState {
        private final Object data;
    }

    public static final class Idle extends UIState {
        private Idle() {}

        public static final Idle INSTANCE = new Idle();
    }
}</code></pre><p></p></div></details><h4>Представление алгебраических типов данных (Algebraic Data Types - ADTs)</h4><p>Sealed классы идеальны для моделирования ADTs, которые являются фундаментом функционального программирования. ADTs состоят из конечного набора конструкторов данных, при этом с каждым таким типом ассоциируются специфичные для него данные.</p><pre><code class="kotlin">sealed class Maybe&lt;T&gt; {
    data class Just(val value: T) : Maybe&lt;T&gt;()
    object Nothing : Maybe&lt;Nothing&gt;()
}</code></pre><details class="spoiler"><summary>Скрытый текст</summary><div class="spoiler__content"><pre><code class="java">public sealed class Maybe&lt;T&gt; permits Maybe.Just, Maybe.Nothing {

    @Getter
    @RequiredArgsConstructor
    public static final class Just&lt;T&gt; extends Maybe&lt;T&gt; {
        private final T value;
    }

    public static final class Nothing&lt;T&gt; extends Maybe&lt;T&gt; {
        private Nothing() {}

        private static final Nothing&lt;?&gt; INSTANCE = new Nothing();

        public static &lt;T&gt; Nothing&lt;T&gt; get() {
            return (Nothing&lt;T&gt;) Nothing.INSTANCE;
        }
    }
}</code></pre><p></p></div></details><h4>Реализация паттернов проектирования </h4><p>Sealed классы могут использоваться для реализации паттернов проектирования, таких как “Состояние” (State), в которых состояние объекта может меняться с течением времени.  Различные состояния могут быть представлены в виде подклассов sealed класса.  </p><h4>Расширение сопоставления паттернов </h4><p>Sealed классы идеально работают с сопоставлением паттернов, позволяя вам полностью проверить все возможные случаи:</p><pre><code class="kotlin">fun processUIState(uiState: UIState) {
    when (uiState) {
        is UIState.Loading -&gt; showLoadingIndicator()
        is UIState.Error -&gt; showErrorMessage(uiState.message)
        is UIState.Success -&gt; showData(uiState.data)
        is UIState.Idle -&gt; doNothing()
    }
}</code></pre><details class="spoiler"><summary>Скрытый текст</summary><div class="spoiler__content"><pre><code class="java">public void processUIState(UIState uiState) {
    switch (uiState) {
        case UIState.Loading loading -&gt; showLoadingIndicator();
        case UIState.Error error -&gt; showErrorMessage(error.getMessage());
        case UIState.Success success -&gt; showData(success.getData());
        case UIState.Idle idle -&gt; doNothing();
        default -&gt; throw new IllegalStateException("Unexpected value: " + uiState);
    }
}</code></pre><p></p></div></details><h4>Ограничение наследования</h4><p>Sealed классы предлагают способ контроля за тем, какие классы могут расширять базовый класс, тем самым улучшая поддерживаемость кода и предотвращая появление неожиданных подклассов.  </p><h3>3. Сравнение с традиционным наследованием</h3><p>Классическое наследование обеспечивает неограниченную расширяемость, позволяя любому классу наследовать другой, если это не ограничено явно с помощью ключевого слова final. Такая гибкость дает много возможностей, но при неправильном подходе может привести к неожиданным результатам.</p><p>Sealed классы вводят контролируемую форму наследования. Задавая в явном виде разрешенные подклассы, они обеспечивают более предсказуемый и управляемый подход. В то время как традиционное наследование предлагает нам максимальную гибкость, sealed классы приоритезируют безопасность типов и понятность кода.</p><p><strong>Ключевые различия:</strong></p><ul><li><p><strong>Открытые / закрытые:</strong> традиционное наследование является незамкнутым, в то время как sealed классы закрыты и ограничивают количество потенциальных подклассов.</p></li><li><p><strong>Предсказуемость:</strong> sealed классы предлагают нам больше предсказуемости в том, что касается иерархии классов и поведения. </p></li><li><p><strong>Безопасность типов:</strong> sealed классы могут повысить безопасность типов, ограничивая набор возможных типов во время компиляции. </p></li></ul><p>В целом, sealed классы представляют собой компромисс между полной свободой традиционного наследования и жесткой ограниченностью final классов. Они предоставляют полезный инструмент для создания надежных и предсказуемых иерархий классов.</p><h3>4. Преимущества и недостатки sealed классов </h3><p>Внизу представлена таблица, сравнивающая sealed классы и традиционное наследование, за которой следует объяснение преимуществ и недостатков sealed классов.</p><div><div class="table"><table><tbody><tr><td><p align="center"><strong>Показатель</strong></p></td><td><p align="center"><strong>Традиционное наследование</strong></p></td><td><p align="center"><strong>Sealed классы</strong></p></td></tr><tr><td><p align="center">Расширяемость</p></td><td><p align="center">Незамкнутая</p></td><td><p align="center">Закрытая (ограниченный набор подклассов)</p></td></tr><tr><td><p align="center">Предсказуемость</p></td><td><p align="center">Ниже </p></td><td><p align="center">Выше</p></td></tr><tr><td><p align="center">Безопасность типов</p></td><td><p align="center">Умеренная</p></td><td><p align="center">Выше</p></td></tr><tr><td><p align="center">Поддерживаемость</p></td><td><p align="center">Умеренная</p></td><td><p align="center">Выше</p></td></tr><tr><td><p align="center">Сопоставление паттернов</p></td><td><p align="center">Менее эффективное</p></td><td><p align="center">Более эффективное</p></td></tr><tr><td><p align="center">Обеспечение соблюдения спроектированной архитектуры</p></td><td><p align="center">Ограниченное</p></td><td><p align="center">Более сильное</p></td></tr></tbody></table></div></div><h4>Преимущества sealed классов</h4><ul><li><p><strong>Повышенная предсказуемость кода:</strong> ограничивая набор возможных подклассов, sealed классы делают код более предсказуемым и упрощают его анализ. Разработчики могут с уверенностью предполагать, что инстансы sealed классов могут принадлежать только к специфическому набору типов. </p></li><li><p><strong>Улучшенная безопасность типов:</strong> sealed классы вносят свой вклад в повышение безопасности типов, предотвращая появление подклассов, которые могут нарушать инварианты класса. Это может помочь поймать потенциальные ошибки во время компиляции. </p></li><li><p><strong>Лучшая поддерживаемость:</strong> когда набор возможных подклассов ограничен, код становится более прост в поддержке. Также менее вероятной становится ситуация, когда изменения в базовом классе приводят к нежелательным последствиям.</p></li><li><p><strong>Облегчают сопоставление паттернов:</strong> sealed классы идеально работают с сопоставлением паттернов, позволяя совершать полные проверки и писать более чистый код. </p></li><li><p><strong>Обеспечивают соблюдение решений по спроектированной архитектуре:</strong> sealed классы могут использоваться для обеспечения специфических паттернов проектирования и ограничений, предотвращая неправильное использование иерархии классов.</p></li></ul><h4>Недостатки sealed классов</h4><ul><li><p><strong>Пониженная гибкость</strong>: sealed классы ограничивают способность к расширению класса за пределы предзаданных подклассов, что может оказаться чрезмерно строгим ограничением в некоторых случаях.</p></li><li><p><strong>Повышенная сложность:</strong> в то время как они могут повысить читаемость кода во многих ситуациях, sealed классы вводят дополнительную сложность в язык. </p></li><li><p><strong>Потенциал для чрезмерного использования:</strong> sealed классы должны использоваться разумно. Чрезмерное их использование может привести к появлению слишком закостенелых иерархий классов и сделать код менее адаптивным к будущим изменениям. </p></li></ul><details class="spoiler"><summary>Скрытый текст</summary><div class="spoiler__content"><p>Редакция Spring АйО упоминает тот факт, что sealed классы провоцируют разработчиков на использование instanceof-проверок. Например, у некоторого класса X наследник может A или B. В случае необходимости расширения класса, происходит явное нарушение Open/Closed principle. Данный код является по своей сути бомбой замедленного действия, так как при расширении данный код: </p><pre><code class="java">if (o instanceof A) {
// logic for A
} else {
// logic for B
}</code></pre><p>перестает работать. Митигировать можно использованием конструкции switch-case<span class="habrahidden">, однако факт остается фактом</span>.</p></div></details><p>Если подвести итог, sealed классы предлагают ценный инструмент для улучшения качества кода и упрощения процесса его поддержки во многих сценариях. Однако, важно взвешивать преимущества и потенциальные недостатки, чтобы использовать их должным образом в вашей кодовой базе. </p><details class="spoiler"><summary>Скрытый текст</summary><div class="spoiler__content"><p>Команда Spring АйО отдельно подчеркивает, что до введения sealed классов в Java <a href="https://openjdk.org/jeps/409#Motivation">отсутствовала возможность именно ограничить (не запретить) наследование без того, чтобы делать предка inaccesible</a></p></div></details><h3>5. Best practices и на что обратить внимание</h3><p>Sealed классы предлагают значительные преимущества, но их эффективное использование требует внимательного рассмотрения всех факторов. Далее приведено обобщение ключевых best practices и отдельных моментов, которые могут потребовать дополнительного рассмотрения:</p><div><div class="table"><table><tbody><tr><td><p align="left"><strong>Практика / повод к рассмотрению</strong></p></td><td><p align="left"><strong>Описание</strong></p></td></tr><tr><td><p align="left"><strong>Идентифицируйте подходящие случаи использования</strong></p></td><td><p align="left">Используйте sealed классы, когда у вас есть хорошо определенный конечный набор подклассов, и вы хотите гарантированно обеспечить строгую иерархию. </p></td></tr><tr><td><p align="left"><strong>Ищите баланс между гибкостью и контролем</strong></p></td><td><p align="left">В то время как sealed классы обеспечивают контроль, старайтесь не использовать их чрезмерно, так как это может помешать расширению функционала приложения в будущем.</p></td></tr><tr><td><p align="left"><strong>Подумайте о влиянии на производительность</strong></p></td><td><p align="left">В коде, где критична производительность, проанализируйте потенциальное влияние вызовов виртуальных методов и непроизводительных затрат ресурсов при создании объектов.</p></td></tr><tr><td><p align="left"><strong>Используйте сопоставление шаблонов</strong></p></td><td><p align="left">Комбинируйте sealed классы с сопоставлением паттернов для полных проверок и улучшения читаемости. </p></td></tr><tr><td><p align="left"><strong>Придерживайтесь соглашений по именованию</strong></p></td><td><p align="left">Используйте понятные и описательные имена для sealed классов и их разрешенных подклассов.</p></td></tr><tr><td><p align="left"><strong>Подумайте о версионировании</strong></p></td><td><p align="left">Если вы ожидаете, что в разрешенных подклассах со временем появятся изменения, планируйте стратегии по версионированию или миграции.</p></td></tr><tr><td><p align="left"><strong>Оцените поддержку в рамках инструмента</strong></p></td><td><p align="left">Убедитесь, что ваша IDE и другие инструменты в должной мере поддерживают sealed классы.</p></td></tr><tr><td><p align="left"><strong>Регулярно проверяйте код</strong></p></td><td><p align="left">Периодически перепроверяйте, как вы используете sealed классы, чтобы убедиться в том, что они все еще соответствуют вашим запланированным целям. </p></td></tr></tbody></table></div></div><h4>Дополнительные поводы для рассмотрения</h4><ul><li><p><strong>Избегайте чрезмерного использования подклассов:</strong> в то время как sealed классы позволяют использовать более одного подкласса, наличие чрезмерного их количества может привести к невозможности контролировать иерархию, как планировалось изначально.</p></li><li><p><strong>Подумайте об альтернативах:</strong> в некоторых случаях перечисления (enums) или интерфейсы могут быть более подходящим выбором, чем sealed классы.</p></li><li><p><strong>Ищите баланс между открытостью и закрытостью:</strong> в то время как sealed классы ограничивают расширение, они все еще могут соответствовать принципу открытости-закрытости через разрешение модификаций путем внесения изменений в поведение.</p></li></ul><h3>6. Выводы</h3><p>Sealed классы в Java предлагают мощный механизм для контроля наследования и улучшения надежности кода. Ограничивая набор разрешенных подклассов, они улучшают предсказуемость кода, безопасность типов и поддерживаемость. В то время как они вводят дополнительное усложнение в код, из преимущества часто перевешивают недостатки. Тщательное рассмотрение случаев использования, потенциальные компромиссы и использование  best practices чрезвычайно важно для эффективного использования sealed классов. За счет понимания их сильных сторон и ограничений, разработчики могут принимать взвешенные решения по поводу того, где и как внедрять такие классы в их кодовую базу.</p><figure class="full-width"><img
