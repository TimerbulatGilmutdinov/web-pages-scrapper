Masood</span></a></div><div class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>В наши дни архитектуры <a href="https://developers.redhat.com/blog/category/microservices/">на базе микросервисов</a> стали внедряться практически повсеместно. И нередки ситуации, когда какая-нибудь бизнес-функция может генерировать большое количество сетевого трафика в форме обмена сообщениями между несколькими микросервисами, которые она использует. Если мы сможем сделать способ передачи сообщений более эффективным за счет, например, уменьшения размера сообщений, то мы сможем использовать ту же инфраструктуру для работы с более высокими нагрузками.</p><p><a href="https://developers.google.com/protocol-buffers/">Protobuf</a> (сокращение от «protocol buffers») предоставляет независящие от языка и платформы механизмы сериализации структурированных данных для использования в коммуникационных протоколах, хранилищах данных и т. д. <a href="https://grpc.io/docs/what-is-grpc/faq/">gRPC</a> — это современный фреймворк удаленного вызова процедур («remote procedure call» — RPC) с открытым исходным кодом, который может работать где угодно. Их сочетание позволяет создать эффективный формат сообщений, который автоматически сжимается и обеспечивает первоклассную поддержку сложных структур данных, а также ряд <a href="https://www.bizety.com/2018/11/12/protocol-buffers-vs-json/">других преимуществ (в отличие от JSON)</a>.</p><p>Микросервисные среды требуют большого количества коммуникаций между сервисами, и для этого между сервисами должен быть согласован ряд моментов. Они должны иметь согласованое API для обмена данными, например, POST (или PUT) и GET для отправки и получения сообщений. Они также должны договориться о формате данных (JSON). Клиентам, вызывающим сервис, также необходимо позаботиться о написании кода для удаленных вызовов (фреймворки!). Protobuf и gRPC предоставляют возможность определить схему сообщения (а JSON — нет) и сгенерировать скелет программы для использования gRPC‑сервиса (без фреймворков).</p><p>Хоть JSON — это человекочитаемый формат со вложенной структурой данных, у него есть несколько недостатков, например, отсутствие схемы, объекты могут быть довольно громоздкими, и кое‑где может не хватать комментариев.</p><p>В этой статье я покажу вам, как с помощью gRPC и Protobuf можно создать решения, которые помогут вам обойти эти ограничения. </p><h3>Так что же такое gRPC и Protobuf?</h3><p>gRPC — это современный фреймворк удаленного вызова процедур с открытым исходным кодом, который может работать где угодно. Он обеспечивает прозрачное взаимодействие клиентских и серверных приложений и упрощает создание связанных систем. gRPC в находится <a href="https://www.cncf.io/project/grpc/attachment/grpc-2/">в стадии разработки</a> в <a href="https://www.cncf.io/">CNCF</a>.</p><p>Я рекомендовал бы вам в качестве практики создать сервер потоковой передачи данных в формате JSON по HTTP. Тогда вы поймете, о чем я говорю. <a href="https://grpc.io/docs/what-is-grpc/core-concepts/">Потоковая передача</a> данных встроена в gRPC. О концепциях gRPC можно почитать <a href="https://grpc.io/docs/what-is-grpc/core-concepts/">здесь</a>. Лично мне gRPC чем-то напоминает <a href="http://www.corba.org/">CORBA.</a></p><p>Protobuf — это инструмент для сериализации данных. Protobuf предоставляет возможность определять полностью типизированные схемы для сообщений. Он также позволяет вставлять документацию прямо в само сообщение.</p><p>gRPC использует <a href="https://http2.github.io/">HTTP/2</a> с <a href="https://http2.github.io/faq/#why-just-one-tcp-connection">постоянным соединением</a> и <a href="https://http2.github.io/faq/#why-is-http2-multiplexed">мультиплексированием</a> для повышения производительности по сравнению с сервисами, работающими на REST по HTTP 1.1. Однако у постоянного соединения есть проблемы с прокси <a href="https://en.wikipedia.org/wiki/OSI_model#Layer_4:_Transport_Layer">4-го уровня</a>. Нам нужен прокси, поддерживающий балансировку нагрузки на 7-м уровне. <a href="https://www.envoyproxy.io/docs/envoy/v1.7.0/intro/what_is_envoy">Envoy</a> как раз может проксировать вызовы gRPC с поддержкой балансировки нагрузки на стороне сервере. Envoy также обеспечивает обнаружение сервисов на основе внешнего сервиса, известного как <a href="https://www.envoyproxy.io/docs/envoy/v1.26.2/intro/arch_overview/upstream/service_discovery.html">EDS</a>, и я покажу вам, как использовать и эту функцию.</p><h3>Что мы будем создавать</h3><p>В этой статье я создаю gRPC-сервис <a href="https://kotlinlang.org/">на основе Kotlin</a>. Я буду балансировать нагрузку между несколькими инстансами моего сервиса с помощью прокси Enovy. А также я настрою простой REST-сервис, который обеспечит обнаружение сервиса для Envoy. Базовая архитектура выглядит следующим образом.</p><figure class="full-width"><img alt="Basic architecture" data-src="https://habrastorage.org/getpro/habr/upload_files/d67/e5b/b56/d67e5bb56c3e83f51a4462e49d17c486.png" height="850" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d67/e5b/b56/d67e5bb56c3e83f51a4462e49d17c486.png" width="1024"/><div><figcaption>Basic architecture</figcaption></div></figure><h3>Подготовка компонентов</h3><p>Во-первых, нам нужно определить <a href="https://developers.google.com/protocol-buffers/docs/overview">сообщение Protobuf</a>, которое будет служить контрактом между клиентом и сервером (полный файл см. в <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/proto/events/events.proto">event.proto</a>):</p><pre><code>syntax  = "proto3";
import "google/protobuf/empty.proto";
package event;

option java_package = "com.proto.event";
option java_multiple_files = true;

message Event {
    int32 event_id = 1;
    string event_name = 2;
    repeated string event_hosts = 3;
}

enum EVENT_TYPE {
    UNDECLARED = 0;
    BIRTHDAY = 1;
    MARRIAGE = 2;
}

message CreateEventResponse{
    string success = 1;
}

message AllEventsResponse{
    Event event = 1;
}

service EventsService{
    rpc CreateEvent(Event) returns (CreateEventResponse) {};
    rpc AllEvents(google.protobuf.Empty) returns (stream AllEventsResponse) {};
}</code></pre><p>Это сообщение будет использовано плагином <a href="https://grpc.io/blog/kotlin-gradle-projects">Gradle gRPC</a> для генерации заглушек. Эти заглушки будет использовать код клиента и сервера. Чтобы сгенерировать заглушки, вы можете запустить задачу Gradle generateProto.</p><p>Теперь пришло время написать <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/grpc/server/EventServer.kt">сервер</a>:</p><pre><code>val eventServer = ServerBuilder.forPort(50051)
.addService(EventsServiceImpl()) //серверная реализация
.build()
eventServer.start()
println("Event Server is Running now!")

Runtime.getRuntime().addShutdownHook( Thread{
eventServer.shutdown()
} )

eventServer.awaitTermination()</code></pre><p>После того как мы расправились с основным кодом сервера, мы напишем <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/grpc/server/EventsServiceImpl.kt">бизнес-логику</a>, которая выводит захардкоженное сообщение и возвращает фиксированный ответ.</p><pre><code>override fun createEvent(request: Event?, responseObserver: StreamObserver&lt;CreateEventResponse&gt;?) {
        println("Event Created ")
        responseObserver?.onNext(CreateEventResponse.newBuilder().setSuccess("true").build())
        responseObserver?.onCompleted()
    }</code></pre><p>Далее давайте напишем <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/grpc/client/EventClient.kt">клиент</a>, который будет использовать наш сервис событий:</p><pre><code>fun main(args: Array&lt;String&gt;) {
    var eventsChannel = ManagedChannelBuilder.forAddress("10.0.0.112", 8080)
            .usePlaintext()
            .build()

    var eventServiceStub = EventsServiceGrpc.newBlockingStub(eventsChannel)

    for(i in 1..20) {
        eventServiceStub.createEvent(Event.newBuilder().setEventId(i).setEventName("Event $i").build())
    }

    eventsChannel.shutdown()
}</code></pre><p>Я скопировал код сервера в другой <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/grpc/server/EventServer2.kt">файл</a> и изменил номер порта, чтобы имитировать несколько инстансов нашего сервиса.</p><p>Конфигурация прокси-сервера Envoy состоит из трех частей. Все эти настройки находятся в файле <a href="https://github.com/masoodfaisal/grpc-example/blob/master/envoy-docker/envoy.yaml">envoy.yaml</a>. Убедитесь, что вы изменили IP-адрес EDS-сервиса в соответствии с вашими настройками. Обновить IP-адрес сервиса можно в файле <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/eds/EDSServer.kt">EDSServer.kt</a>.</p><p>Теперь определим фронтенд-сервис. Этот сервис будет принимать запросы от клиентов.</p><pre><code>  listeners:
        - name: envoy_listener
          address:
            socket_address: { address: 0.0.0.0, port_value: 8080 }
          filter_chains:
            - filters:
                - name: envoy.http_connection_manager
                  config:
                    stat_prefix: ingress_http
                    codec_type: AUTO
                    route_config:
                      name: local_route
                      virtual_hosts:
                        - name: local_service
                          domains: ["*"]
                          routes:
                            - match: { prefix: "/" }
                              route: { cluster: grpc_service }
                    http_filters:
                      - name: envoy.router
     </code></pre><p>Определим бекенд-сервис (его имя <code>grpc_service</code> в файле <code>envoy.yaml</code>). Бекенд-сервис будет заниматься балансировкой нагрузки вызовов этой группы серверов. Обратите внимание, что мы не знаем фактическое местоположение бекенд-сервиса. Местоположение бекенд-сервиса (service discovery) предоставляется через EDS-сервис. Чуть-чуть дальше мы поговорим об определении конечной точки EDS.</p><pre><code>        - name: grpc_service
          connect_timeout: 5s
          lb_policy: ROUND_ROBIN
          http2_protocol_options: {}
          type: EDS
          eds_cluster_config:
            eds_config:
              api_config_source:
                api_type: REST
                cluster_names: [eds_cluster]
                refresh_delay: 5s</code></pre><p>По желанию вы можете определить конечную точку EDS. (Вы также можете предоставить фиксированный список серверов). Это еще один сервис, который предоставляет список конечных точек бекенда. Таким образом, Envoy сможет динамически подстраиваться под доступные серверы. Я написал этот EDS-сервис в виде <a href="https://github.com/masoodfaisal/grpc-example/blob/master/src/main/java/com/faisal/eds/EDSServer.kt">простого класса</a>.</p><pre><code>   - name: eds_cluster
          connect_timeout: 5s
          type: STATIC
          hosts: [{ socket_address: { address: 10.0.0.112, port_value: 7070 }}]</code></pre><h3>Запуск проекта</h3><p>Скопируйте <a href="https://github.com/masoodfaisal/grpc-example">проект</a> локально:</p><pre><code>git clone https://github.com/masoodfaisal/grpc-example.git</code></pre><p>Соберите проект с помощью Gradle:</p><pre><code>cd grpc-example
./gradlew generateProto
./gradlew build</code></pre><p>Поднимите EDS-сервер, чтобы обеспечить обнаружение сервисов для прокси-сервера Envoy:</p><pre><code>cd grpc-example
./gradlew -PmainClass=com.faisal.eds.EDSServerKt execute</code></pre><p>Инициализируйте несколько инстансов сервиса:</p><pre><code>cd grpc-example
./gradlew -PmainClass=com.faisal.grpc.server.EventServerKt execute
./gradlew -PmainClass=com.faisal.grpc.server.EventServer2Kt execute</code></pre><p>Запустите прокси Enovy:</p><pre><code>cd envoy-docker
docker build -t envoy:grpclb .
docker run -p 9090:9090 -p 8080:8080 envoy:grpclb</code></pre><p>Мой клиент выполняет вызов в цикле, который демонстрирует, как нагрузка распределяется по кругу.</p><pre><code>./gradlew -PmainClass=com.faisal.grpc.client.EventClientKt execute</code></pre><h3>Заключение</h3><p>gRPC обеспечивает более высокую производительность, меньше шаблонного кода для управления и сильно типизированную схему для ваших микросервисов. Среди других фич gRPC, полезных в мире микросервисов, можно выделить <a href="https://github.com/grpc/proposal/blob/master/A6-client-retries.md">повторные попытки</a>, <a href="https://grpc.io/docs/what-is-grpc/core-concepts/">таймауты</a> и <a href="https://grpc.io/docs/guides/error.html">обработка ошибок</a>. Так же напоследок хочу порекомендовать вам замечательную статья о gRPC, доступную на <a href="https://www.cncf.io/blog/2018/08/31/grpc-on-http-2-engineering-a-robust-high-performance-protocol/">сайте CNCF</a>.</p><p>И пусть ваш следующий сервис будет с gRPC!</p><hr/><p>На рынке труда не многие Kotlin-разработчики могут похвастаться навыкам работы с Kotlin DSL. Однако он по факту стал неотъемлемой частью экосистемы Kotlin. Владение этим инструментом является одним из показателей квалификации разработчика. Приглашаем всех желающих на открытый урок, на котором разложим Kotlin DSL по полочкам:</p><ul><li><p>обсудим, что это такое и когда его уместно применять;</p></li><li><p>из каких элементов он состоит;</p></li><li><p>на практике напишем несложный пример.</p></li></ul><p>Записаться на урок можно бесплатно <a href="https://otus.pw/Xgz8/">на странице курса «Kotlin Backend Developer.
