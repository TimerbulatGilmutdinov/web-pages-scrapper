На связи Дима Котиков, и я все еще люблю разбираться в технологиях, разрабатывать под Android и KMP и пить латте на фундучном молоке :)</p><p>Рассказываю о генерации файлов с boilerplate-кодом с помощью удобного механизма задания File Templates в средах разработки Intellij. File Templates позволяет в пару кликов создавать несколько файлов с каким-либо boilerplate-кодом. Хоть статья приводит примеры создания File Templates для Android/Kotlin Multiplatform, она может быть полезна всем, кто работает в средах разработки от Intellij.</p><h2>Шаблонный код</h2><p>Каждый разработчик в повседневной работе сталкивается с созданием однотипных файлов и шаблонных конструкций, будь то файлы а-ля Controller/Service/Repository в backend-разработке или Fragment-, ViewModel-, State-, SideEffect-, Store-сущности в Android/KMP-разработке. </p><p>Файлы с таким кодом могут иметь схожие префиксы и суффиксы в названии, похожий или одинаковый код внутри них по типу инициализации сущностей, логирования, аналитики, вызовов методов жизненного цикла и тому подобное. </p><p>Каждый раз писать один и тот же код бывает довольно утомительно, неинтересно, затратно по времени, да и фокус внимания на бизнес-задаче или какой-то уникальной логике начинает уменьшаться. </p><p>Для примера приведу код одного шаблонного экрана мобильного приложения:</p><details class="spoiler"><summary>Шаблонный код экрана мобильного приложения</summary><div class="spoiler__content"><pre><code class="kotlin">// file: SomeFeatureScreen

            .launchIn(this)

    val state: SomeFeatureState by viewModel.stateFlow.collectAsStateWithLifecycle()
 
    SomeFeatureStateContent(
        state = state,
        onEvent = { uiEvent -&gt; viewModel.dispatch(uiEvent) }
    )
}

    // some feature-specific logic

     // some feature-specific logic

</code></pre><p></p></div></details><p>Код немного утрирован для понимания того, что происходит в представленном примере, в production-коде некоторые конструкции могут быть вынесены в интерфейсы или абстрактные классы.</p><p>Если мы внимательно посмотрим на код, то легко сможем выделить некоторые конструкции, которые можно назвать шаблонными:</p><ul><li><p>Названия сущностей именуются как "&lt;feature name&gt;" + "&lt;suffix&gt;", где "&lt;feature name&gt;" — специфичное название для фичи или экрана, а "&lt;suffix&gt;" — шаблонный суффикс, обозначающий, что это за сущность: экран, вьюмодель, роутер, модель состояния экрана и тому подобное.</p></li><li><p>У сущностей есть шаблонные вызовы, например в <code>SomeFeatureScreen</code> вызываются подписки на состояние (<code>viewModel.stateFlow.collectAsStateWithLifecycle()</code>) и side effect-ы (код внутри <code>LaunchedEffect</code>). В классе <code>SomeFeatureViewModel</code> мы видим стандартный подход к хранению сущности состояния (<code>StateFlow&lt;SomeFeatureState&gt;</code>) и его смены (функция <code>reduceState()</code>), а также хранение и отправка <code>sideEffect</code>-ов (one-time-events).</p></li></ul><p>Если бы приложения состояли только из одного экрана и нам нужно было единожды написать шаблонный код, то особых проблем не возникло бы. На деле даже в небольших мобильных приложениях количество фич и экранов может достигать 50—100 штук. Представим, сколько времени суммарно может уйти, чтобы вручную 100 раз создать по семь файлов с шаблонным кодом, не говоря уже о том, что на каждом экране нужно прописать бизнес-логику, верстку, взаимодействие с сетью, локальным хранилищем, навигацию и так далее.</p><figure class=""><img data-src="https://habrastorage.org/getpro/habr/upload_files/d59/8fe/8a2/d598fe8a284c92842739bc9fc63716f3.png" height="484" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d59/8fe/8a2/d598fe8a284c92842739bc9fc63716f3.png" width="356"/></figure><h2>Как уменьшить боль от boilerplate-кода</h2><p>Вариантов для решения проблемы несколько:</p><p><br/><strong>Сделать отдельную папку,</strong> в которой можно создать файлы с кодом, а потом копипастить в <code>package</code>-ы. Наверное, это самый плохой способ решения проблемы, потому что каждый раз придется вручную переименовывать каждый файл, класс и так далее. Ускорение по времени от такого подхода сомнительное.</p><p><strong>Написать annotation- или symbol-processor</strong> для генерации файлов или классов на базе аннотаций. Инструмент мощный, но не особо подходит для решения нашей проблемы. Все равно нужно руками создать как минимум один файл, написать какой-то базовый код, расставить аннотации, запустить компиляцию и только после этого пользоваться сгенерированными файлами и кодом в них. </p><p>А еще нужно помнить, что в сгенерированных файлах не получится дописать свой код, так как при следующей компиляции файлы запросто могут быть переписаны процессором аннотаций. Добавим к этому тот факт, что написание таких процессоров — штука нетривиальная и требует специфичного API, рефлексии и т. д. В общем, путь, который подходит скорее для создания библиотек и фреймворков, но не для решения нашей проблемы.</p><p><strong>Написать плагин для Intellij</strong> — вариант рабочий, но потребует погружения <a href="https://plugins.jetbrains.com/docs/intellij/plugins-quick-start.html"><u>в документацию по написанию плагинов</u></a> и, по сути, тоже достаточно сложный процесс для решения нашей задачи.</p><p><strong>Использовать встроенный в Intellij-среду механизм Live Templates</strong> — частично решает проблему с генерацией шаблонного кода, но не решает задачу создания и именования семи файлов.</p><p><strong>Использовать встроенный в Intellij-среду механизм File Templates </strong>— наш выбор, так как умеет создавать файлы по шаблонам, причем сразу несколько. Плюс ко всему есть возможность для написания шаблонного кода по различным условиям.</p><p>А если вы знаете другие способы работы с бойлерплейтом — пишите в комментариях 😊</p><p>В моих вариантах нет обобщения в базовые классы и интерфейсы. Этими механизмами можно и нужно пользоваться, но мы приняли за аксиому, что от boilerplate нельзя уйти. Мы все равно с ним столкнемся, но уже с использованием абстракций и обобщений.</p><h2>Плюсы, возможности и синтаксис File Templates </h2><p>Есть несколько весомых причин выбрать <a href="https://www.jetbrains.com/help/idea/using-file-and-code-templates.html#syntax"><u>File Templates</u></a>:</p><ul><li><p>инструмент решает нашу проблему — генерирует файлы и контент в них по шаблонам;</p></li><li><p>генерирует несколько файлов в одно действие;</p></li><li><p>задает пользовательские переменные, которые можно использовать при генерации шаблонного кода;</p></li><li><p>для написания шаблонов не нужно глубоких и специфических знаний — достаточно изучить несколько базовых конструкций, чтобы решать большинство проблем с созданием шаблонов;</p></li><li><p>инструмент интегрирован в Intellij-подобные среды разработки, поэтому дополнительных действий, связанных с установкой и первоначальной настройкой (как это нужно для плагинов), не требуется.</p></li></ul><p>Примеры покажу в Android Studio, но практически вся информация валидна и для остальных сред разработки на базе Intellij IDEA.</p><p>Чтобы добраться до инструмента File Templates, нужно открыть Settings → Editor → File and Code Templates:</p><figure class="full-width"><img alt="Путь до меню File Templates" data-src="https://habrastorage.org/getpro/habr/upload_files/ce6/e24/e37/ce6e24e378d4dd321ad620220c524a81.png" height="1096" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/ce6/e24/e37/ce6e24e378d4dd321ad620220c524a81.png" title="Путь до меню File Templates" width="1600"/><div><figcaption>Путь до меню File Templates</figcaption></div></figure><p>В меню много заготовленных шаблонов, но большинство из них имеют пару строчек с базовыми конструкциями, поэтому будем писать свои.</p><p>Нас интересуют вкладки <code>Files</code> и <code>Includes</code>. На вкладке <code>Files</code> располагаются коллекции с шаблонами, а во вкладке <code>Includes</code> — шаблоны, предназначенные для переиспользования в шаблонах из вкладки Files.</p><figure class="full-width"><img alt="Шаблоны переиспользуем в шаблонах, осталось прикрутить мониторы" data-src="https://habrastorage.org/getpro/habr/upload_files/329/b91/8f4/329b918f436e2612ccb676874c2f0325.png" height="215" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/329/b91/8f4/329b918f436e2612ccb676874c2f0325.png" title="Шаблоны переиспользуем в шаблонах, осталось прикрутить мониторы" width="595"/><div><figcaption>Шаблоны переиспользуем в шаблонах, осталось прикрутить мониторы</figcaption></div></figure><p>Весь шаблонный код, который мы будем писать, по сути своей будет миксом из кода языка и из вспомогательного кода, состоящего из специальных <a href="https://www.jetbrains.com/help/idea/file-template-variables.html"><u>переменных для File Templates</u></a> и конструкций, написанных на <a href="https://velocity.apache.org/"><u>Apache Velocity</u></a> — языке, используемом для шаблонов.</p><p>Из стандартных переменных чаще всего приходится использовать <abbr class="habraabbr" data-abbr="${NAME}" data-title="&lt;p&gt;имя, задаваемое при создании шаблона&lt;/p&gt;&lt;p&gt;&lt;/p&gt;" title="имя, задаваемое при создании шаблона">${NAME}</abbr> и <abbr class="habraabbr" data-abbr="${PACKAGE_NAME}" data-title="&lt;p&gt;путь до файла&lt;/p&gt;&lt;p&gt;&lt;/p&gt;" title="путь до файла">${PACKAGE_NAME}</abbr>. Можно создавать свои переменные, для этого их достаточно просто объявить в месте использования в шаблоне и тогда при создании шаблона появится отдельное поле для ввода кастомной переменной.</p><p>Посмотрим синтаксис Apache Velocity, так как на нем пишется логика для шаблонов.</p><p>Первое, что можно увидеть в стандартных шаблонах, — указание package name для файлов через конструкцию <code>if</code>:</p><pre><code class="java">#if (${PACKAGE_NAME} &amp;&amp; ${PACKAGE_NAME} != "")
    package ${PACKAGE_NAME} 
#end</code></pre><p>Есть возможность писать конструкции <code>if-else</code>:</p><pre><code class="java">#if (${NAME} != "")
    class ${PACKAGE_NAME} 
#else 
    class Unknown 
#end</code></pre><p>Или <code>if-elseif-else</code>:</p><pre><code class="java">#if (${NAME} == "")
    class Unknown 
#elseif (${NAME} == "a")
    class OneLetter 
#else 
    class ${NAME} 
#end</code></pre><p>Все ключевые слова начинаются со знака <code>#</code>, ссылки на переменные с <code>$</code>. <a href="https://velocity.apache.org/engine/devel/user-guide.html#:~:text=The%20following%20rule%20of%20thumb%20may%20be%20useful%20to%20better%20understand%20how%20Velocity%20works%3A%20References%20begin%20with%20%24%20and%20are%20used%20to%20get%20something.%20Directives%20begin%20with%20%23%20and%20are%20used%20to%20do%20something."><u>Как указано в документации:</u></a> ссылки начинаются со знака <code>$</code> и используются для получения чего-либо. Директивы начинаются со знака <code>#</code> и используются, чтобы сделать что-либо (присвоить значение, обработать что-либо и т. п.).</p><p>Присвоение переменной и ее последующее применение выполняется так:</p><pre><code class="java">#set( $someVar = "Hello, World!" ) $someVar</code></pre><p>Если мы хотим, чтобы переменная была задана путем пользовательского ввода, ее нужно сразу указывать в месте использования без <code>set</code>-конструкции:</p><pre><code class="java">$USER_INPUT  // or
${USER_INPUT}</code></pre><p>А еще нам доступны циклы:</p><pre><code class="java">#set( $customerParts = ["John", "Doe"] )
#set( $customerFull = "" )
#set( $part = "" )
 
#foreach( $part in $customerParts )
    #set( $customerFull = $customerFull + $part )
#end

$customerFull  // prints `JohnDoe`</code></pre><p>Ключевые слова <code>#break</code> позволяют прервать цикл <code>#foreach</code>, а <code>#stop</code> прекращает выполнение кода в template-файле.</p><p>Для переиспользования скриптов между файлами шаблонов нужно задать шаблон во вкладке <code>Includes</code>, а потом в целевых файлах вызвать с помощью конструкции <code>#parse</code> — простейший пример есть в шаблонах для проставления хедера файла:</p><pre><code class="java">#parse("File Header.java")</code></pre><p>Аналогично <code>#parse</code> есть еще конструкции <code>#include</code>:</p><pre><code class="java">#include("SharedData.java")</code></pre><p>Ключевые отличия <code>#parse</code> от <code>#include</code>:</p><ul><li><p><code>#parse</code> допускает, что в передаваемом в параметры файле может содержаться VTL-код, который будет выполнен в месте вызова. <code>#include</code> же просто добавит контент из передаваемого в параметрах файла прямиком как есть в место вызова, без обработки VTL-кода.</p></li><li><p><code>#parse</code> может принимать в себя только один аргумент, а <code>#include</code> — несколько:</p></li></ul><pre><code class="java">#parse("File1.java", "File2.java") // ERROR, 2 parameters passed. Must be 1 parameter
#include("File1.java", "File2.java") // CORRECT, more than 1 parameters available</code></pre><p>Еще есть директива <code>#define</code>, которая позволяет задать какой-то шаблон с переменными, которые должны вычислиться позже, к примеру:</p><pre><code class="java">#define( $block )
    Hello $who
#end
#set( $who = 'World!' )

$block // "Hello World!" will be print</code></pre><p>Есть возможность задать переиспользуемые части с помощью конструкции <code>#macro:</code></p><pre><code class="java">#macro( cell )
&lt;tr&gt;&lt;td&gt;&lt;/td&gt;&lt;/tr&gt;
#end</code></pre><p>Заданную в <code>#macro</code> конструкцию дальше в шаблоне можно вызвать так:</p><pre><code class="java">#cell // "&lt;tr&gt;&lt;td&gt;&lt;/td&gt;&lt;/tr&gt;" will be print
#cell // "&lt;tr&gt;&lt;td&gt;&lt;/td&gt;&lt;/tr&gt;" will be print</code></pre><p>В VTL-коде нам доступны функции из Java-мира. Особенно полезны функции работы со строками, например:</p><pre><code class="java">$NAME.replaceAll("([a-z])([A-Z])", "$1_$2").toLowerCase() // replaces "SomeString" camel-case string to "some_string" snake-case string</code></pre><p>Указанных конструкций в большинстве случаев должно хватить при написании шаблонов. Больше информации можно найти в документации по ссылкам: <a href="https://velocity.apache.org/engine/devel/user-guide.html"><u>User Guide</u></a>, <a href="https://velocity.apache.org/engine/devel/vtl-reference.html"><u>VTL Reference</u></a>, <a href="https://velocity.apache.org/engine/devel/glossary.html"><u>Glossary.</u></a></p><h2>Итоги первой части</h2><p>В этой части мы выделили проблему с необходимостью написания шаблонного кода и рассмотрели возможные варианты уменьшения боли. А еще познакомились со встроенным инструментом File Templates и основным синтаксисом.<br/><br/><a href="https://habr.com/ru/companies/tbank/articles/873234/">В следующей статье</a> рассмотрим примеры использования File Templates для решения типовых сценариев, где требуется шаблонный код.
