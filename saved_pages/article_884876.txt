lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Перед вами ещё один гайд о том, как написать свой telegram-бот, который взаимодействует с нейросетью. Мы напишем его таким образом, чтобы с нейросетью можно было вести диалог, т.е. рассмотрим, как сохранять контекст между сообщениями. Но таких гайдов, особенно для Python, уже написано немало. Поэтому напишем его на новом фреймворке <strong>Spring AI</strong> из эксосистемы Spring. А чтобы совсем было интересно - писать будем на <strong>Kotlin</strong>)</p><figure class=""><img alt="Spring AI" data-src="https://habrastorage.org/getpro/habr//post_images/1a5/747/6f6/1a57476f6bf1c270f818bf98b0f8e5df.png" src="https://habrastorage.org/r/w1560/getpro/habr//post_images/1a5/747/6f6/1a57476f6bf1c270f818bf98b0f8e5df.png"/><div><figcaption>Spring AI</figcaption></div></figure><p><strong>Spring AI</strong> — это один из фреймворков экосистемы Spring. Его цель — обеспечить при работе с нейросетями принципы проектирования экосистемы Spring, такие как переносимость и модульная конструкция. В Spring AI есть различные провайдеры к популярным нейросетям: DeepSeek, Google Vertex, Groq, Ollama и ChatGPT. Работу с последним мы и рассмотрим в данной статье. Однако надо заметить, что многие нейросети поддерживают унифицированный API, поэтому переход с одной нейросети на другую (например, на DeepSeek) благодаря Spring AI ограничится изменением пары параметров в конфиге.</p><h3>Делаем заготовку проекта</h3><p>Итак, зайдём на <a href="https://start.spring.io" rel="noopener noreferrer nofollow">start.spring.io</a> и создадим заготовку проекта. В качестве языка выбираем <strong>Kotlin</strong>, тип проекта - <strong>Gradle Kotlin</strong>. Версию Java можно выбрать 21 как последнюю long-term support на текущий момент.</p><figure class=""><img alt="Создаём заготовку проекта" data-blurred="true" data-src="https://habrastorage.org/getpro/habr//post_images/dd7/514/8a5/dd75148a50ead507f7a173aba0f9c137.jpg" src="https://habrastorage.org/r/w780q1/getpro/habr//post_images/dd7/514/8a5/dd75148a50ead507f7a173aba0f9c137.jpg"/><div><figcaption>Создаём заготовку проекта</figcaption></div></figure><p>Далее переходим к зависимостям. Нужный нам стартер называется <strong>spring-ai-openai-spring-boot-starter</strong>. Он позволяет настраивать подключение к ChatGPT в декларативном стиле. На момент написания статьи он ещё не имеет релизной версии, но скорее всего она появится в ближайшее время.</p><p>Также хочу отметить, что вместо spring-ai-openai-spring-boot-starter вы можете выполнять запросы к нейросетям с помощью обычного <strong>RestClient</strong> (если у вас в зависимостях есть <strong>spring-boot-starter-web</strong>) или <strong>WebClient</strong> (для <strong>spring-boot-starter-webflux</strong>). Просто кода вы напишете чуть больше.</p><p>В итоге в секции <strong>dependencies</strong> в файле <strong>build.gradle.kts</strong> у вас должны быть следующие зависимости:</p><pre><code class="kotlin">implementation("org.springframework.ai:spring-ai-openai-spring-boot-starter")
implementation("org.springframework.boot:spring-boot-starter-web")

implementation("org.telegram:telegrambots-spring-boot-starter:6.9.7.1")
implementation("org.telegram:telegrambotsextensions:6.9.0")
</code></pre><p>Для подключения к ChatGPT нам потребуется <strong>Api-Key</strong>, который можно сгенерировать в личном кабинете <a href="https://platform.openai.com/api-keys" rel="noopener noreferrer nofollow">https://platform.openai.com/api-keys</a>. Однако в настоящее время для работы с ChatGPT по API нужно каким-то образом пополнить лицевой счёт и подключить VPN. Последнее, кстати, не обязательно, если вы будете разворачивать ваше приложение у хостера с автоматической поддержкой проксирования запросов. Например, <a href="https://dockhost.ru/?utm_source=habr&amp;utm_medium=organic&amp;utm_campaign=devmark_1" rel="noopener noreferrer nofollow">dockhost.ru</a>.</p><p>Теперь давайте зарегистрируем нового telegram-бота с помощью <a href="https://t.me/botfather" rel="noopener noreferrer nofollow">BotFather</a>. От него нам надо получить botName (который без постфикса "_bot") и токен.</p><p>Все эти параметры надо прописать в настройках приложения в файле <strong>application.yml</strong>. С точки зрения безопасности все чувствительные данные будем подставлять через переменные окружения.</p><pre><code class="yaml">spring:
  ai:
    openai:
      api-key: ${CHATGPT_API_KEY}
      base-url: ${CHATGPT_BASE_URL:https://api.openai.com}

telegram:
  token: ${TELEGRAM_BOT_TOKEN}
  botName: ${TELEGRAM_BOT_NAME}
</code></pre><p>Также, если вы подключаетесь через какой-либо прокси-сервис ChatGPT, надо ещё прописать целевой хост в переменной <strong>CHATGPT_BASE_URL</strong>. Если же подключаетесь напрямую - этот параметр можно вообще не указывать и будет использовано значение по умолчанию.</p><h3>Сервис для взаимодействия с ChatGPT</h3><p>Прежде чем написать сервис, следует отметить пару моментов по работе с нейросетью.</p><p>В ChatGPT (да и во многих других нейросетях) все сообщения могут быть трёх типов:</p><ul><li><p><strong>system</strong> - как правило, первое сообщение, в котором нейросети можно указать предметную область, чтобы повысить качество ответов.</p></li><li><p><strong>user</strong> - сообщения, которые пишет пользователь.</p></li><li><p><strong>assistant</strong> - ответы, которые даёт нейросеть.</p></li></ul><p>spring-ai-openai-spring-boot-starter представляет эти сообщения в виде классов <strong>SystemMessage</strong>, <strong>UserMessage</strong> и <strong>AssistantMessage</strong> с общим интерфейсом <strong>Message</strong>.</p><p>Второй момент заключается в том, что если нейросети отправлять по одному сообщению, то свой ответ она будет формировать только на основании этого сообщения. Так диалог выстроить не удастся. Тогда как ChatGPT помнит весь контекст беседы, когда мы общаемся с ним через веб-интерфейс? Ответ прост. На самом деле достаточно в каждом запросе отправлять всю переписку - тогда контекст будет сохраняться.</p><p>Теперь создадим обычный спринговый сервис, который будет обращаться к ChatGPT. В конструктор ему мы будем подставлять <strong>OpenAiChatModel</strong> - этот бин как раз предназначен для отправки запросов в нейросеть.</p><pre><code class="kotlin">@Service
class OpenAiService(
    private val chatModel: OpenAiChatModel,
) {
    
}
</code></pre><p>Теперь добавим метод <strong>sendMessages()</strong>, который будет отправлять всю историю нашей переписки в виде списка объектов <strong>Message</strong>.</p><pre><code class="kotlin">fun sendMessages(messages: List&lt;Message&gt;): List&lt;String&gt; {
    // ...
}
</code></pre><p>Реализация этого метода довольно проста. Вызываем метод <strong>call()</strong> у бина <strong>chatModel</strong>. В параметрах передаём ему объект <strong>Prompt</strong>, в который подставляем список наших сообщений и с помощью <strong>OpenAiChatOptions.builder()</strong> указываем ряд параметров запроса:</p><pre><code class="kotlin">chatModel.call(
        Prompt(
            messages,
            OpenAiChatOptions.builder()
                .model(OpenAiApi.ChatModel.GPT_4_O)
                .temperature(0.0)
                .responseFormat(ResponseFormat.builder().type(ResponseFormat.Type.TEXT).build())
                .build()
        )
    )
</code></pre><p>В параметрах мы указываем целевую модель. Разные модели тарифицируются по-разному: более дорогие модели выдают более качественный результат и лучше удерживают общий контекст беседы. В этом примере я использую <strong>GPT 4.0</strong>.</p><p>Второй параметр запроса - "температура", которая имеет тип <strong>Double</strong> и может принимать значения от 0.0 до 2.0. Если вы хотите, чтобы нейросеть выдавала более креативные ответы - выставляйте температуру побольше. Это может иметь смысл если вы попросите ChatGPT придумать, например, название для нового проекта. Если же хотите получить максимально точный ответ, соответствующий правилам предметной области - ставьте температуру в 0. Это полезно для математических вычислений (хотя точность и не гарантирована на 100%).</p><p>Наконец, третий параметр запроса позволяет указать формат ответа. Здесь я указываю формат "простой текст", но можно также попросить выдавать ответы в json.</p><p>После вызова метода call() мы получим список объектов типа <strong>Generation</strong>. Каждый из них содержит поле text - это и есть текстовый ответ нейросети. По умолчанию список будет содержать всегда 1 элемент.</p><pre><code class="kotlin">return chatModel.call(
    // Prompt
)
    .results
    ?.map {
        it.output.text
    }
    ?: emptyList()
</code></pre><p>Если по каким-то причинам результат мы не получили, то, благодаря элвис-оператору котлина, возвращаем пустой список.</p><p>На этом разработка клиентской части ChatGPT завершена.</p><h3>Компонент для сохранения контекста беседы</h3><p>Нам нужно где-то сохранять все сообщения текущего диалога, причём отдельно для каждого пользователя telegram.</p><p>В идеале нужно создать две таблицы в БД:</p><ul><li><p><strong>dialog</strong>, который будет связан с <strong>chatId</strong> (идентификатор пользователя в Telegram, имеет тип <strong>Long</strong>)</p></li><li><p><strong>message</strong>, каждая запись в которой будет связана с dialog отношением "один-ко-многим" через поле dialog_id.</p></li></ul><p>Но взаимодействие с БД выходит за рамки данного гайда, поэтому не будем усложнять и обойдёмся без неё.</p><p>Вместо этого мы воспользуемся встроенным в Spring механизмом кеширования. В кеше будем накапливать все сообщения в разрезе текущего chatId, а при старте нового диалога будем сбрасывать этот кеш.</p><p>Реализация компонента <strong>MessageCache</strong> довольно тривиальна. Она содержит три метода для чтения, обновления и сброса кеша соответственно:</p><pre><code class="kotlin">@Component
class MessageCache {

    @Cacheable(cacheNames = ["messages"], key = "#chatId")
    fun getOrInitMessages(chatId: Long): List&lt;Message&gt; = emptyList()

    @CachePut(cacheNames = ["messages"], key = "#chatId")
    fun saveMessages(chatId: Long, messages: List&lt;Message&gt;): List&lt;Message&gt; = messages

    @CacheEvict(cacheNames = ["messages"], key = "#chatId")
    fun clearMessages(chatId: Long): List&lt;Message&gt; = emptyList()
}
</code></pre><p>Аннотация <code>@Cacheable</code> содержит параметры <strong>cacheNames</strong> (имя кеша) и <strong>key</strong> (ключ, по которому извлекаются данные из кеша). У всех трёх методов одинаковые имя кеша и ключа (chatId).</p><p>Метод <strong>getOrInitMessages()</strong> вызывается в том случае, если в кеше ещё нет сущности с указанным chatId. Это происходит при начале нового диалога, поэтому возвращаем пустой список. Последующие вызовы этого же метода будут возвращать данные из кеша по указанному ключу и реальный вызов метода выполняться не будет.</p><p>Аннотация <code>@CachePut</code> обновляет содержимое кеша. Мы будем передавать туда каждый раз полный актуальный набор сообщений из диалога. И эти же сообщения возвращаем.</p><p>Таким образом, поочерёдный вызов методов с аннотациями <code>@Cacheable</code> и <code>@CachePut</code> обеспечивает нам накопление сообщений для пользователя.</p><p>Аннотация <code>@CacheEvict</code> удаляет все сообщения с указанным ключом. Метод с такой аннотацией будем вызывать при начале новой беседы с нейросетью.</p><h3>Команда начала диалога</h3><p>В telegram можно отправлять чат-боту специальные команды. Это фиксированная строка, которая начинается со слеша. Любой telegram-бот должен поддерживать команду "/start", т.к. именно эта команда автоматически выполняется при первом обращении к боту. Однако никто не мешает вызывать её повторно любое количество раз.</p><p>В нашем чат-боте эта команда будет начинать новый диалог. То есть когда захотите поменять тему - просто отправьте чат-боту команду "/start", чтобы он "забыл" всё, о чём вы говорили до этого.</p><p>Расширение <strong>telegrambotsextensions</strong> добавляет к стандартному компоненту возможность добавлять команды независимо друг от друга в виде отдельных бинов. Подтянем в нашу команду бин <strong>MessageCache</strong> и будем сбрасывать историю сообщений.</p><pre><code class="kotlin">@Component
class StartCommand(
    private val messageCache: MessageCache,
) : BotCommand("/start", "") {

    override fun execute(absSender: AbsSender, user: User, chat: Chat, arguments: Array&lt;out String&gt;) {
        messageCache.clearMessages(chat.id)
        absSender.execute(createMessage(chat.id.toString(), "Начинаем диалог!"))
    }
}
</code></pre><p>При выполнении этой команды telegram-бот отправляет пользователю сообщение "Начинаем диалог!" с помощью <strong>absSender.execute()</strong>. Создание сообщения происходит через утилитарный метод <strong>createMessage()</strong>:</p><pre><code class="kotlin">fun createMessage(chatId: String, text: String) =
    SendMessage(chatId, text)
        .apply { enableMarkdown(true) }
        .apply { disableWebPagePreview() }
</code></pre><p>Здесь создаём стандартный объект <strong>SendMessage</strong> из <strong>telegrambots-spring-boot-starter</strong> и передаём ему chatId и текст сообщения. Также активируем формат сообщений markdown и отключаем превью гиперссылок (это по желанию, чтобы не загромождать переписку).</p><h3>Реализация telegram-бота</h3><p>Взаимодействие с серверами telegram возможно в двух режимах: <strong>webhook</strong> и <strong>long-polling</strong>.</p><p>В режиме <strong>webhook</strong> вы разворачиваете полноценное веб-приложение, к которому будет обращаться сам telegram. Однако обязательным требованием является наличие публичного домена и сертификата https.</p><p>В случае с <strong>long-polling</strong> общедоступная точка доступа не требуется, т.к. приложение само делает запрос к серверу telegram и ждёт ответа до тех пор, пока не произойдёт какое-то событие, связанное с ботом (например, получение сообщения от пользователя). В таком режим вы можете поднимать telegram-бота откуда угодно, в том числе локально на вашем устройстве в целях отладки.</p><p>С точки зрения простоты мы будем использовать long-polling. Для этого создадим новый сервис и унаследуем его от класса <strong>TelegramLongPollingCommandBot</strong>. Также в этот сервис подтянем созданные ранее бины OpenAiService, MessageCache и все команды в виде множества Set (у нас там будет только StartCommand). Токен и имя бота подгрузим из конфига с помощью аннотации <code>@Value</code>.</p><pre><code class="kotlin">@Component
class TelegramBot(
    private val openAiService: OpenAiService,
    private val messageCache: MessageCache,
    commands: Set&lt;BotCommand&gt;,
    @Value("\${telegram.token}")
    token: String,
) : TelegramLongPollingCommandBot(token) {

    @Value("\${telegram.botName}")
    private val botName: String = ""

    init {
        registerAll(*commands.toTypedArray())
    }

    override fun getBotUsername(): String = botName
}
</code></pre><p>В секции init мы регистрируем все подгруженные команды бота с помощью метода <strong>registerAll()</strong>.</p><p>Ну а теперь осталось определить ключевой метод <strong>processNonCommandUpdate()</strong> нашего telegram-бота. Он обрабатывает все сообщения, которые "не-команды", т.е. не начинаются со слеша. Метод получает объект <strong>Update</strong>. Для него мы сначала проверяем, содержит ли он сообщение. Если да - можем извлечь из него chatId отправителя. Далее проверяем, является ли сообщение текстом (а не картинкой, стикером, файлом и т.д.). И после этого мы получаем доступ к тексту сообщения, которое отправил пользователь через telegram-бот.</p><pre><code class="kotlin">override fun processNonCommandUpdate(update: Update) {
    if (update.hasMessage()) {
        val chatId = update.message.chatId
        if (update.message.hasText()) {
            val messageText = update.message.text
            // ...
        } else {
            execute(createMessage(chatId.toString(), "Я понимаю только текст!"))
        }
    }
}
</code></pre><p>При получении сообщения от пользователя мы извлекаем другие сообщения из кеша, чтобы "вспомнить" контекст диалога. При первом обращении диалог будет пустой, т.к. в кеше ничего нет. Добавляем к сообщениям из кеша новое сообщение от пользователя как объект UserMessage. И затем этот список отправляем в ChatGPT.</p><pre><code class="kotlin">val messages = messageCache.getOrInitMessages(chatId).toMutableList()
messages += UserMessage(messageText)
val assistantMessages = openAiService.sendMessages(messages)
</code></pre><p>В ответ мы получаем список сообщений AssistantMessage. В нашем случае там будет всегда один элемент. Добавляем эти сообщения к списку сообщений, обновляем кеш и отправляем пользователю через telegram только последние сообщения от нейросети.</p><p>В общем-то это вся бизнес-логика нашего telegram-бота.</p><h3>Активируем telegram-бот</h3><p>Чтобы Spring корректно подключился к telegram и зарегистрировал бота при старте приложения, нам нужно ещё добавить небольшую конфигурацию:</p><pre><code class="kotlin">@Configuration
class BotConfig {
    @Bean
    fun telegramBotsApi(bot: TelegramBot): TelegramBotsApi =
        TelegramBotsApi(DefaultBotSession::class.java).apply {
            registerBot(bot)
        }
}
</code></pre><p>А чтобы включить механизм-кеширования, нужно добавить аннотацию <code>@EnableCaching</code> к main-классу приложения (который помечен <code>@SpringBootApplication</code>):</p><pre><code class="kotlin">@EnableCaching
@SpringBootApplication
class OpenaiExampleApplication

fun main(args: Array&lt;String&gt;) {
    runApplication&lt;OpenaiExampleApplication&gt;(*args)
}
</code></pre><p>Если этого не сделать, ошибки при старте вы не получите, но кеширование работать не будет.</p><h3>Разворачиваем бота в облаке</h3><p>В том, чтобы запустить чат-бот локально, никаких проблем не будет.
