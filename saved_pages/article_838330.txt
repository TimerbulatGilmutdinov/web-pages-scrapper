В процессе чтения исходников Android SDK я замечал интересные механики и приёмы написания кода, какие-то из них до сих пор используются при создании новых библиотек, другие, напротив, заменены более логичными и понятными конструкциями. В этой статье я постараюсь перечислить всё, что смог заметить сам при изучении исходников Android'а. Сразу отмечу: эта статья не претендует на полноту материала и возможно вы нашли даже больше интересных моментов при чтении кода, ладно, погнали, короче!<br/>
<a name="habracut"></a><br/>
<h2><font color="#3AC1EF">▍ Переопределение protected метода на public в наследуемом классе</font></h2><br/>
Думаю все, кто изучал Java, знают, что можно сделать так (в Kotlin такой возможности нет):<br/>
<br/>
<pre><code class="java">public abstract class Property&lt;T&gt; {

    private T value;

    public Property(T value) {
        this.value = value;
    }

    // метод setValue() недоступен, так как он protected
    protected void setValue(T value) {
        this.value = value
    }

    public T getValue() {
        return value;
    }

}

public class MutableProperty&lt;T&gt; extends Property&lt;T&gt; {

    // метод setValue() переопределён как public, в Kotlin так нельзя(
    @Override
    public void setValue(T value) { super.setValue(value); }

}</code></pre><br/>
Такая механика языка используется для реализации <a href="https://developer.android.com/reference/android/arch/lifecycle/MutableLiveData">MutableLiveData</a>:<br/>
<br/>
<pre><code class="java">public abstract class LiveData&lt;T&gt; {

    protected void postValue(T value) {
        ...
    }

    @MainThread
    protected void setValue(T value) {
        ...
    }

}

public class MutableLiveData&lt;T&gt; extends LiveData&lt;T&gt; {

    ...

    @Override
    public void postValue(T value) {
        super.postValue(value);
    }

    @Override
    public void setValue(T value) {
        super.setValue(value);
    }
    
}</code></pre><br/>
На самом деле такой способ создания изменяемых/неизменяемых классов нарушает концепцию наследования, так как мы не добавляем новую функциональность, а «включаем» её.<br/>
Более предпочтительный способ, как это можно сделать, используя наследование:<br/>
<br/>
<pre><code class="java">public class LiveData&lt;T&gt; {
    
    protected T value;
    
    public LiveData(T value) {
        this.value = value;
    }
    
    public T getValue() {
        return value;
    }
    
}

class MutableLiveData&lt;T&gt; extends LiveData&lt;T&gt; {
    
    public MutableLiveData(T value) {
        super(value);
    }
    
    public void setValue(T newValue) {
        this.value = newValue;
    }
    
}</code></pre><br/>
В любом случае механика переопределения protected на public имеет место.<br/>
<br/>
<h2><font color="#3AC1EF">▍ ThreadLocal переменные</font></h2><br/>
Если вы никогда не слышали, есть такая штука, которая позволяет создать уникальный экземпляр объекта в пределах одного потока, своеобразный Singleton потока.<br/>
<br/>
Посмотрим, для чего это можно использовать:<br/>
<br/>
<pre><code class="java">public final class Looper {

    static final ThreadLocal&lt;Looper&gt; sThreadLocal = new ThreadLocal&lt;Looper&gt;();

    private static void prepare(boolean quitAllowed) {
        if (sThreadLocal.get() != null) {
            throw new RuntimeException("Only one Looper may be created per thread");
        }
        sThreadLocal.set(new Looper(quitAllowed));
    }

}</code></pre><br/>
<a href="https://developer.android.com/reference/android/os/Looper">Looper</a> — один из самых базовых классов Android SDK, на котором построен бесконечный цикл и очередь событий (сообщений).<br/>
<br/>
<a href="https://docs.oracle.com/javase/8/docs/api/java/lang/ThreadLocal.html">ThreadLocal</a> гарантирует, что Looper будет единственным экземпляром в пределах текущего потока, так как в одном потоке может быть только один бесконечный цикл для обработки событий.<br/>
<br/>
Если вы создадите новый поток и вызовете Looper.prepare() на нём, то для него будет создан свой уникальный экземпляр Looper и т. д.<br/>
<br/>
Сложно предположить, где ThreadLocal может пригодиться в повседневной Android разработке, но имейте в виду, если вам нужен уникальный экземпляр в пределах потока, используйте ThreadLocal и обязательно посмотрите документацию с примером.<br/>

<h2><font color="#3AC1EF">▍ Проксирование/Делегирование методов другому классу</font></h2><br/>
Гораздо проще показать на примере <a href="https://developer.android.com/reference/androidx/appcompat/app/AppCompatActivity">AppCompatActivity</a> из библиотеки <a href="https://developer.android.com/jetpack/androidx/releases/appcompat">appcompat</a>:<br/>
<br/>
<pre><code class="java">public class AppCompatActivity extends ... {

    @Override
    protected void attachBaseContext(Context newBase) {
        super.attachBaseContext(getDelegate().attachBaseContext2(newBase));
    }

    @Override
    public void setTheme(@StyleRes final int resId) {
        super.setTheme(resId);
        getDelegate().setTheme(resId);
    }

    @Override
    protected void onPostCreate(@Nullable Bundle savedInstanceState) {
        super.onPostCreate(savedInstanceState);
        getDelegate().onPostCreate(savedInstanceState);
    }

    @Nullable
    public ActionBar getSupportActionBar() {
        return getDelegate().getSupportActionBar();
    }

    public void setSupportActionBar(@Nullable Toolbar toolbar) {
        getDelegate().setSupportActionBar(toolbar);
    }

}</code></pre><br/>
Метод getDelegate() возвращает объект класса <a href="https://developer.android.com/reference/androidx/appcompat/app/AppCompatDelegate">AppCompatDelegate</a>, методы которого реализуют функциональность для методов AppCompatActivity.<br/>
<br/>
Это может пригодиться, когда требуется прозрачно добавить новую функциональность для класса с дальнейшей возможностью на её расширение, «прозрачно» — значит, без влияния на пользователей этого класса.<br/>
<br/>
Приведу простой пример добавления новой функциональности:<br/>
<br/>
<pre><code class="java">class AppCompatActivity extends ... {

    @NonNull
    public AppCompatDelegate getDelegate() {
        if (mDelegate == null) {
            // в Android 34 появились специфичные штуки 
            if (Build.VERSION.SDK_INT &gt;= 34) {
                mDelegate = AppCompatDelegate.create34(this, this);
            } else {
                mDelegate = AppCompatDelegate.create(this, this);
            }
        }
        return mDelegate;
    }

}</code></pre><br/>
Пользователю Android SDK не придётся менять свой код + на более свежих версиях Android'а будут работать новые фишки.<br/>
<br/>
<h2><font color="#3AC1EF">▍ Наследование с реализацией интерфейсов для построения единого API</font></h2><br/>
Многие AppCompat*View классы реализованы таким образом для обеспечения единого API:<br/>
<br/>
<pre><code class="java">public class AppCompatImageView extends ImageView implements TintableBackgroundView, ... {}

public class AppCompatButton extends Button implements TintableBackgroundView, ... {}

public class AppCompatTextView extends TextView implements TintableBackgroundView, ... {}</code></pre><br/>
<a href="https://developer.android.com/reference/androidx/core/view/TintableBackgroundView">TintableBackgroundView</a> — это простой интерфейс для изменения цвета background'а:<br/>
<br/>
<pre><code class="java">public interface TintableBackgroundView {

    void setSupportBackgroundTintList(@Nullable ColorStateList tint);

    @Nullable
    ColorStateList getSupportBackgroundTintList();

    @Nullable
    PorterDuff.Mode getSupportBackgroundTintMode();
  
}</code></pre><br/>
Такой механизм использования интерфейсов имеет несколько преимуществ:<br/>
<br/>
<ol>
<li>легко добавить новую функциональность в независимости от существующей: например, изменение цвета для background'а,</li>
<li>простой и единый интерфейс: не нужно смотреть документацию для каждого компонента, чтобы понять, как у него поменять цвет,</li>
<li>полиморфизм.</li>
</ol><br/>
Последнее проще продемонстрировать:<br/>
<br/>
<pre><code class="kotlin">val views: List&lt;TintableBackgroundView&gt; = listOf(
    AppCompatTextView(this),
    AppCompatButton(this),
    AppCompatImageView(this)
)

val newColor = ColorStateList.valueOf(0xff333333.toInt())

views.forEach { view -&gt;
    view.supportBackgroundTintList = newColor
}</code></pre><br/>
<h2><font color="#3AC1EF">▍ Создание дополнительного типа в качестве пустого значения</font></h2><br/>
Иногда возникают ситуации, когда null не совсем подходит на роль «нет значения», и в таких случаях приходится выкручиваться дополнительным типом:<br/>
<br/>
<pre><code class="java">private class SynchronizedLazyImpl&lt;out T&gt;(initializer: () -&gt; T, lock: Any? = null) : Lazy&lt;T&gt;, Serializable {
    private var initializer: (() -&gt; T)? = initializer

    // дополнительный тип UNINITIALIZED_VALUE указывает, что поле _value ещё не было инициализировано
    @Volatile private var _value: Any? = UNINITIALIZED_VALUE

    private val lock = lock ?: this

    override val value: T
        get() {
            val _v1 = _value
            // проверка состояния поля
            if (_v1 !== UNINITIALIZED_VALUE) {
                @Suppress("UNCHECKED_CAST")
                return _v1 as T
            }

            return synchronized(lock) {
                val _v2 = _value
                // вторая проверка состояния поля на случай, если другой поток уже проинициализировал его
                if (_v2 !== UNINITIALIZED_VALUE) {
                    @Suppress("UNCHECKED_CAST") (_v2 as T)
                } else {
                    val typedValue = initializer!!()
                    _value = typedValue
                    initializer = null
                    typedValue
                }
            }
        }

    // если поле не равно UNINITIALIZED_VALUE значит оно уже было проинициализировано,
    // неважно каким значением, им может быть даже null
    override fun isInitialized(): Boolean = _value !== UNINITIALIZED_VALUE
    
}</code></pre><br/>
Дополнительным типом здесь является UNINITIALIZED_VALUE:<br/>
<br/>
<pre><code class="kotlin">internal object UNINITIALIZED_VALUE</code></pre><br/>
Здесь нельзя обойтись null значением, так как оно входит в диапазон возможных значений:<br/>
<br/>
<pre><code class="kotlin">// временный кэш может быть пустым и тогда значение будет null
val temporaryCache by lazy { getTemporaryCache() }</code></pre><br/>
<h2><font color="#3AC1EF">▍ Переиспользуемый пул объектов, реализованный с помощью связанного списка</font></h2><br/>
Возвращаемся к системе обработки событий в Android, а конкретнее нас интересует класс <a href="https://developer.android.com/reference/android/os/Message">Message</a>:<br/>
<br/>
<pre><code class="java">public final class Message implements Parcelable {

    public static final Object sPoolSync = new Object();
    private static Message sPool;
    private static int sPoolSize = 0;

    private static final int MAX_POOL_SIZE = 50;

    // поле для организации связанного списка
    Message next;

    public static Message obtain() {
        synchronized (sPoolSync) {
            // если пул сообщений не пустой, берём первое доступное 
            // и возвращаем для переиспользования
            if (sPool != null) {
                Message m = sPool;
                sPool = m.next;
                m.next = null;
                m.flags = 0; // clear in-use flag
                sPoolSize--;
                return m;
            }
        }
        // в случае, если пул был пустым или закончился, создаём новое сообщение
        return new Message();
    }

    void recycleUnchecked() {
        // очистить поля для переиспользования объекта сообщения
        flags = FLAG_IN_USE;
        what = 0;
        arg1 = 0;
        arg2 = 0;
        obj = null;
        replyTo = null;
        sendingUid = UID_NONE;
        workSourceUid = UID_NONE;
        when = 0;
        target = null;
        callback = null;
        data = null;

        synchronized (sPoolSync) {
            // если лимит сообщений в пуле не превышен, добавляем текущее для переиспользования
            // в противном случае объект сообщения будет собран сборщиком мусора
            if (sPoolSize &lt; MAX_POOL_SIZE) {
                next = sPool;
                sPool = this;
                sPoolSize++;
            }
        }
    }

}</code></pre><br/>
Пул реализуется с помощью статических полей и методов, конструктор в таком случае желательно делать приватным, чтобы пользователь вашего класса случайно не обошёл логику переиспользования объектов.<br/>

<h2><font color="#3AC1EF">▍ Хранение нескольких значений в целочисленном типе с помощью битовых масок</font></h2><br/>
В Android есть так называемый <a href="https://developer.android.com/reference/android/view/View.MeasureSpec">MeasureSpec</a>, кто писал кастомные вьюшки тот в курсе, как извлекаются значения из него:<br/>
<br/>
<pre><code class="kotlin">class CustomView(ctx: Context) : View(ctx) {

    override fun onMeasure(widthMeasureSpec: Int, heightMeasureSpec: Int) {

        val width = MeasureSpec.getSize(widthMeasureSpec)
        val widthMode = MeasureSpec.getMode(widthMeasureSpec)
                
        val height = MeasureSpec.getSize(heightMeasureSpec)
        val heightMode = MeasureSpec.getMode(heightMeasureSpec)

        ...
    }

}</code></pre><br/>
Если глянуть внутрь этих методов, то можно увидеть битовые операции с одним и тем же целочисленным значением:<br/>
<br/>
<pre><code class="java">public static int getSize(int measureSpec) {
    return (measureSpec &amp; ~MODE_MASK)
}

public static int getMode(int measureSpec) {
    return (measureSpec &amp; MODE_MASK);
}

private static final int MODE_SHIFT = 30;
private static final int MODE_MASK  = 0x3 &lt;&lt; MODE_SHIFT;</code></pre><br/>
Чтобы понять, как это работает, распишем значение константы MODE_MASK в двоичной системе (битовые операции работают с отдельными битами):<br/>
<br/>
<pre><code class="plaintext">MODE_MASK = 00000000 00000000 00000000 00000011 &lt;&lt; 30
// выполняем побитовый сдвиг влево и получаем значение:
MODE_MASK = 11000000 00000000 00000000 00000000</code></pre><br/>
Снова вернёмся к методу MeasureSpec.getMode():<br/>
<br/>
<pre><code class="java">public static int getMode(int measureSpec) {
    return (measureSpec &amp; MODE_MASK);
}</code></pre><br/>
Оператор &amp; выполняет побитовую операцию И (bitwise AND), простыми словами, выставляет единичный бит, если оба бита являются таковыми:<br/>
<br/>
<pre><code class="plaintext">01101110 00110001 10001100 01101111 &amp; 11000000 00000000 00000000 00000000 = 
01000000 00000000 00000000 00000000</code></pre><br/>
Таким образом метод MeasureSpec.getMode() берёт только первые два бита целочисленного числа, а остальные зануляет.<br/>
<br/>
Два бита нужны для хранения одного из следующих режимов при измерении вьюшек:<br/>
<br/>
<pre><code class="java">// 00000000 00000000 00000000 00000000
public static final int UNSPECIFIED = 0 &lt;&lt; MODE_SHIFT;

// 01000000 00000000 00000000 00000000
public static final int EXACTLY = 1 &lt;&lt; MODE_SHIFT;

// 10000000 00000000 00000000 00000000
public static final int AT_MOST = 2 &lt;&lt; MODE_SHIFT;</code></pre><br/>
Второй метод работает практически аналогично, но только извлекает все биты, кроме первых двух:<br/>
<br/>
<pre><code class="java">public static int getSize(int measureSpec) {
    return (measureSpec &amp; ~MODE_MASK)
}</code></pre><br/>
Оператор ~ выполняет побитовую инверсию, меняет нулевые биты на единичные и наоборот:<br/>
<br/>
<pre><code class="plaintext">~11000000 00000000 00000000 00000000 = 00111111 11111111 11111111 11111111</code></pre><br/>
После применения инвертированной маски ~MODE_MASK остаются все биты кроме первых двух:<br/>
<br/>
<pre><code class="plaintext">01101110 00110001 10001100 01101111 &amp; 00111111 11111111 11111111 11111111 =
00101110 00110001 10001100 01101111</code></pre><br/>
Обобщим полученные результаты:<br/>
<br/>
<ol>
<li>MeasureSpec.getMode() берёт только первые два бита целочисленного значения, а остальные зануляет.</li>
<li>MeasureSpec.getSize() зануляет первые два бита целочисленного значения и берёт все остальные.</li>
</ol><br/>
Вот таким элегантным и эффективным способом MeasureSpec хранит в одном целом числе два значения: <br/>
<br/>
<ol>
<li>одно из значений: UNSPECIFIED, EXACTLY, AT_MOST,</li>
<li>размер вьюшки, может быть высота или ширина.</li>
</ol><br/>
Чтобы создать MeasureSpec из отдельных кусочков, нужно сначала пропустить каждое значение через свою битовую маску, а затем сложить получившиеся значения с помощью побитового оператора ИЛИ (bitwise OR):<br/>
<br/>
<pre><code class="java">final int mode = EXACTLY;
final int size = 320;
final int measureSpec = (size &amp; ~MODE_MASK) | (mode &amp; MODE_MASK);</code></pre><br/>
Для более любопытных предлагаю чекнуть исходники android.graphics.Color и глянуть, как извлекаются отдельные компоненты <a href="https://ru.wikipedia.org/wiki/RGB">RGB</a> модели.<br/>
