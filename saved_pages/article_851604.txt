class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>В новом переводе от команды <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> вы узнаете, как минимизировать риски и обеспечить безопасность Docker-контейнеров с помощью CIS Benchmark рекомендаций и автоматизированных инструментов вроде Docker Bench for Security.</p><hr/><p>Вы используете Docker для разработки и тестирования, но еще не предпринимали никаких шагов по использованию его в продакшене? Тогда продолжайте читать, потому что эта статья покажет вам, как гарантировать, что вы запускаете свои  Docker контейнеры безопасным образом.</p><h3>1. CIS Benchmark</h3><p>Стандартная конфигурация Docker недостаточно защищена для продакшн-окружений. Это касается и большинства Dockerfile'ов, доступных в сети. Даже некоторые примеры из наших прошлых статей могут не соответствовать требованиям безопасности. Как же в таком случае узнать, как запустить безопасный Docker-контейнер? Чтобы понять, как защитить контейнеры, можно воспользоваться рекомендациями <a href="https://www.cisecurity.org">Центра интернет-безопасности (CIS)</a>. CIS предоставляет лучшие практики по тому, как можно обезопасить IT системы и данные от атак извне. Эти практики признаны и проверены сообществом опытных IT экспертов. Нам сегодня поможет страница <a href="https://www.cisecurity.org/cis-benchmarks/"><u>CIS Benchmarks</u></a>. Здесь мы найдем множество эталонных решений для операционных систем, устройств и программ. В этом списке мы также можем найти <em>CIS Benchmark for Docker Community Edition 1.1.0</em> (эталонное решение для Docker Community Edition, версия 1.0.0). Оно доступно для бесплатного скачивания, но вам придется указать свои контакты, после чего ссылка для скачивания будет отправлена на ваш адрес электронной почты. Помимо этого вам также предоставят доступ к другим решениям от CIS.</p><p>CIS Benchmark предоставляет вам рекомендации, сгрупированные по:</p><ul><li><p>Конфигурации хоста;</p></li><li><p>Конфигурации Docker daemon;</p></li><li><p>Файлам конфигурации Docker daemon;</p></li><li><p>Образам контейнера и файлу сборки;</p></li><li><p>Рантайму контейнера;</p></li><li><p>Операциям безопасности Docker;</p></li><li><p>Конфигурации Docker Swarm.</p></li></ul><p>Каждая рекомендация содержит описание, способ проверки того, применима ли она к вашему Docker и, что важнее всего, как её можно применить. Рекомендация также может включать в себя статус, отражающий ваш персональный рейтинг. Если вы исполняете рекомендацию, она повышает ваш персональный рейтинг. Если вы ее не выполняете, ваш рейтинг падает. Это относится только к тем рекомендациям, которые содержат статус с рейтингом. Чем выше общий рейтинг, тем более безопасна ваша сборка. </p><h3>2. Docker Bench for Security</h3><p>Проверка рекомендаций вручную может занять много времени, но есть инструмент, который поможет — <a href="https://github.com/docker/docker-bench-security">Docker Bench for Security</a>. Этот скрипт автоматически проверяет настройки Docker по рекомендациям CIS и генерирует отчёт о проблемах, требующих внимания.</p><h3>3. На практике</h3><p>Давайте посмотрим, как это работает. Мы будем использовать следующую конфигурацию:</p><ul><li><p>Ubuntu 18.04</p></li><li><p>Чистая <a href="https://docs.docker.com/install/linux/docker-ce/ubuntu/"><u>сборка</u></a> Docker</p></li></ul><p>Docker-образ, который мы будем использовать, взят из <a href="https://mydeveloperplanet.com/2018/05/16/build-and-deploy-a-spring-boot-app-on-minikube-part-1/">предыдущей статьи</a> и доступен на <a href="https://hub.docker.com/r/mydeveloperplanet/mykubernetesplanet">Docker Hub</a>. Этот образ основан на openjdk 10 и содержит простое Spring Boot Web MVC приложение с одним HTTP-эндпоинтом: <a href="http://ipaddress:8080/hello"><u>http://ipaddress:8080/hello</u></a>. Этот эндпоинт возвращает приветственное сообщение "Hello Kubernetes". </p><p>Dockerfile:</p><pre><code class="powershell">FROM openjdk:10-jdk

VOLUME /tmp

ARG JAR_FILE

ADD ${JAR_FILE} app.jar

ENTRYPOINT ["java","-Djava.security.egd=file:/dev/./urandom","-jar","/app.jar"]</code></pre><p>Первым делом необходимо запустить контейнер:</p><pre><code class="powershell">$ docker run --name=mykubernetesplanet mydeveloperplanet/mykubernetesplanet:0.0.2-SNAPSHOT</code></pre><p>После успешного запуска проверьте, можете ли вы получить доступ к указанному URL. IP-адрес вашего Docker-контейнера можно узнать с помощью команды:</p><pre><code class="powershell">$ docker inspect mykubernetesplanet</code></pre><p>В разделе "Networks" можно найти IP-адрес Docker-контейнера.</p><h4>3.1 Запуск Docker Bench for Security</h4><p>Самый простой способ запустить Docker Bench for Security — это использовать готовый контейнер:</p><pre><code class="powershell">$ docker run -it --net host --pid host --userns host --cap-add audit_control \

    -e DOCKER_CONTENT_TRUST=$DOCKER_CONTENT_TRUST \

    -v /var/lib:/var/lib \

    -v /var/run/docker.sock:/var/run/docker.sock \

    -v /usr/lib/systemd:/usr/lib/systemd \

    -v /etc:/etc --label docker_bench_security \

    docker/docker-bench-security</code></pre><p>Отчет, который будет сгенерирован:</p><pre><code class="powershell"># ------------------------------------------------------------------------------

Как видно, у нас есть несколько предупреждений и заметок. Предупреждения (warnings) — это рекомендации с оценочным статусом, а заметки (notes) — это рекомендации без оценочного статуса. В конце виден наш общий оценочный статус, который не очень высокий. Все запущенные контейнеры проанализированы, включая контейнер Docker Bench for Security.</p><h4>3.2 Исправления по некоторым рекомендациям</h4><p>На основе вывода мы можем легко исправить некоторые проблемы, которые есть в нашем Dockerfile. Как упоминалось ранее, рекомендации CIS Benchmark содержат способы исправления предупреждений, и мы используем их для решения проблем. </p><p><em>Мы увеличиваем версию в нашем файле pom и вносим изменения в ветке feature/dockerbenchsecurity. Источники можно найти на </em><a href="https://github.com/mydeveloperplanet/mykubernetesplanet/tree/feature/dockerbenchsecurity"><em>GitHub</em></a><em>.</em></p><h4>3.2.1 Убедитесь в том, что создали пользователя для контейнера</h4><p>Поскольку мы не создали пользователя для нашего контейнера, наше приложение сейчас запускается от имени root. Мы не хотим этого, потому что у root много привилегий, а для нашего приложения это не требуется. Добавим пользователя appuser в наш Dockerfile, чтобы исправить это предупреждение:</p><pre><code class="powershell">RUN useradd -d /home/appuser -m -s /bin/bash appuser

USER appuser</code></pre><h4>3.2.2 Убедитесь, что в образ контейнера добавлена команда HEALTHCHECK</h4><p>Команда health check позволяет Docker engine отслеживать состояние контейнера. Docker engine может перезапустить контейнер или запустить новые инстансы при необходимости. Мы используем Spring Boot и также добавили Spring Actuator в наш pom файл. Spring Actuator содержит URL для проверки состояния, который мы можем использовать в данном случае. Определим health check в нашем Dockerfile:</p><pre><code class="powershell">HEALTHCHECK --interval=5m --timeout=3s CMD 
curl -f http://localhost:8080/actuator/health/ || exit 1</code></pre><h4>3.2.3 Убедитесь, что используется COPY вместо ADD в Dockerfile</h4><p>Последнее, что мы исправим, — использовать COPY вместо ADD. Оба почти одинаковы, но COPY более точный. Строка с командой ADD:</p><pre><code class="powershell">ADD ${JAR_FILE} app.jar</code></pre><p>Меняем ее на:</p><pre><code class="powershell">COPY ${JAR_FILE} app.jar</code></pre><h4>3.3 Повторный запуск анализа</h4><p>Пересоберем приложение, чтобы сгенерировать новый Docker image, с помощью mvn clean install. Затем остановим запущенный контейнер и удалим его.</p><pre><code class="powershell">$ docker stop mykubernetesplanet
