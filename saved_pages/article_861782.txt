class="tm-article-presenter__origin-link" href="https://www.infoq.com/news/2024/11/spring-6-2-spring-boot-3-4/" target="_blank">Автор оригинала: <span>Karsten Silz</span></a></div><div class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Команда <a href="https://t.me/+QmrKn0wA8CdkNjAy">Spring АйО</a> перевела статью InfoQ, посвящённую новым версиям Spring Framework и Spring Boot. В статье обсуждаются улучшения последних релизов, анонс следующего поколения Spring, запланированного на 2025 год, а также ответы на вопросы редакции InfoQ от руководителя проекта Spring Framework Юргена Хёллера и контрибьютера Spring Framework core Себастьяна Делюза.</p><hr/><p>В ноябре 2024 года компания Broadcom выпустила <a href="https://spring.io/blog/2024/11/14/spring-framework-6-2-0-available-now">Spring Framework 6.2</a> и <a href="https://spring.io/blog/2024/11/21/spring-boot-3-4-0-available-now">Spring Boot 3.4</a>, основываясь на Java 17 и Jakarta EE 9. Spring Boot 3.4 предлагает структурное логирование, добавляет контейнерные images для Docker Compose и Testcontainers, а также улучшает сборку контейнерных images и работу актуаторов. Broadcom также <a href="https://spring.io/blog/2024/10/01/from-spring-framework-6-2-to-7-0">объявила о выпуске Spring Framework 7 и Spring Boot 4</a> в ноябре 2025 года. Новые версии сохранят поддержку Java 17, но перейдут на Jakarta EE 11, внедрят <a href="https://www.infoq.com/news/2024/08/jspecify-java-nullability/">JSpecify для обеспечения null-safety подхода</a>, и <a href="https://www.youtube.com/watch?v=78HV0MRtfiw">ускорят запуск приложений</a> благодаря Project Leyden. InfoQ обсудил эти релизы 2025 года с руководителем проекта Spring Framework Юргеном Хёллером (Juergen Hoeller) и коммитером Spring Framework core Себастьяном Делюзом (Sébastien Deleuze).</p><p><a href="https://docs.spring.io/spring-boot/3.4/reference/features/logging.html#features.logging.structured">Структурное логирование</a> обеспечивает формат вывода логов с четко определенной, зачастую машиночитаемой структурой. Spring Boot поддерживает форматы Elastic Common Schema (ecs), Logstash (logstash) и Graylog Extended Log Format (gelf). Для включения логирования в файл в структурном формате используется параметр logging.structured.format.file, а для вывода в консоль — logging.structured.format.console.</p><p>Docker Compose теперь может принимать аргументы командной строки через новые параметры spring.docker.compose.start.arguments и spring.docker.compose.stop.arguments, а также поддерживает переменную окружения <a href="https://github.com/docker-library/docs/tree/657557822ecebebf0fa6bea96536125881c5bfb2/postgres#postgres_host_auth_method">POSTGRES_HOST_AUTH_METHOD=trust</a>. Spring Boot добавил различные контейнерные образы для Docker Compose и Testcontainers: Redis Stack и Redis Stack Server с redis/redis-stack и redis/redis-stack-server, <a href="https://grafana.com/blog/2024/03/13/an-opentelemetry-backend-in-a-docker-image-introducing-grafana/otel-lgtm/">Grafana LGTM</a> с grafana/otel-lgtm и класс <a href="https://github.com/testcontainers/testcontainers-java/blob/main/modules/grafana/src/main/java/org/testcontainers/grafana/LgtmStackContainer.java">LgtmStackContainer</a> для Testcontainers. Для Kafka теперь доступна поддержка Testcontainers с использованием класса <a href="https://github.com/testcontainers/testcontainers-java/blob/main/modules/kafka/src/main/java/org/testcontainers/kafka/KafkaContainer.java">KafkaContainer</a>.</p><p>Сборка OCI-образов с использованием билдера из недоверенных источников стала более безопасной благодаря функции trustBuilder (документацию можно найти для <a href="https://docs.spring.io/spring-boot/3.4/maven-plugin/build-image.html#build-image.customization">Maven</a> и <a href="https://docs.spring.io/spring-boot/3.4/gradle-plugin/packaging-oci-image.html#build-image.customization">Gradle</a>). Опция <code>imagePlatform</code> позволяет создавать образы для операционных систем и архитектур, отличающихся от платформы хоста. По умолчанию билдер Cloud Native Buildpacks для JVM-приложений — это <code>paketobuildpacks/builder-jammy-java-tiny</code>, что обеспечивает создание более компактных образов. Он поддерживает ARM и x64 прямо из коробки.</p><p>В Spring Boot появился новый режим доступа для актуаторов — «read‑only» — наряду с существующими режимами «unrestricted access» (ранее назывался enabled) и "none" (ранее назывался disabled). Параметр <code>management.endpoints.enabled-by-default</code> теперь называется <code>management.endpoints.access.default</code>, а <code>management.endpoint.&lt;id&gt;.enabled</code> — <code>management.endpoint.&lt;id&gt;.access</code>. Новый актуатор <code>/actuator/info</code> показывает информацию о SSL-сертификатах, включая те, чей срок действия скоро истечет. Актуатор <code>/scheduledtasks</code> предоставляет более детальную информацию о запланированных тасках, включая время следующего выполнения, статус последнего выполнения и исключения.</p><p>Embedded веб-серверы теперь по умолчанию корректно завершают свою работу. Для изменения этого поведения можно установить параметр <code>server.shutdown</code> в значение immediate. Веб-сервер Undertow поддерживает виртуальные потоки, если они включены. RestClient и RestTemplate теперь поддерживают пять HTTP-клиентов: Apache HTTP Components (<code>HttpComponentsClientHttpRequestFactory</code>), Jetty Client (<code>JettyClientHttpRequestFactory</code>), Reactor Netty HttpClient (<code>ReactorClientHttpRequestFactory</code>), JDK HttpClient (<code>JdkClientHttpRequestFactory</code>), и Simple JDK HttpURLConnection (<code>SimpleClientHttpRequestFactory</code>).</p><p>После возвращения поддержки embedded брокера в ActiveMQ Classic, Spring Boot снова может автоматически его конфигурировать с использованием зависимости <code>org.apache.activemq:activemq-broker</code>. Однако, в отличие от Spring Boot 2.7.x, стартер ActiveMQ теперь поддерживает только клиент. Свойство observability spring.application.group объединяет приложения, которые являются частью одной системы. Docker Compose и Testcontainers теперь поддерживают логирование OTLP.</p><p>Spring Boot больше не зависит от библиотеки <a href="https://square.github.io/okhttp/">OkHttp</a>, поэтому пользователям, которым она необходима, придется самостоятельно управлять версией. Аннотации <code>@MockBean</code> и <code>@SpyBean</code> устарели и заменены на аннотации Spring Framework <code>@MockitoBean</code> и <code>@MockitoSpyBean</code>. Spring Boot 3.4 работает только с Gradle версии 7.6.4 и 8.4.0 или новее.</p><p>Обновления зависимостей в Spring Boot 3.4 включают Spring Security 6.4, Spring Session 3.4, Spring Integration 6.4, Spring Batch 5.2 и Spring for Apache Kafka 3.3. Среди сторонних обновлений: Log4j 2.24, Liquibase 4.29, Flyway 10.20, MySQL 9.1.0, MongoDB 5.2.0, Hibernate 6.6, Reactor 2024.0.0, Kafka 3.8, Jackson 2.18, Apache Http Client 5.4, Micrometer 1.14, HtmlUnit 4.3.0, Selenium 4.25.0, Mockito 5.13, JUnit Jupter 5.11 и Testcontainers 1.20.3.</p><p>Spring Framework 6.0 и 6.1, а также Spring Boot 3.0 и 3.2 предлагали больше пользовательских функций, таких как <a href="https://www.infoq.com/news/2022/11/spring-6-spring-boot-3-launch/">Ahead-of-Time-компиляция с использованием GraalVM Native Image</a> или поддержка <a href="https://www.infoq.com/articles/spring-boot-3-2-spring-6-1/">виртуальных потоков и CRaC</a>. В Spring Framework 6.2 и Spring Boot 3.4 таких громких обновлений нет. Это неудивительно для последнего года поколения Spring Framework: в <a href="https://endoflife.date/spring-framework">2021</a> году не было никаких релизов Spring Framework перед выходом версии 6.0 в 2022 году.</p><p><a href="https://www.linkedin.com/in/juergenhoeller/">Юрген Хёллер</a> (Juergen Hoeller) недавно анонсировал выход нового поколения Spring Framework 7 и Spring Boot 4 в ноябре 2025 года. Эти версии сохранят поддержку JDK 17, но «примут грядущий JDK 25 LTS», выпуск которого ожидается за два месяца до этого. Базовая версия Kotlin будет обновлена до версии 2, а Jakarta EE — с 9 до 11, что включает Tomcat 11, Hibernate ORM 7 и Hibernate Validator 9. Spring Framework также перейдет на использование JSpecify для обеспечения null-safety подхода.</p><p>Юрген Хёллер и <a href="https://www.linkedin.com/in/deleuze/">Себастьян Делюз</a> (Sébastien Deleuze), основной коммитер Spring Framework из Broadcom, любезно ответили на вопросы InfoQ.</p><p><strong>InfoQ:</strong> <strong>Spring Framework 6 — это самое короткое поколение с тех пор, как версия 2.0 завершилась через три года в 2009 году. Почему?</strong></p><p><strong>Юрген Хёллер:</strong> Исторически поколения нашего фреймворка формировались под влиянием конкретных факторов своего времени. Мы никогда не ставили целью фиксированную продолжительность жизненного цикла поколения, а скорее ориентировались на определённый набор функций, который служил триггером для нового релиза. По этой причине наш цикл минорных релизов обычно составлял раз в год, но иногда мы пропускали год. Долгосрочная поддержка всегда предоставлялась для последнего функционального релиза каждого поколения, и это оставалось неизменным на протяжении многих лет.</p><p>Сейчас мы адаптируемся к тенденциям индустрии, где предсказуемые релизы и, в частности, предсказуемый цикл долгосрочной поддержки становятся нормой. Кроме того, мы учитываем более частый цикл релизов JDK и более частые мажорные релизы сторонних зависимостей. В результате мы сейчас меньше ориентируемся на объём функций и больше — на временные рамки, в первую очередь для Spring Boot, но также и для Spring Framework.</p><p><strong>InfoQ:</strong> <strong>Spring Framework 6 требовал JDK 17, которому на тот момент был всего один год. Spring Framework 7 сохраняет JDK 17 в качестве базовой версии, несмотря на то, что JDK 21 будет уже двухлетней. Почему Spring 7 впервые с 2006 года остаётся на базовой версии Java предыдущего поколения?</strong></p><p><strong>Хёллер:</strong> Выбор базовой версии обычно диктуется техническими соображениями. JDK 17 предложил множество улучшений по сравнению с JDK 11 (альтернативой в то время), поэтому мы сразу выбрали его для Spring Framework 6.0. Это позволило значительно модернизировать наш код и дать некий толчок индустрии, хотя это усложнило процесс миграции, особенно в сочетании с переходом на неймспейс Jakarta EE.</p><p>Для Spring Framework 7.0 техническая выгода от обновления базовой версии практически отсутствует. Специфические возможности новых версий Java могут быть легко реализованы с помощью multi-release jars, как это уже сделано для виртуальных потоков в Java 21. Текущий консенсус в индустрии явно указывает на базу в виде Java 17, например, в API Jakarta EE 11 и грядущем поколении провайдеров. Поскольку наш текущий код фреймворка не получит преимуществ от обновления базовой версии, мы планируем оставить Java 17 в качестве базы, предоставляя при этом полноценную поддержку Java 21 и Java 25 в рамках одного поколения. Вместо этого повышение базы произойдёт за счёт перехода на API Jakarta EE 11, такие как Jakarta Servlet 6.1 и Jakarta Persistence 3.2, а также Kotlin 2.</p><p>Мы в целом рекомендуем использовать последнюю LTS-версию Java для production deployment, независимо от базовой версии, которая является лишь техническим минимумом, поддерживаемым фреймворком в runtime. Для Spring Framework 7.0, несмотря на базу в виде Java 17, наш основной фокус направлен на поддержку Java 25. В будущем мы рассматриваем Java 25 как привлекательную новую базу для более широкой Java-экосистемы. Однако пока лучшее, что мы можем сделать, — это поддерживать её в рамках диапазона версий JDK.</p><p><strong>InfoQ:</strong> <strong>В анонсе Spring 7 упоминался JSpecify, который стандартизирует аннотации для null-safety. Как обеспечивается null-safety в Spring Framework 6 и Spring Boot 3 сейчас? И какие изменения нас ждут в Spring Framework 7 и Spring Boot 4?</strong></p><p><strong>Себастьян Делюз:</strong> В Spring Framework 6 предоставляются аннотации для null-safety, такие как @Nullable из пакета org.springframework.lang. Эти аннотации являются мета-аннотациями для JSR 305 (замороженного, но широко используемого JSR), что позволяет инструментам, таким как IDEA или Kotlin, обеспечивать поддержку null-safety в универсальном формате.</p><p>В Spring Framework 7 мы планируем перейти на аннотации <a href="http://safety">JSpecify</a> — отраслевой стандарт, над которым команда Spring работала несколько лет, — и объявить как deprecate аннотации для null-safety из Spring.</p><p>Переход на JSpecify принесёт следующие улучшения:</p><ul><li><p>Чёткие <a href="https://jspecify.dev/docs/spec/">спецификации</a> для инструментов и <a href="https://jspecify.dev/docs/start-here/">документации</a> для конечных пользователей.</p></li><li><p>Избежание split-package, вызванного аннотациями JSR 305, что улучшит совместимость с JPMS.</p></li><li><p>Улучшенная интеграция с Kotlin.</p></li><li><p>Возможность указывать null-safety для generic-типа, массивов и элементов varargs.</p></li><li><p>Применимость для других библиотек, не основанных на Spring, например, Reactor и Micrometer.</p></li></ul><p>Мы также планируем всесторонне специфицировать null-safety API в других проектах Spring.</p><p><strong>InfoQ: В анонсе Spring 7 говорилось о «стратегическом выравнивании с GraalVM и Project Leyden». Почему это выравнивание стратегическое?</strong></p><p><strong>Делюз</strong>: Наше выравнивание с GraalVM и Project Leyden стратегическое по трём причинам.</p><p>Во-первых, они дополняют друг друга. GraalVM обеспечивает значительные преимущества, такие как мгновенный запуск/разогрев, но с издержками в виде длительного времени сборки и проблем совместимости. AOT-кэш Project Leyden ускоряет запуск в 3–4 раза — возможно, не так значительно, как native-образы GraalVM, но достаточно для многих пользователей и с минимальными побочными эффектами при использовании поддержки Spring Boot. Сосредоточение на этих двух технологиях значительно повысит эффективность Spring Boot 4 для различных сценариев развертывания.</p><p>Во-вторых, команда Spring активно сотрудничает с обеими командами в рамках R&amp;D, о чём свидетельствуют совместные доклады <a href="https://www.youtube.com/watch?v=78HV0MRtfiw">с командой Java Platform по Project Leyden</a> и <a href="https://www.youtube.com/watch?v=yQC6LBSw2zs">с командой GraalVM на тему native-образов</a> на Devoxx Belgium 2024.</p><p>В-третьих, это стратегическое выравнивание, поскольку мы сосредотачиваем усилия R&amp;D на этих технологиях, предлагая два золотых пути для разработчиков Spring, которые ищут руководство. Обе технологии могут использовать оптимизации Spring Ahead-Of-Time и Buildpacks. Мы ожидаем, что AOT-кэш Project Leyden станет преемником <a href="https://docs.oracle.com/en/java/javase/23/vm/class-data-sharing.html">Class Data Sharing</a> (CDS), который мы уже поддерживаем в Spring Boot 3.3.</p><p>На данный момент мы не планируем значительных усилий по Project CRaC, который сложнее использовать для наших пользователей (<a href="https://x.com/sdeleuze/status/1849471065552015637">см. сравнение</a>). Таким образом, с AOT-кэшем Project Leyden на JVM (и CDS в промежуточный период) и native-образами на базе GraalVM мы создаём две основные опции для повышения эффективности в поколении Spring Boot 4.</p><p>Для получения дополнительной информации ознакомьтесь с примечаниями к релизу <a href="https://github.com/spring-projects/spring-framework/wiki/Spring-Framework-6.2-Release-Notes">Spring Framework 6.2</a> и <a href="https://github.com/spring-projects/spring-boot/wiki/Spring-Boot-3.4-Release-Notes">Spring Boot 3.4</a>.</p><figure class="full-width"><img data-src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/725/c0c/ddb/725c0cddbe78ce9d7a9dd868c3632ad6.png" height="334" на Spring Boot в телеграм — <a href="https://t.me/+oBwACbkvJ19iODAy">Spring АйО</a>, чтобы быть в курсе последних новостей из мира разработки на Spring Boot и всего, что с ним связано.</p><p>Ждем всех, <a href="https://t.me/+oBwACbkvJ19iODAy">присоединяйтесь</a>!</p><p></p></div></div></div><!-- --><!-- --></div><!-- --><!-- --></div><!--]--><!-- --><div class="tm-article-presenter__meta"
