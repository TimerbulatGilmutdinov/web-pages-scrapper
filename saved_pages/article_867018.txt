xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/d0d/b49/2af/d0db492afd0a7e6cb388e83dd3460038.png" height="800" Меня зовут Симонова Анастасия, я Android‑разработчик. </p><p>В современном мире разработка программного обеспечения становится всё более сложной и многообразной. Одним из ключевых инструментов, используемых для создания мобильных приложений, является Android Studio. И помимо стандартных возможностей, Android Studio позволяет разработчикам расширять функциональность своей среды с помощью плагинов. Их использование имеет несколько направлений:</p><ul><li><p>автоматизация рутинных задач;</p></li><li><p>интеграция дополнительных инструментов и библиотек;</p></li><li><p>настройка интерфейса под специфические нужды команды.</p></li></ul><h2>Оглавление</h2><ol><li><p><a href="#%D0%98%D0%BD%D1%81%D1%82%D1%80%D1%83%D0%BC%D0%B5%D0%BD%D1%82%D1%8B">Инструменты для разработки плагинов</a></p></li><li><p><a href="#%D0%9A%D0%BE%D0%BC%D0%BF%D0%BE%D0%BD%D0%B5%D0%BD%D1%82%D1%8B">Базовые компоненты</a></p></li><li><p><a href="#precommitspellchecker">Разработка плагина для проверки орфографии в тексте</a></p></li><li><p><a href="#%D0%9F%D1%83%D0%B1%D0%BB%D0%B8%D0%BA%D0%B0%D1%86%D0%B8%D1%8F">Публикация в маркетплейс</a></p></li></ol><a class="anchor" id="Инструменты" name="%D0%98%D0%BD%D1%81%D1%82%D1%80%D1%83%D0%BC%D0%B5%D0%BD%D1%82%D1%8B"></a><h2>Инструменты для разработки плагинов</h2><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/61c/dd1/d3b/61cdd1d3b0b42501bceae0533ae9819c.png" height="295" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/61c/dd1/d3b/61cdd1d3b0b42501bceae0533ae9819c.png" width="925"/></figure><p>В контексте разработки под Android и JetBrains Marketplace, «плагины» — это модульные расширения, которые добавляют функциональность в среду разработки (IDE), такую как Android Studio. Для начала напомню, где их найти:</p><figure class="full-width"><img alt="Путь до плагинов." data-src="https://habrastorage.org/getpro/habr/upload_files/108/a9e/0d2/108a9e0d2a02096d90c07ddac33d9040.png" height="972" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/108/a9e/0d2/108a9e0d2a02096d90c07ddac33d9040.png" title="Путь до плагинов." width="1548"/><div><figcaption>Путь до плагинов.</figcaption></div></figure><p>Для разработки плагинов нужно всего три вещи:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/365/797/905/3657979052ad0f329bd5d8649b4fd85a.png" height="458" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/365/797/905/3657979052ad0f329bd5d8649b4fd85a.png" width="1226"/></figure><ol><li><p><a href="https://www.jetbrains.com/idea/download/">IntelliJ IDEA</a>, минимум Community Edition. Можно работать и в Ultimate‑версии, но особых преимуществ при разработке плагинов это не даст.</p></li><li><p>Подключенный к ней <a href="https://www.jetbrains.org/intellij/sdk/docs/basics/getting_started/setting_up_environment.html">Plugin DevKit</a> — специальный плагин, который добавляет возможность писать другие плагины.</p></li><li><p>И любой JVM‑ный язык, на котором вы хотите писать плагин. Это может быть Kotlin, Java, Groovy — что угодно.</p></li></ol><a class="anchor" id="Компоненты" name="%D0%9A%D0%BE%D0%BC%D0%BF%D0%BE%D0%BD%D0%B5%D0%BD%D1%82%D1%8B"></a><h2>Базовые компоненты</h2><p>Начинаем с создания проекта плагина. Выбираем <strong>New project</strong>  и создаём проект.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/d8e/301/0d0/d8e3010d0b1a295db941aa23ebf0c315.png" height="1450" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d8e/301/0d0/d8e3010d0b1a295db941aa23ebf0c315.png" width="1628"/></figure><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/ee1/249/53f/ee124953f42abfc6b12b5f143b6a046d.png" height="1116" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/ee1/249/53f/ee124953f42abfc6b12b5f143b6a046d.png" width="1144"/></figure><p>Видим проект плагина по умолчанию, который состоит из:</p><ul><li><p>Папок, в которых вы будете писать код будущего проекта (<em>main/java</em>, <em>main/kotlin</em>, etc)</p></li><li><p>Файла <em>build.gradle</em>, в котором вы будете объявлять зависимости вашего плагина от каких‑то библиотек</p></li><li><p>Файла <em>plugin.xml</em>. В нём содержится необходимая метаинформация о вашем плагине: идентификатор, название, описание, сведения о вендоре, change log, описание зависимостей от других плагинов. Также здесь описывают actions и extensions. Actions позволяет создавать удобные для пользователя действия в интерфейсе, в то время как extensions позволяет интегрировать новую функциональность и взаимодействовать с различными точками расширения Android Studio.</p></li></ul><a class="anchor" id="precommitspellchecker" name="precommitspellchecker"></a><h2>Разработка плагина для проверки орфографии в тексте</h2><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/bd0/828/e2f/bd0828e2fcc1c5b0e5de27d5506e318c.png" height="516" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/bd0/828/e2f/bd0828e2fcc1c5b0e5de27d5506e318c.png" width="882"/></figure><p>Я задумалась: а для решения какой реальной проблемы можно написать свой плагин, но так, чтобы не изобретать велосипед? Ведь на миллион проблем сообщества Android‑разработчиков уже придуманы разные решения. </p><div class="floating-image"><figure class="float"><img alt="баг" data-src="https://habrastorage.org/getpro/habr/upload_files/db6/74b/7c5/db674b7c5727eaf2b37747c1a553292d.png" height="929" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/db6/74b/7c5/db674b7c5727eaf2b37747c1a553292d.png" title="баг" width="455"/><div><figcaption>баг</figcaption></div></figure><p></p><p>Прошло несколько недель, и мне присылают скриншот с орфографической ошибкой в тексте. И тут пазл складывается, у меня возникает идея…</p><p>Я увидела такую <em>проблему</em>: Разработчики и ревьюверы кода часто не замечают орфографические ошибки в текстах, видимых пользователю. Ошибки в тексте могут негативно сказываться на пользовательском опыте и приводить к низким отзывам о приложении.</p><p>И <em>решение</em>: написать плагин для автоматизации проверки орфографии в тексте.</p></div><p>Поставила я себе следующее <strong>техническое задание</strong>: </p><blockquote><p>Функциональность плагина должна работать таким образом: перед каждым <code>git commit</code> должен запускаться механизм, который автоматически проверяет тексты, включённые в изменения, представленные в <code>git diff</code>. В случае обнаружения ошибок разработчик получает уведомление, которое позволяет быстро обратить внимание на проблемы и устранить их до завершения коммита. </p></blockquote><p>Естественно, я решила поискать готовые решения. </p><h2>Russian spell checker</h2><p>Первым решением, которое я обнаружила, стал плагин <a href="https://plugins.jetbrains.com/plugin/13763-russian-spell-checker">Russian spell checker</a>. Он проверяет выбранный текст и показывает подсказку для быстрого исправления.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/07d/7e1/bd6/07d7e1bd637f0b3e3874e5bb37c2feb4.png" height="612" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/07d/7e1/bd6/07d7e1bd637f0b3e3874e5bb37c2feb4.png" width="1382"/></figure><p>У плагина есть два недостатка:</p><ol><li><p>Он использует <a href="https://yandex.ru/dev/speller/"><u>Yandex Speller</u></a> (технологию поиска и исправления ошибок) для проверки орфографии русскоязычного текста. Казалось бы, машинное обучение, последние инновационные технологии, будущее… </p></li><li><p>Чтобы запустить проверку, нужно нажимать на определённые команды, то есть проверка не обязательна и не автоматическая, скажем честно — на усмотрение разработчика, а значит её можно игнорировать…</p></li></ol><p>Использование машинного обучения из сторонних библиотеĸ или решений может нести значительные рисĸи для вашей компании. Однако, возможно, в нашем случае это не так необходимо, поскольку тексты мы копируем из Figma, и дизайнеры, как правило, для текстов привлекают копирайтеров, которые не допускают ошибок в контексте и не искажают слова. Могут возникать лишь небольшие опечатки. Кроме того, хотелось бы иметь решение, которое будет активироваться в «добровольно‑принудительном порядке». Для этого необходимо интегрировать его в какой‑либо обязательный процесс, например, в выполнение команд Git.</p><div class="floating-image"><p>В качестве механизма для проверки орфографии возьмем <a href="https://community.languagetool.org/rule/list?lang=ru">LanguageTool</a> как и у плагина <a href="https://plugins.jetbrains.com/plugin/12175-grazie-lite">Grazie Lite</a>, который будет работать локально, а следовательно безопасно для конфиденциальной информации компании.</p><p></p></div><p>Создаем проект, открываем plugin.xml и настраиваем:</p><pre><code class="kotlin">&lt;idea-plugin&gt;

Они управляют процессом интеграции изменений в систему контроля версий. Получается, что мы можем встроиться в этот процесс для выполнения своих различных проверок и валидаций. 	 val filteredText = text
 		.replace(htmlTagRegex, "")
 		.replace(whitespaceRegex, " ")

Для проверок будем ориентироваться на измененный или новый текст (<code>DeltaType.CHANGE</code>, <code>DeltaType.INSERT</code>), а также будем фильтровать сами файлы и выбирать только текстовые ресурсы.</p><pre><code class="kotlin">private fun getSpellingWarnings(changes: List&lt;LocalChangeList&gt;): String {
 	val textForChecking = StringBuilder()

 	for (changeList in changes) {
     	val changesBeforeCommit = changeList.changes
     	for (change in changesBeforeCommit) {
         	val beforeRevision: ContentRevision? = change.beforeRevision
         	val afterRevision: ContentRevision? = change.afterRevision

         	val beforeContent = beforeRevision?.content
         	val afterContent = afterRevision?.content

         	val filePath = beforeRevision?.file?.path ?: afterRevision?.file?.path
         	if (filePath != null &amp;&amp; filePath.endsWith(".xml") &amp;&amp; filePath.contains("/values/")) {
             	val diff = DiffUtils.diff(beforeContent?.split("\n"), afterContent?.split("\n"))

             	diff.deltas.forEach { deltas -&gt;
                 	when (deltas?.type) {
                     	DeltaType.CHANGE, DeltaType.INSERT -&gt; {
                         	deltas.target.lines?.forEach { text -&gt;
                             	val isCorrectText = checkCorrectText(text)
                             	if (isCorrectText.not()) textForChecking.appendLine(text.trim())
                         	}
                     	}

                     	else -&gt; {}
                 	}
             	}
         	}
     	}
 	}

 	return textForChecking.toString()
 }</code></pre><p>Далее создаём диалоговое окно для пользователя с использованием объекта <code>DialogWrapper</code>. Реализуем простое окно с двумя кнопками «Проверить» и «Пропустить». </p><pre><code class="kotlin">private fun showWarningDialog(warningText: String) {
