class="tm-article-body" data-gallery-root="" lang="ru"><div><!--[--><!--]--></div><div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><p>Предположим, что вы уже какое-то время работаете с Kotlin-корутинами и знакомы с базовыми концепциями, такими как приостанавливаемые (<code>suspend</code>) функции и билдер <code>launch</code>. Однако по мере усложнения проектов вы всё чаще сталкиваетесь с необходимостью искать решения для более продвинутых задач и обращаетесь к поисковым системам или искусственному интеллекту за помощью.</p><p>Эта шпаргалка создана как удобный справочник для сложных сценариев работы с корутинами и содержит ключевые идеи, накопленные мной за всё время работы.</p><h3>Словарь корутин</h3><p><strong>Контекст корутины</strong> (из <a href="https://kotlinlang.org/docs/coroutine-context-and-dispatchers.html">документации Kotlin</a>): «Набор различных элементов. Основными элементами являются Job корутины и её диспетчер». </p><p><strong>Job</strong> (из <a href="https://kotlinlang.org/docs/coroutine-context-and-dispatchers.html">документации Kotlin</a>): «Отменяемая сущность с жизненным циклом, который завершается по завершении корутины. Каждая корутина создаёт свой собственный Job (это единственный элемент контекста корутины, который не наследуется от родительской корутины). </p><p><strong>Dispatcher</strong>: Позволяет выбрать, на каком потоке (или пуле потоков) должна выполняться корутина (как на старте, так и при возобновлении).</p><p><strong>Coroutine scope (область видимости корутины)</strong>: Определяет время жизни и контекст корутины. Она отвечает за управление жизненным циклом корутин, включая их отмену и обработку ошибок.</p><p><strong>Coroutine builder (билдеры корутин)</strong>: это функции-расширения для <code>CoroutineScope</code>, которые позволяют запускать асинхронные корутины (например, <code>launch</code>, <code>async</code> и другие).</p><h4>Основные правила работы с корутинами</h4><ul><li><p>Для запуска корутины необходим <code>CoroutineScope</code> (например, с помощью <code>launch</code> или <code>async</code>). Чаще всего в Android используется <code>viewModelScope</code>, но можно создать и свою область видимости.</p></li><li><p>Дочерняя корутина (корутина, запущенная из другой корутины) наследует контекст корутины-родителя (кроме <code>Job</code>).</p></li><li><p>Job корутины-родителя используется в качестве родительского <code>Job</code> для новой корутины. Это создаёт иерархию задач.</p></li><li><p>Корутина-родитель приостанавливается до тех пор, пока все её дочерние корутины не завершатся.</p></li><li><p>Если корутина-родитель отменяется, то отменяются и все её дочерние корутины.</p></li><li><p>Если дочерняя корутина завершилась с необработанным исключением, это исключение приведёт к отмене корутины-родителя (если не используется <code>SupervisorJob</code>, см. ниже).</p></li><li><p>Не следует использовать <code>GlobalScope</code>, так как это может привести к утечкам памяти и оставит корутину «живой», даже если активность или фрагмент, из которых она была запущена, уже завершили своё выполнение.</p></li><li><p>Не передавайте область видимости корутины в качестве аргумента — вместо этого используйте функцию <code>coroutineScope</code> (пример ниже).</p></li></ul><h3>Функции для работы с областью видимости корутин:</h3><ul><li><p><code>coroutineScope</code>: приостанавливаемая функция, которая запускает область видимости и возвращает значение, произведённое переданной функцией.</p></li><li><p><code>supervisorScope</code>: похожа на <code>coroutineScope</code>, но переопределяет Job контекста, поэтому функция не будет отменена, если дочерняя корутина выбросит исключение.</p></li><li><p><code>withContext</code>: похожа на <code>coroutineScope</code>, но позволяет вносить некоторые изменения в область видимости (обычно используется для того, чтобы задать <code>Dispatcher</code>).</p></li><li><p><code>withTimeout</code>: похожа на <code>coroutineScope</code>, но устанавливает ограничение по времени на выполнение тела, и если оно превышено, функция будет отменена. Выбрасывает <code>TimeoutCancellationException</code>.</p></li><li><p><code>withTimeoutOrNull</code>: аналогична <code>withTimeout</code>, но вернёт null вместо выброса исключения при превышении времени.</p></li></ul><h3>Диспетчеры (Dispatchers) </h3><p>Запуск корутины в конкретном диспетчере требует ресурсов. При вызове <code>withContext</code> корутина приостанавливается и может ожидать в очереди перед возобновлением (см. ниже, как избежать ненужных повторных запусков).</p><p><code>Dispatchers.Default</code></p><ul><li><p>Используется по умолчанию, если диспетчер не задан. </p></li><li><p>Предназначен для выполнения операций, требующих высокой нагрузки на процессор (CPU). </p></li><li><p>Размер пула потоков соответствует количеству ядер на устройстве. </p></li><li><p>Можно ограничить количество потоков, доступных операции, с помощью <code>Dispatchers.Default.limitedParallelism(3)</code>.</p></li></ul><p><strong>Dispatchers.Main</strong></p><ul><li><p>Для Android запускает корутины в основном потоке (UI thread). </p></li><li><p>Важно избегать блокировки этого потока. </p></li><li><p>Не существует в юнит-тестах (при необходимости можно создать собственный Main-диспетчер).</p></li></ul><p><strong>Dispatchers.IO</strong></p><ul><li><p>Предназначен для выполнения блокирующих операций (ввода-вывода, чтение/запись файлов, доступ к Shared Preferences и т.д.). </p></li><li><p>Размер пула потоков составляет 64 (или соответствует числу ядер, если их больше 64). </p></li><li><p>Использует тот же пул потоков, что и <code>Dispatchers.Default</code>, но с независимыми ограничениями. </p></li><li><p>Применяется для функций, выполняющих блокирующие операции. </p></li><li><p>Для запуска корутины с этим диспетчером: <code>withContext(Dispatchers.IO) { // блокирующая функция }</code>. </p></li><li><p>Можно ограничить количество потоков для операции с помощью <code>Dispatchers.Default.limitedParallelism(3)</code>. </p></li><li><p>Использование <code>limitedParallelism</code> с <code>Dispatchers.IO</code> имеет особенность: создаётся новый диспетчер с независимым пулом потоков (лимит может превышать 64).</p></li></ul><p><strong>Dispatchers.Unconfined</strong></p><ul><li><p>Корутина выполняется в том же потоке, в котором была запущена, без смены потоков. </p></li><li><p>Полезен для юнит-тестов. </p></li><li><p>По производительности это самый «лёгкий» диспетчер, так как не происходит переключения потоков.</p></li><li><p>Опасен для использования в продакшене, так как случайное выполнение блокирующего вызова в основном потоке может вызвать проблемы.</p></li></ul><h3>Наблюдения по производительности</h3><ul><li><p>Во время приостановки количество используемых потоков не имеет значительного значения.</p></li><li><p>При блокировке выполнения, чем больше потоков задействовано, тем быстрее завершатся все корутины.</p></li><li><p>При выполнении задач, требующих высокой нагрузки на процессор (CPU), оптимальным будет использование <code>Dispatchers.Default</code>.</p></li><li><p>Для задач, требующих большой памяти, увеличение числа потоков может незначительно улучшить производительность.</p></li></ul><p>Чтобы лучше понять, как работают конкурентность, параллелизм и порядок отмены, посмотрите отличное видео Дейва Лидса:</p><div class="tm-iframe_temp" data-src="https://embedd.srv.habr.com/iframe/671f99ec08571a136eed9c93" data-style="" id="671f99ec08571a136eed9c93" width=""></div><h3>Запуск вызовов параллельно</h3><p>Когда необходимо выполнить два действия одновременно и дождаться результата обоих перед тем, как вернуть итог:</p><p><strong>Если у вас есть доступ к области (например, из ViewModel)</strong></p><pre><code class="kotlin">suspend fun getConfigFromAPI(): UserConfig {
  // do API call here or any suspend functions
}

suspend fun getSongsFromAPI(): List&lt;Song&gt; {
  // do API call here or any suspend functions
}

fun getConfigAndSongs() {
  // scope can be any scope you'd want a typical case would be viewModelScope
  scope.launch {
    val userConfig = async { getConfigFromAPI() }
    val songs = async { getSongsFromAPI()}
    return Pair(userConfig.await(), songs.await())
  }
}</code></pre><p>Предположим, у вас есть API с пагинацией, и вы хотите скачать все страницы до того, как показать их пользователю. При этом вы хотите загрузить все страницы параллельно:</p><pre><code class="kotlin">suspend fun getSongsFromAPI(page: Int): List&lt;Song&gt; {
  // do API call
}
const val totalNumberOfPages = 10

fun getAllSongs() {
  // scope can be any scope you'd want a typical case would be viewModelScope
  scope.launch {
    val allNews = (0 until totalNumberOfPages)
                  .map { page -&gt; async { getSongsFromAPI(page) } }
                  .awaitAll()
  }
}</code></pre><p><em>Примечание по </em><code>async</code><em> / </em><code>await</code><em>. Корутина начнёт выполнение сразу после вызова. </em><code>async</code><em> возвращает объект типа </em><code>Deferred&lt;T&gt;</code><em> (в нашем случае </em><code>Deferred&lt;List&lt;Song&gt;&gt;</code><em>). </em><code>Deferred</code><em> имеет приостанавливаемую функцию </em><code>await</code><em>, которая возвращает значение, когда оно готово.</em></p><p><strong>Если у вас нет доступа к области (например, из репозитория)</strong></p><p>В вашем репозитории или случае использования (use case) необходимо определить корутину, которая запустит два (или более) вызова параллельно. Проблема в том, что для использования <code>async</code> вам нужна область, но вы находитесь вне ViewModel или Presenter и не имеете здесь доступа к области <span class="habrahidden">(не забывайте о том, что передавать область в качестве аргумента — не лучшее решение).</span></p><p>Пример, основанный на нашем сценарии выше:</p><pre><code class="kotlin">suspend fun getConfigAndSongs(): Pair&lt;UserConfig, List&lt;Song&gt; = coroutineScope {
   val userConfig = async { getConfigFromAPI() }
   val songs = async { getSongsFromAPI()}
   Pair(userConfig.await(), songs.await())
}</code></pre><h3>Очистка при отмене корутины</h3><p>Если корутина отменяется, она сначала переходит в состояние cancelling, прежде чем станет cancelled. В состоянии cancelling у нас есть время для выполнения необходимых действий по очистке, если это требуется (например, очистка локальной базы данных, поскольку операция не завершилась успешно, или выполнение API-запроса, чтобы уведомить сервер о неудаче операции).</p><p>Для этого можно использовать блок <code>finally</code>.</p><pre><code class="kotlin">viewModelScope.launch {
  try {
    // call some suspend function here
  } finally {
    // execute clean up operation here
  }
}</code></pre><p>Однако при очистке нельзя использовать функции приостановки. Если вам нужно выполнить функцию приостановки, следует сделать следующее</p><pre><code class="kotlin">viewModelScope.launch {
  try {
    // call some suspend function here
  } finally {
    withContext(NonCancellable) {
      // execute clean up suspend function here
    }
  }
}</code></pre><p><em>Примечание:</em> Отмена произойдёт в первую точку приостановки. Следовательно, отмена не произойдет, если в функции нет точек приостановки.</p><h3>Очистка корутины при её завершении</h3><p>Аналогично очистке при отмене корутины вы можете захотеть выполнить операцию, когда корутина достигает конечного состояния (<code>completed</code> или <code>cancelled</code>).</p><pre><code class="kotlin">suspend fun myFunction() = coroutineScope {
  val job = launch { /* suspend call here */ }
  job.invokeOnCompletion { exception: Throwable -&gt; 
    // do something here
  }
}</code></pre><h3>Как избежать отмены корутины, если один из её дочерних элементов завершился с ошибкой</h3><p>Вы можете использовать <code>SupervisorJob</code>, и он будет игнорировать все исключения в своих дочерних процессах.</p><h4>Создание собственной области видимости корутины</h4><pre><code class="kotlin">val scope = CoroutineScope(SupervisorJob())
// if one throw an error the other coroutine will not be cancelled
scope.launch { myFirstCoroutine() }
scope.launch { mySecondCoroutine() }</code></pre><h4>Использование функции области видимости</h4><pre><code class="kotlin">suspend fun myFunction() = supervisorScope {
  // if one throw an error the other coroutine will not be cancelled
  launch { myFirstCoroutine() }
  launch { mySecondCoroutine() }
}</code></pre><h4>Обработка исключений</h4><pre><code class="kotlin">suspend fun myFunction() {
  try {
    coroutineScope {
      launch { myFirstCoroutine() }
    }
  } catch (e: Exception) {
    // handle error here
  }
  try {
    coroutineScope {
      launch { mySecondCoroutine() }
    }
  } catch (e: Exception) {
    // handle error here
  }
}</code></pre><p><code>CancellationException</code><em> не распространяется на своего родителя, отменяется только текущая корутина.</em> <em>Можно расширить </em><code>CancellationException</code><em>, чтобы создать свой собственный тип исключения, который не будет распространяться на родителя.</em></p><h3>Определите поведение по умолчанию в случае возникновения исключения</h3><p>Используйте <code>CoroutineExceptionHandler</code></p><p>Может быть использован, чтобы автоматически выйти из системы, когда сервер отвечает, например, с кодом <code>401</code>.</p><pre><code class="kotlin">val handler = CoroutineExceptionHandler { context, exception -&gt;
  // define default behaviour like showing a dialog or error message
}
val scope = CoroutineScope(SupervisorJob() + handler)
scope. launch { /* suspend call here */ }
scope. launch { /* suspend call here */ }</code></pre><h3>Запуск вспомогательной операции</h3><p>Если вы хотите запустить функцию suspend, которая не должна влиять на другие (например, если она выбросит ошибку, то только она не отменит родительскую корутину, а другие отменят).</p><p>Пример: вызов аналитики</p><pre><code class="kotlin">val nonEssentialOperationScope = CoroutineScope(SupervisorJob())

suspend fun getConfigAndSongs(): Pair&lt;UserConfig, List&lt;Song&gt; = coroutineScope {
   val userConfig = async { getConfigFromAPI() }
   val songs = async { getSongsFromAPI()}
   nonEssentialOperationScope.launch { /* non essential op here */ }
   Pair(userConfig.await(), songs.await())
}</code></pre><p><em>Идеально — внедрить </em><code>nonEssentialOperationScope</code><em> в класс (упрощает тестирование).</em></p><h3>Запуск операции в одном потоке, чтобы избежать проблем с синхронизацией</h3><pre><code class="kotlin">suspend fun myFunction() = withContext(Dispatchers.Default.limitedParallelism(1)) {
  // suspend call here
}
// Can also use Dispatchers.IO</code></pre><h4>Для предотвращения проблем синхронизации при многопоточности есть другие подходы:</h4><p>Вы можете использовать <code>AtomicReference</code> (из Java).</p><pre><code class="kotlin">private val myList = AtomicReference(listOf(/* add objects here */))

suspend fun fetchNewElement() {
  val myNewElement = // fetch new element here
  myList.getAndSet { it + myNewElement }
}</code></pre><p>Или <code>Mutex</code></p><pre><code class="kotlin">val mutex = Mutex()
private var myList = listOf(/* add objects here */)

suspend fun fetchNewElement() {
  mutex.withLock {
    val myNewElement = // fetch new element here
    myList = myList += myNewElement
  }
}</code></pre><h4>Избегайте повторного перенаправления корутины на тот же диспетчер</h4><p>Избегайте ненужных затрат на переключение диспетчера, если мы уже используем Main диспетчер:</p><pre><code class="kotlin">// this will only dispatch if it is needed
suspend fun myFunction() = withContext(Dispatcher.Main.immediate) {
  // suspend call here
}</code></pre><p>На данный момент только <code>Dispatchers.Main</code> поддерживает немедленное (<code>immediate</code>) выполнение.</p><p>Спасибо за прочтение, делитесь своими идеями и профессиональными советами о корутинах. Moskała). </a></p><blockquote><p>Научиться разрабатывать тесты для всех платформ, где используется Kotlin, можно <a href="https://otus.pw/0J3S/">на онлайн‑курсе «Kotlin QA Engineer»</a> под
